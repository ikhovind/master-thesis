\chapter{Evaluation}\label{chap:eval}
This chapter will evaluate CTaps against two reference implementations as benchmarks:
native TCP and a Picoquic client. Native TCP uses OS-level sockets
directly, while the Picoquic client uses the Picoquic packet
loop as described in \cref{sec:picoquic}.
We will use four experiments to compare CTaps
with these benchmarks:
the handshake latency, the benefit of multistreaming and CWND reuse enabled
by cloning, the effect of candidate racing when a fast and a
slow endpoint are available, and connection migration
after a local path is blocked.

In addition, we will compare a CTaps client with the two benchmarks
in terms of complexity, as measured in lines of code, in order to
show the relative simplicity of a CTaps client given the
features it provides.

\subsection{Software Tools} \label{sec:softwaretools}
All four experiments were performed with clients
and servers running on localhost. In order to reflect
more realistic network conditions, bandwidth emulation
and delay were added using the Linux utility
\mono{tc-netem}~\cite{tc-netem}. Jitter was added
to the candidate racing experiment only, and is
described in \cref{sec:handshake-kde}.

Since \mono{tc-netem} could be invoked programmatically
from the Python testing script, it allowed for
a single experiment to vary its network conditions.
This was used to test multiple delay values in the handshake experiment and the small
file transfer experiment.

The connection migration experiment used \mono{iptables}~\cite{iptables_man} to block the initial
path and force migration. This block was performed based on
source IP, destination IP, protocol and destination port. The specific command
used can be seen in~\cref{lst:iptables-command}.
An equivalent command blocking
TCP data was also run, in order to illustrate the TCP connection failing after a path event.

\begin{listing}[htbp]
    \begin{minted}[fontsize=\small]{bash}
iptables -A OUTPUT -s 127.0.0.1 -d 127.0.0.1 -p udp --dport 8080 -j DROP 
\end{minted}
\caption[iptables command forcing path migration.]{iptables command to block UDP data from local endpoint to file server on localhost with port 8080.}
\label{lst:iptables-command}
\end{listing}

The rule is added to the \mono{OUTPUT} chain of the default \mono{filter} table~\cite{iptables_man}.
Within the Linux kernel, the \mono{OUTPUT} chain is registered at the \mono{NF_INET_LOCAL_OUT}
netfilter hook, which is invoked for every locally-generated outgoing packet~\cite{LinuxKernelIpOutput}.
When the registered hook function returns \mono{NF_DROP} for a packet,
\mono{nf_hook_slow()} returns \mono{-EPERM}~\cite{LinuxKernelNfHookSlow}.
This propagates back up through the networking stack and
\mono{sendmsg()} returns \mono{-1} with \mono{errno} set to
\mono{EPERM}.\footnote{
This specifically simulates a synchronous network failure. For asynchronous
failures, where the packet is lost somewhere along its path, \mono{sendmsg()}
would not return an error. The implications of this are discussed in
\cref{sec:connection-mig-benchmark-discssion}. TODO - add discussion
}

For CTaps this works well as it could
report any \mono{sendmsg()} failure directly to Picoquic.
Picoquic would then communicate back to CTaps that the path has become unavailable through
a callback and CTaps could start probing new paths.

For the Picoquic client a small patch had to be applied in order to achieve
the same migration behavior. The Picoquic packet loop only
notifies the QUIC engine of a path failure if \mono{sendmsg()} fails with \mono{errno} set to a value which implies
that the remote is unreachable. Internally in the Picoquic packet loop this means one of the following errors: \mono{EAFNOSUPPORT},
\mono{ECONNRESET}, \mono{EHOSTUNREACH}, \mono{ENETDOWN} or \mono{ENETUNREACH}
\cite{PicoquicImpliesUnreachable}.

For the simulation it was desirable for the Picoquic client
to also treat \mono{EPERM} as implying that the remote is unreachable. Picoquic was therefore forked
and \mono{EPERM} was added to the list of socket errors which imply that
the remote is
unreachable~\cite{PicobrigtImpliesUnreachable}. The forked version of Picoquic was only used
for the Picoquic client when benchmarking. All other instances of Picoquic,
including the one used in CTaps, were fetched from the main Picoquic repository.

In addition to the clients, two servers were implemented for 
benchmarking.\footnote{
The servers and clients used for benchmarking can be found in the \mono{benchmark/src/server/}
and \mono{benchmark/src/client/}
folders in the main CTaps repository. }
Each server accepted incoming connections and supported
two commands, one for initiating the transfer of the small file and one
for initiating the transfer of the large file.

\subsection{Measurement Approach}
Experiment results were gathered through a common Python script.
This script is invoked with a specific set of command line arguments for each

scripts for the sake of reproducibility.\footnote{The associated shell scripts
can be found in the \mono{benchmark/scripts/} folder of the main CTaps repository,

\section{Experiments}
This section will describe the experiments themselves and
their results.

\subsection{Single Endpoint Handshake Experiment}
This experiment can be seen in~\cref{fig:racing-handshake} and
intends to show that a single CTaps client
can connect to multiple servers
serving different protocols and
that there is no significant overhead in CTaps when connecting to a server.
It also shows the cost of the staggered racing delay as it currently exists in CTaps.

This experiment iterates
over delays of 25, 50, 75, and \qty{100}{\milli\second} to demonstrate the linear
relationship between network delay and handshake time. No jitter was added
to the network and results are therefore very consistent between runs.
Consequently, the graph is based on a single run.

CTaps does not support TLS and the
TCP handshakes are therefore run without encryption.
The TCP 3-way handshake completes in one RTT, while 
QUIC mandates TLS 1.3 and requires two RTTs for a full handshake
when including the \mono{HANDSHAKE_DONE} frame.

Because the handshake client used for CTaps here is the same as
the client used in~\cref{sec:handshake-kde}, the client has a preference
for QUIC and is given two remote endpoints. This is the reason for the
\qty{500}{\milli\second} stagger delay between the CTaps TCP and QUIC
handshake times, since CTaps tries two remote endpoints for QUIC before trying TCP
and the stagger delay is a static \qty{250}{\milli\second} for each attempt.
The high cost of this stagger delay is discussed further in~\cref{sec:discussion}.

For each of the handshakes
the bandwidth was set to \qty{6.25}{\mega\byte\per\second}, although the bandwidth
does not have an effect on the time taken for the initial handshake.

\begin{figure}[htbp]
    \input{./resources/racing_handshake.pgf}
    \caption[Single endpoint handshake experiment]{Measured handshake times for TCP, Picoquic and CTaps clients, ordered by RTT.}
    \label{fig:racing-handshake}
\end{figure}

\subsection{Candidate Racing Experiment}\label{sec:handshake-kde}
This experiment can be seen in~\cref{fig:handshake-kde}. It intends to
show the benefits of candidate racing and how it helps find the
endpoint with the smallest delay.
This experiment was modeled after the theoretical
scenario described in~\cref{fig:racingSeqDiagram} in~\cref{chap:impl}.

Two possible paths were used with different remote endpoints.
The slow endpoint had a delay of \qty{150}{\milli\second}
while the fast endpoint had a delay of \qty{50}{\milli\second}.
Bandwidth was limited to \qty{50}{\mega\bit\per\second}, although
this has no effect on the handshake times.

Both the Picoquic
and CTaps clients were given the slow endpoint as a first alternative.
Since CTaps supports candidate racing it was also given the fast endpoint
as a second alternative. This is intended to emulate the scenario after
for example a DNS resolution where multiple endpoints are available but
the client does not know which one is best.

In order to gather a distribution of handshake times, the test was run 200~times with a
jitter of \qty{7}{\milli\second} and a latency correlation of \qty{25}{\percent}.
In the scenario that this experiment models,
candidate racing would lead to not only shorter
handshake times, but also a significantly smaller delay throughout the entire connection lifetime.

The theoretical ideal handshake times for each endpoint are featured
in the graph. With a \qty{250}{\milli\second} racing delay, 2~RTTs for a full handshake and delays of
\qtylist{50;150}{\milli\second}
the theoretical minimum handshake times work out to 
$\qty{50}{\milli\second} \cdot 4 + \qty{250}{\milli\second} = \qty{450}{\milli\second}$
and
$\qty{150}{\milli\second} \cdot 4 = \qty{600}{\milli\second}$ respectively.

    \input{./resources/handshake_kde.pgf}
    \caption[Candidate racing experiment]{Distribution of handshake times when a fast and a slow endpoint are available, but the slow
    endpoint is attempted first, showing the benefit of candidate racing. ($N = 200$)}
    \label{fig:handshake-kde}
\end{figure}

\subsection{Small File Transfer Experiment}\label{sec:smallfiletransfer}
This experiment can be seen in~\cref{fig:small-file-bar} and the
intention is to show that a TAPS system can benefit
from QUIC multistreaming while using a generic TAPS client
compatible with both TCP and QUIC.

This experiment shows an example of lost capacity when performing Connection Cloning
over TCP when compared to QUIC, as modeled in~\cref{fig:connection-clone-gains} in 
\cref{chap:impl}.
In order to show that the performance gap between TCP and QUIC grows with increasing delay
the experiment sweeps across four delays of
\qtylist{25;50;75;100}{\milli\second}

The measurement consists of the time taken to transfer a file with 70~packets'
worth of data. For TCP this works out to
$70 \cdot 1460 \approx \qty{100}{\kilo\byte}$ after discounting the 20 byte IP and TCP
headers from a typical 1500-byte~MTU. QUIC has slightly more overhead per packet,
but not enough to make a significant difference in the required file size.

Before the small file transfer was started, a large file of \qty{4.6}{\mega\byte} was
transferred in order to increase the CWND to be greater than the size of the small file.
This ensures that the small file transfer could be performed in a single RTT if the
CWND is reused. This large file is the same as the one used in the migration experiment
described in~\cref{sec:path-migration}.

To verify that the small file transfer can actually be completed in
a single RTT we need to check that the large file is large enough for the CWND to ramp up to at least 70~packets
and that the BDP of the link is large enough to carry the entire small file in a single RTT.

Assuming an initial CWND of 10~packets, the number of RTTs required for the CWND to grow to at least
70~packets
during slow start is
$10 \cdot 2^{x-1} \geq 70 \Rightarrow x \geq 4 \text{ RTTs}$. During the minimum of 4~RTTs
$\sum_{n=1}^{4} 1460 \cdot 10 \cdot 2^{n-1} = \qty{219}{\kilo\byte}$ of data would
be transferred, which is well below the \qty{4.6}{\mega\byte} file used in the experiment.

To verify that the BDP is large enough, we can calculate the minimum BDP for
70~packets which is
$70 \cdot 1500 \cdot 8 = \qty{840}{\kilo\bit}$.

In our experiment, the smallest delay is \qty{25}{\milli\second}
giving an RTT of \qty{50}{\milli\second}.
The bandwidth is constant at \qty{50}{\mega\bit\per\second} which gives a minimum BDP of
$\qty{50}{\mega\bit\per\second} \cdot \qty{50}{\milli\second} = \qty{2.5}{\mega\bit}$.
Since the bandwidth is constant and BDP scales
with RTT we can therefore conclude that the BDP of the link is large enough
to carry the small file for all tested delay values.

  \input{./resources/small_file.pgf}
  \caption[Small file transfer experiment]{Transfer time of a \qty{100}{\kilo\byte}~file after first transferring a large file to grow the CWND, showing the benefit of multistreaming in QUIC.}
  \label{fig:small-file-bar}
\end{figure}

\subsection{Connection Migration Experiment}\label{sec:path-migration}
This experiment can be seen in~\cref{fig:migration-throughput} and it intends
to show the benefits of connection migration
and that it is available with minimal code changes in CTaps.

In this experiment, a \qty{4.6}{\mega\byte} file is transferred over an emulated
\qty{5}{\mega\bit\per\second} link with a delay of \qty{25}{\milli\second}.

The combination of file size and bandwidth was chosen to give the transfer enough
time to reach its steady-state and to still have data left
to transfer after recovering from the path failure.

This test uses two local IP addresses consisting of \mono{127.0.0.1} and \mono{127.0.0.2}.
All clients successfully connect through \mono{127.0.0.1} and start the transfer,
but after five seconds the local endpoint is blocked through iptables as
described in~\cref{sec:softwaretools}.

After the local endpoint is blocked both CTaps and the Picoquic client
detect this failure and
start the connection migration process, while TCP fails and cannot recover.

This experiment was gathered with the CTaps and QUIC client described in~\cref{sec:client-complexity},
showing that connection migration is made available through CTaps with minimal complexity
for the application.

\section{Discussion} \label{sec:discussion}
\subsection{Single Endpoint Handshake Experiment}\label{sec:single-endpoint-handshake-discussion}
When it comes to CTaps, the results in~\cref{fig:racing-handshake} are promising
but also illustrate some weaknesses.
The experiment shows that 
there is little overhead in connection setup for
CTaps and that a single CTaps client can successfully connect to multiple
servers with different protocols, due to candidate racing.

The experiment does however also show that candidate racing
is not implemented optimally in CTaps.
Currently, CTaps uses static stagger delay
of 250~milliseconds between attempts as recommended by Happy Eyeballs V2~\cite{rfc8305}, but it does
not implement interleaving as described in the same document.
This means that all
combinations involving the most desirable protocol are attempted
before a new protocol is attempted.

This easily becomes a very significant delay, as the number of
raced parameters easily becomes very high when racing both
path combinations and protocol options.

If even a simple form of interleaving was implemented, then the
stagger delay present for TCP could be halved from \qty{500}{\milli\second}
to \qty{250}{\milli\second}.

This means that if an application is faced with a large number of Endpoints,
as it for example might be after a remote hostname resolution,
then it could face a multi second delay in CTaps
if the most desirable protocol is
not compatible to the server it is trying to connect to.

This lack of protocol interleaving is made worse by CTaps treating
each individual ALPN value as a separate protocol option, as this
inflates the number of candidates for racing for all protocols which
support ALPN. This design choice was however necessary 
in order to support 0-RTT as session resumption relies on
using the same ALPN as the previous session. In theory it
would be possible to only generate candidates for each ALPN like
this if 0-RTT was supported according to the Transport Properties
and Security Parameters set by the user, but this was deemed to be out
of scope for CTaps.

Having access to several different transport layer protocols
becomes far less appealing if the racing
delays introduce multi-second handshake delays, limiting
the current appeal of CTaps.

The 250~millisecond delay recommended in Happy Eyeballs
V2 has a very specific background in that it is racing two protocol families
on a single local path.
With the larger number of
possible combinations of candidates, a TAPS implementation should
follow the recommendations for racing on multihomed hosts as
described in~\cite{rfc8421} more closely.

Other potential mitigations for this problem with interleaving are discussed further
in~\cref{sec:future-work}.

In addition to the stagger delay, the connection establishment time of
a CTaps client is also vulnerable to slow DNS resolutions,
although this does not show in the handshake time experiment.
Currently, CTaps waits until all local and remote
endpoints have been resolved before sorting the gathered candidates according
to the application preferences and then working its way down the list.
This could be optimized by integrating candidate gathering and candidate
racing more tightly, utilizing a Resolution Delay similar to the one described
in Happy Eyeballs~\cite{rfc8305}.

\subsection{Candidate Racing Experiment}
The stagger shows up not only in the raw handshake measurement times,
but also in the candidate racing experiment. Here the experiment itself becomes
more contrived than needed because it needs to make the faster endpoint
preferred even with the long \qty{250}{\milli\second} stagger delay. This means making the
delay for the slow endpoint three times larger than for the fast
endpoint. While this
still serves to show the benefit gained on a suboptimal ordering from
a DNS resolution, the difference in the two endpoints has to be
unrealistically large due to the CTaps-specific stagger delay.

% Discuss the small file transfer
\subsection{Small File Transfer Experiment}
The small file experiment shows that CTaps can utilize Connection
Cloning to make use of QUIC multistreaming and achieve performance
parity with a Picoquic client. The experiment does
however consist of a 
small I/O bound application and it does
not mean that there is no overhead with using CTaps.

The small file
transfer is a constructed scenario and in real-world
situations downloading large files as required for this experiment may be uncommon.

In 2025 Cloudflare
measured that the median number of packets sent to clients of their service
was 12~\cite{sulemanahmadMeasuringCharacteristicsTCP2025}. A majority of 
connections never even leave slow start.

However, it can also be argued that
these are exactly the connections which necessitate multistreaming. Since
such a large number of connections consist of only short data transfer, they
are never able to be scaled up to make use of the full available bandwidth.
Therefore, multiple streams using the same congestion control allows for a
far more efficient use of the network.

However, the benefit of CTaps over a normal QUIC client is more implicit
than explicit in this scenario,
as it shows that the very same client can connect to both a TCP and a QUIC server,
can be treated exactly the same, and an application does not have to care
whether the server it is connecting to supports QUIC or not.

% Discuss connection migration
\subsection{Connection Migration Experiment}\label{sec:connection-mig-benchmark-discssion}
This experiment shows that connection migration is available through
CTaps and that CTaps can make use of another QUIC-specific advantage
from a client that also supports TCP.

While the experiment shows that connection migration is currently supported
with little overhead, there are however still many scenarios that are not
supported by CTaps. These would not present extra complexity to the
application, but would lead to quite a bit of internal CTaps complexity.

As discussed in~\cref{sec:connection-migration-impl},
the current implementation is a workaround, and there is currently no support
for a CTaps client to migrate from a socket that is bound to a non-wildcard IP.
This naturally limits the number of situations in which the client is actually able to migrate.

% Discuss performance in general
\subsection{Performance}
The discussion of the overhead of a networking library becomes
unproductive in situations where applications anyways
are I/O-bound, which is naturally the case in the CTaps experiments.
They can therefore be used to draw the conclusion that there
is no significant overhead between CTaps and the underlying protocols,
but they cannot for example be used to judge how CTaps client would perform
under load as a pure QUIC or TCP client.

Since both the Picoquic packet loop client and CTaps interact with the network in
a asynchronous manner, the system has time to perform any outstanding computation while
waiting for a response for the network. Since any computationally heavy work performed
on the data sent to or from the network would be in the domain of the application using CTaps,
the only way to truly show the overhead of CTaps would be to create a large amount of
concurrent clients and to transfer large amounts of data in order to stress CTaps as
the interface between the network and the client. When it comes to high performance, CTaps
is naturally limited by the single threaded event loop model of libuv and would therefore
not be able to fully make use of all the resources available to it on a host system.

% Discuss CTaps client complexity
\subsection{Client Complexity}
The complexity of each  client as seen in~\cref{tab:loc_complexity}
tells a very clear story and CTaps successfully abstracts away the complexity presented
to client applications interacting with the network. It does so not only when accessing
advanced features of QUIC such as connection migration, but also in the protocol
selection and the error handling itself. This is the primary aspect which sets
CTaps apart from other networking libraries, especially ones intending to help
make use of a single protocol such as QUIC. With CTaps, the complexity is moved
away from the application and presented in a generic way which can be reused
by any application using the network. This does not only come with the added
simplicity, but also much easier migration to future protocols.

While CTaps is the most complex
individual client by lines of code, it provides TCP and QUIC support, connection racing,
and several QUIC benefits at less than the combined complexity of the QUIC
and TCP clients. A combined TCP and QUIC client would likely feature
some shared logic, so the naive sum in~\cref{tab:loc_complexity} can
be viewed as an upper bound for the non-CTaps clients. 

However, in order to develop a replacement of CTaps the TCP and QUIC
clients would not only have to be combined, but would also have to implement
fallback logic which is not accounted for in this table.

It is clear that the functionality present in QUIC has been desired
for a long time, and
several popular libraries exist which have significant feature overlap
with QUIC. Two examples are Valve's GameNetworkingSockets~\cite{ValveGameNetworkingSockets} and ENet
\cite{Enet}. Both of these predate the finalization of the QUIC RFC~in 2021 and
contain overlapping features which are also implemented through QUIC, such as connection
management over UDP or multistreaming (referred to as "Channels" in Enet).

If an application relying on one of
these networking libraries wants to make use of QUIC it has to switch
its entire networking stack to use a QUIC library. This is naturally a large
refactoring and is a limitation on how fast new networking features can propagate.
If a production ready TAPS-library had existed when this hypothetical application was originally
designed it could have abstracted away the actual underlying transport library and
made a later switch to a newer protocol such as QUIC essentially free. This comes in
addition to built-in fallbacks to the previous protocol should connection establishment fail
for QUIC.

With
the large amount of complexity which is abstracted away by CTaps, it becomes a very
heavy dependency and therefore quite a large risk for downstream applications
which become heavily reliant on the correctness of CTaps.
With the built in libuv event loop, CTaps forces your software to be written through an
async model which does its work through callbacks. This means that an application wanting
to use CTaps for underlying networking operations has to be designed with this in mind,
from the very beginning, creating a barrier for adoption in existing libraries.
