\subsubsection{Software tools} \label{sec:softwaretools}
Netem is a Linux kernel module which allows users to emulate network conditions such
as RTT packet loss and bandwidth limitations. For the benchmarks in this thesis, netem
was used to set RTT times and bandwidth limitations to show TAPS, Picoquic and TCP
behavior in different network conditions. Netem could be invoked programmatically
from the Python testing script, which allowed easy creation of sets of benchmarks
while varying the underlying network conditions, such as the varying RTT times seen
in the handshake benchmark. Netem was also used to vary the bandwidth for the different
benchmarks. For example, in the migration test bandwidth was set to \qty{625}{\kilo\byte\per\second}
in order to
ensure that the test had adequate time to reach its steady state transfer speed before the
path was blocked.

In the migration test, iptables was used in order to block a specific path based on
source IP, destination IP, protocol and destination port. When a packet matches the
assigned rule, netfilter rejects it and propagates \mono{-EPERM} back through the network
stack \cite{LinuxKernelNfHookSlow}. \mono{sendmsg} then returns -1 with
errno set to \mono{EPERM} according to POSIX conventions.

For CTaps this worked well, as we could
report any \mono{sendmsg()} failure directly to Picoquic as the path being unavailable.

For the Picoquic socket loop client, a small patch had to be applied in order to achieve
the expected behavior in this specific benchmark. The Picoquic in packet loop only
notifies the QUIC engine of a path failure if the \mono{sendmsg()} fails with errno set to a value which implies
that the remote is unreachable, which means one of the following errors: \mono{EAFNOSUPPORT},
\mono{ECONNRESET}, \mono{EHOSTUNREACH}, \mono{ENETDOWN} or \mono{ENETUNREACH}
\cite{PicoquicImpliesUnreachable}. For the simulation, it was desirable for the Picoquic client
to treat \mono{EPERM} as implying that the remote is unreachable. Picoquic was therefore forked
for this benchmark and \mono{EPERM} was added to the list of socket errors which imply that
the remote is
unreachable \cite{PicobrigtImpliesUnreachable}. The forked version of Picoquic was only used
for the Picoquic packet loop client when benchmarking, all other instances of Picoquic,
including the one used in CTaps, were fetched from the main Picoquic repository.

\subsubsection{Handshake latency}
This benchmark has several intentions. It was meant to show that
a CTaps client can connect to multiple servers serving different protocols,
that there is no significant overhead between CTaps and the underlying
protocol handshake and show the cost of the staggered racing delay
as it stands now and as an area for future improvement. The handhake
was captured in the same benchmark as the small file transfer as
described in \cref{smallfiletransfer}. The benchmark iterated
over 50, 100, 150, and \qty{200}{\milli\second} RTTs, to show the linear
growth between handshake time and the handshake time. While the Picoquic
and TCP client handshakes had to be measured by different clients compatible
with the given server protocol, the TAPS racing client remained the same.
This was intentional, to show the benefit of TAPS not requiring code change 
despite the protocol offered by the server changing. For each of the handshakes
the bandwidth was set to \qty{6.25}{\mega\byte\per\second}, although the bandwidth
does not have an effect on the time taken for the initial handshake.
The TCP server was a pure socket implementation, while the QUIC server
was pure picoquic, using the built in picoquic packet loop. This was done to
avoid any server-side issues with TAPS affecting the results and giving a
biased result.

\subsubsection{Small file transfer} \label{smallfiletransfer}
For this benchmark, the intention was to show that a TAPS system could benefit
from QUIC multistreaming, with no code change necessary in a generic TAPS client
compatible with both TCP and QUIC. The benchmark is intended to show the
benefit of CWND reuse with QUIC multistreaming with a negligible overhead from TAPS.
The measurement consists of the time taken to transfer a small file of 70 packets
$70 \cdot 1460 \approx \qty{100}{\kilo\byte}$ after a large file already has been
transferred in order to increase the CWND to be greater than the size of the small file.
In the benchmark the emulated bandwidth was set to \qty{6.25}{\mega\byte\per\second} and
a file of \qty{4.6}{\mega\byte} was transferred. In order to achieve the desired CWND
of at least 70, the connection would have to remain in slow start for
$10 \cdot 2^x \geq 70 \Rightarrow x = 4 \text{ RTTs}$, requiring a capacity of
$70 \cdot 1460 \approx \qty{100}{\kilo\byte}$ per round trip. This capacity
is known as the bandwith delay product, and for the shortest RTT tested of
\qty{50}{\milli\second} RTT it is
$\qty{6.25}{\mega\byte\per\second} \cdot \qty{50}{\milli\second} = \qty{313}{\kilo\byte\per\second}$
more than satisfying the \qty{100}{\kilo\byte} BDP required to transfer the entire
the small file in one RTT. During the ramp up to the required CWND
$\sum_{n=1}^{4} 1460 \cdot 10 \cdot 2^n = \qty{219}{\kilo\byte}$ would be transferred.
The combination of inital file size, the simulated bandwidth and the simluated RTT
therefore ensures that the small file can be transferred in a single RTT.

\subsubsection{Path migration}
One of the key faetures of TAPS is exposing QUIC to clients without
them having to implement their own fallback logic. QUIC has several desirable
features which is not available in normal TCP and one of these is path migration.
Since QUIC connections are identified by their connection ID not by their four-tuple
as with TCP, QUIC connections can migrate to a different IP address while still maintaining
their connection status. This is especially useful in mobile devices, where one might expect
certain interfaces to become unavailable frequently, as users enter and exit buildings and switch
between Wi-Fi and cellular frequently. It was therefore desired to show that this is a feature of
QUIC which comes at a negligible overhead compared to a pure QUIC implementation such as Picoquic.
This test uses the same \qty{4.6}{\mega\byte} file as described in \cref{smallfiletransfer}, but
sets the emulated network bandwidth to \qty{625}{\kilo\byte\per\second} and RTT value to \qty{50}{\milli\second}.
This was done to give the transfer enough time to reach its steady-state and to still have data left
to transfer after recovering from the path failure. For this test the CTaps
client was given two local ip addresses \mono{127.0.0.1} and \mono{127.0.0.2}. It would try to connect
from the first stated address of \mono{127.0.0.1} and succeed, the transfer would start, but after
five seconds the benchmark script would run:

\begin{figure}[H]
    \begin{minted}[fontsize=\small]{bash}
iptables -A OUTPUT -s 127.0.0.1 -d 127.0.0.1 -p udp --dport 8080 -j REJECT
\end{minted}
\end{figure}

In order to reject data flowing from the client's intitial endpoint of \mono{127.0.0.1}
to the server. This would cause sendmsg to return an error, which then could be passed on to
Picoquic. Picoquic would notify CTaps that the path had become unavailable, and CTaps could tell
Picoquic to start probing the other path registered with the connection. For Picoquic the built
in packet loop also has support for migration. It automatically detects when a path fails
and notifies the client that the offending path has been suspended. You can then immediately tell
it to probe a new path, and it will start migration if it manages to contact the server on one of
them.

A detailed account on how the iptables behaviour was integrated with the Picoquic packet loop
used in the Picoquic client can be seen in \cref{sec:softwaretools}.

\subsection{Client complexity}
While the advantages from QUIC made available through TAPS are clear, they
can also be achieved through using a QUIC library and writing custom fallback
logic to retreat to TCP on failing to establish a QUIC connection. This does
however introduce a lot of complexity in the client, and makes the application
layer responsible for what should be the responsibility of the transport layer.
One of the goals if TAPS is therefore to reduce complexity presented to the
application. This is achieved by front-loading complexity in the initialization
and context setup and abstracting away live connection management from the application.
This means that even more advanced features such as migration is achieved with
minimal code changes, simply by stating the intention to migrate if issues
arise, the application using CTaps get this feature for nearly free. The
relative complexity of each benchmarking client, measured in lines of code,

\section{QUIC Under TAPS} \label{sec:quicUnderTaps}
Things with QUIC that cannot be covered with TAPS:
\begin{itemize}
    \item Closing a stream without closing the connection. A TAPS connection maps to a stream while a connection group maps to a QUIC connection. However operations are performed on connections, this adds some poorly mapped operations.
    \item TAPS cannot differ between RESET\_STREAM and STOP\_SENDING.?
    \item Close reason code, application cannot provide a reason to peer?
    \item Undirectional streams
    \item Relative priority of streams?
    \item Receiver flow control? Is this configurable for a QUIC sender, or is it handled by the engine
    \item Limiting the number of streams from the peer.
    \item kommet til kapittel 5 i rfc 9000
\item \mono{disable_active_migration}
\end{itemize}

TAPS has been designed with QUIC in mind from the very start,
but there are several aspects of QUIC which do not map directly
to a control flow presented to the application via the standardized
TAPS interface. As the first TAPS implmentation to feature QUIC as
an underlying protocol option, finding out exactly which featues of
QUIC are available through the default TAPS interface was one of the
key research questions for this thesis. In order to fit the concepts of QUIC
into the concepts presented in TAPS, terminology and intented functionality
does not map 1 to 1. This is a comprise which had to made when creating
an API which is supposed to map neatly to both TCP and QUIC. Since the
intention of TAPS is to create an API which reflects how most users
actually interact with the network, so it is only natural that compromises
has to be made in terms of QUIC, in order to reflect the ubiquity of
TCP.

The primary differences when it comes to QUIC and TAPS concepts is
the Connection and the Connection Group. a TAPS Connection maps to an
individual transfer of data, which naturally maps to a single TCP connection
but maps to a QUIC stream. Since multistreaming is handled through connection
groups in TAPS, a single QUIC connection maps to a connection group in TAPS.
This means that the underlying behaviour of the TAPS API is affected by the
underlying transport protocol, where the closing of a single connection to
a specific endpoint with QUIC happens through closing a connection group.
This differing in concepts leads to the first major hurdle in mapping QUIC
to TAPS. A QUIC stream is designed with HTTP/3 in mind, intended to transfer
a single object, with both the client and the server marking the transfer
from their respective ends as independently completed to close the stream.
While TCP connections can be closed one way, the typical behaviour is that
when it closed by one peer, then the other peer closes the connection
in conjunction. This does not reflect the typical behaviour of streams in QUIC.
This means that a TAPS system has to make a choice for how to close individual
TAPS connectoins when QUIC is the underlying protocol. The expected behaviour
from TCP is that closing an individual connection will close both the sending
and receiving side and start teardown. This can be achieved by sending a \mono{FIN}
flag and a \mono{STOP_SENDING} frame. The peer then must respond with a \mono{RESET_STREAM}
frame if their stream is in a ready or sending state \cite{iyengarQUICUDPBasedMultiplexed2021}.
A \mono{RESET_STREAM} frame teards down a QUIC stream in the sending direction, but has
no effect on the receiving direction. It must therefore be combined with
to close the stream in the receving connection, to complete stream teardown.
Even though this handling of underlying QUIC streams in TAPS is superficially
closer to the expected behavior from TCP, is it not a graceful close is
expected to happen on error conditions. For a graceful teardown of a single
stream only, the expected behaviour is both sides sending \mono{FIN} flags when
they are done sending.

In order to avoid unexpected error behaviour for QUIC, the most natural mapping
for a TAPS Connection with QUIC as an underlying protocol is therefore to simply
send a \mono{FIN} flag in the sending direction and remain open in the receving direction.
This will reflect to the appication in the form of it still having to monitor
the connection for incoming messages and it not invoking a closed()-callback until
after the peer has finished sending its data. If a peer wants to gracefully close
a connection before all data has been received, the only way of handling it is to
close the entire QUIC connection. This naturally maps to the \mono{ConnectionGroup.close()}
functionality form RFC9622 \cite{trammellAbstractApplicationProgramming2025}.
If the application takes care of performing TAPS connection cloning in a consistent way,
it can constently use \mono{ConnectionGroup.close()} and achieve the same
expected behaviour no matter the underlying protocol in the TAPS system, this
is however an example of underlying functionality leaking through to how the user
should interact with the API.

As mentioned above QUIC streams do not map perfectly into the expected behavior
if TAPS Connections. This also means that not all functionality in a QUIC stream
is available through a TAPS Connection. For example, the one-way abort mentioned
earlier thorugh \mono{STOP_SENDING}, which requests that the peer abort their
sending, and \mono{RESET_STREAM} which aborts the local sending side has no
mapping through the TAPS API. An individual stream can be aborted through
\mono{Connection.abort()}, however TAPS has no way of knowing whether the application
wants to abort sending, receiving or both. The only natural conclusion is
therefore to abort both sending and receiving, leaving QUIC functionality
unavailable to the application.

In addition, when opening a new TAPS Connection, unidirectional QUIC
streams are not available through the TAPS API. While a stream can be
immediately opened and be immediately for sending, QUIC allows
applications to open streams to be used purely for unidiraction sending,
with stream IDs denominating whether the stream was opened by the client
or the server and whether or not the stream is unidirectional on bidirectional.
A TAPS system has to assume that every new stream opened via \mono{Connection.clone()}
is bidirectional, although since the TAPS API is extensible a parameter to
respesct this can easily be added, although it would likely have to be ignored
or enforced by the TAPS system itself for TCP and several other protocols.
A TAPS system is able to receive an unidirectional stream initiated by the peer,
and can simply mark the incoming connection as closed for sending before presenting
it to the application.

TAPS Connections also do not let the application control \mono{FINAL_SIZE} values
of QUIC streams, which is used to control how much data a QUIC stream can send, although
this will still be reported through sending \mono{FIN} flags, which is supported.

There is also several pieces of lower layer QUIC functionality such as flow control
through frames such as \mono{MAX_STREAM_DATA} which is in theory able to be
sent from QUIC, but possible through a TAPS system. \todo{Er dette relevant, er ikke akkurat som om det er tilgjengelig via TCP heller}

In a pure QUIC system an application can specify a maximum number of streams to
be received on a QUIC connection. While a TAPS application can specify a maximum
number of connections which chan be spawned from a Listener, there is no way
to stop the peer from opening more connections through connection cloning.

On the server side, TAPS does not expose preferred addresses through its API.
These are used by QUIC when it wants to accept on a given address but wants
to migrate it to a new address shortly after the handshake \cite{iyengarQUICUDPBasedMultiplexed2021}.
Migration is however supported, so future support for preferred addresses can
easily be added to the API through a flag in a given local endpoint.

Certain features, such as stateless Reset are not exposed through the TAPS
API directly, but are expected to be handled by the underlying QUIC implementation,
so that there is no need to expose them in a TAPS system.

In the TAPS API ECN is marked as an UDP-lite specific feature, though this
is also useful for QUIC \cite{trammellAbstractApplicationProgramming2025}.

Additionally, when closing connections picoquic exposes no way of passing
an error code to the peer.

TAPS features no standardized way of notifying the connection of migration,
though connection migration is a key feature of QUIC \todo{Er dette sant?}.
With an extensible callback system, this could be implemented quite easily by
an individual TAPS implementation.

\section{Discussion}
% Discuss handshake time
The handshake time shown in \cref{fig:racing-handshake} tells several stories.
Fist of all, it shows that there is little overhead in connection setup for
CTaps. Since the action is I/O-bound, the the network is the largest bottleneck
by far. It also shows a slightly biased comparison between the TCP and QUIC
handshakes as it does not include TLS, which would add another RTT onto
the TCP handshake and bring it in line with QUIC. It does however show that
there is a linear trend between all four combinations of server/client tried,
and that all of the handshake times grows in relation to their required number
of RTTs, nothing else. With the TCP implementation it also shows both a weakness
in the current implementation of CTaps, and a result of the desire to as much
of the benchmarking to a single CTaps client as possible, showing that modifying the
code is not necessary. For candidate racing CTaps currently uses a stagger delay
of 250 milliseconds between attempts as recommended by Happy Eyeballs V2, it does
however not implement interleaving as described in the same document
\cite{schinaziHappyEyeballsVersion2017}. As CTaps races local endpoints,
protocols, ALPN options and remote endpoints the consequences become
even more apparent for CTaps, as if it prefers an incompatible protocol
it will iterate through all possible combinations of local endpoints,
remote endpoints and ALPNs if applicable before trying the next protocol.
This means that if you are faced with a large number of endpoints, as you
might be after a hostname resolution for example, or when binding to an
ephemeral local endpoint, then you could face a multi second delay in CTaps
before even trying the correct protocol, if the highest ranked protocol is
not compatible to the server you are trying to connect to.

If an attempt was to be made to make CTaps into a real-world production
ready system, then candidate interleaving would likely be one of the first
features to be implemeted. The biggest advantage of CTaps, that you can
access to several different transport layer protocols, without having to
implement racing and fallback logic yourself becomes moot if the racing
overhead is so large, that any other fallback system would be much faster.
It would have to be carefully considered exactly how these candidates were
to be interleaved. The applications does rank its most desired protocols,
so interleaving with no care as to the order could lead to a lower ranked
protocol being tried second after the most desirable one, to the application's
surprise. It would also be beneficial to include the local and remote endpoints
in the interleaving algorithm somehow. In the hypothetical situation that the
first candidates all try to connect to the same remote endpoint, but that endpoint
turns out to be unreachable, the current stagger delay would lead to a one
second delay before the next endpoint was tried.

In retrospect, the \qty{250}{\milli\second} delay recommended in Happy Eyeballs
V2 has a very specific background in that it is racing two protocol families.
While racing is one of the key features of TAPS, with the larger combination
of candidates where each one can represent an unreachable server or an incompatible
protocol, the staggering should likely be much shorter than \qty{250}{\milli\second}
for a TAPS system. This would allow CTaps to race its larger number of candidates
more efficiently, while still avoiding the undue strain on the network which
simultaneous racing can present.

Another mitagation against the tyranny of the stagger
delay would be to start connection attempts upon first resolution at least
one local and remote endpoint. Currently, CTaps waits until all local and remote
endpoints have been resolved before sorting the gathered candidates according
to the application preferences and then working its way down the list. Connection
attempts in CTaps can be made the same way as recommended in Happy Eyeballs V2, racing
as soon as a resolved endpoint comes back, with a \qty{50}{\milli\second} resolution delay
to give preferred addresses a chance to return before initiating an
attempt with a less desirable endpoint.

% Discuss the small file transfer
While the above benchmarks show that CTaps achieves performance parity
with TCP sockets and Picoquic on small I/O bound applications, it does
not mean that there is no overhead at all with using CTaps. The small file
transfer is a constructed scenario used to show advantages with QUIC and
how CTaps makes them available through no extra cost to the user. In real-world
situations downloading large files like this is uncommon.
In 2025 Cloudflare
measured that the median number of packets sent to visitors to their service
was 12 \cite{sulemanahmadMeasuringCharacteristicsTCP2025}. A majority of 
connections never even leave slow start. However, it can also be argued that
thesee are exactly the connections which necessitate multistreaming. Since
such a large number of connections consist of only short data transfer, they
are never able to be scaled up to make use of the full available bandwidth.
Therefore, multiple streams using the same congestion control allows for a
far more efficient use of the network. This is also exactly what the small
file transfer benchmark shows, that the second transfer is so small that TCP
cannot compete with QUIC, since it has to go through slow start for the new data
transfer. This is a benefit of QUIC over TCP, not a benefit of TAPS vs a normal
QUIC implementation. However, the benefit of CTaps in this scenario is implicit,
as it shows that the very same client can connect to both a TCP and a QUIC server,
can be treated exactly the same, and the application does not have to care
whether the server it is connecting to supports QUIC or not.

% Discuss connection migration
As mentioned above, one of the key features with QUIC is Connection Migration.
This is another aspect of QUIC hich helps utilize the avaiable bandwidth more
efficiently. If QUIC can successfully migrate from a dead path to a live one
without having to restart the connection and go through slow start again then
it leads to a much lower experienced latency by the user, for any transfer less
than the 10 packets sent during the initial RTT for TCP, in addition to avoiding
an additional handshake. It was therefore desirable to show that this feature
too was available through CTaps, and that there is minimal overhead. The results
are absolutely promising and show that the necessary complexity presented to the application
in order to achieve conneciton migration is very small. The graph shows familiar
sawtooth patterns of both CTaps, TCP and Picoquic having reached their steady-state
throughput. After the initiated path failure, TCP naturally fails, while Picoquic
and CTaps can both probe new paths, allowing them to continue until the transfer
completes.

When it comes to connection migration there is still a lot of functionality
which is not supported. This would not present extra complexity to the application,
but would lead to quite a bit of internal CTaps complexity. For example, if
the clients binds to a specific local endpoint with a specific port, there is no support
in the client currently for migrating away from this. This is passed to \mono{bind()}
and CTaps does not support opening new local sockets for migration. This naturally
limits the number of situations in which the client is actually able to migrate,
compared to what is desirable.

If adding this feature in the future, it should come as a part of a larger refactor,
removing the binding between individual connections and sockets in general. This would
also help listeners listen on several local endpoints, something which is not currently
supported in CTaps.


% Discuss performance in general
The discussion of the overhead of a networking library often becomes
quite moot if you are discussing situations in which the applications anyways
is I/O-bound. This is naturally the case with several of our benchmarks, where
the actual performance of the library does not matter much, if it will take a
minimum of \qty{50}{\milli\second} to set up a single QUIC handshake. This can
very clearly be seen in several ways in the handshake, small file transfer and
migration benchmarks. As with any library there is overhead to CTaps compared
to using purely picoquic, however this does not become apparent since those three
benchmarks are all I/O-bound, and do not place a load on the actual client or server.
Since both the Picoquic packet loop client and CTaps interact with the network in
a asynchronous manner, the system has time to perform any outstanding computation while
waiting for a response for the network. Since any computationally heavy work performed
on the data sent to or from the network would be in the domain of the client using CTaps,
the only way to truly show the overhead of CTaps would be to create a large amount of
concurrent clients and to transfer large amounts of data, in order to stress CTaps as
the interface between the network and the client. When it comes to high performance CTaps
is naturally limited by the single threaded event loop model of libuv and would therefore
not be able to fully make use of all the resources available to it on a host system.

% Discuss CTaps client complexity
The complexity of each benchmarking client as seen in \cref{tab:loc_complexity}
tells a very clear story. CTaps successfully abstracts away the complexity presented
to client applications interacting with the network. It does so not only when accessing
advanced features of QUIC such as connection migration, but also in the protocol
selection and the error handling itself. This is the primary aspect which sets
CTaps apart from other networking libraries, especially ones intending to help
make use of a single protocol such as QUIC. With CTaps, the complexity is moved
away from the application and presented in a generic way which can be reused
by any appliaction using the network. This does not only come with the added
simplicity, but also much easier migration to future protocols.

Several popular libraries exist which has significant feature overlap
with QUIC such as Valve's GameNetworkingSockets or ENet \cite{ValveGameNetworkingSockets}
\cite{Enet}. Both of these predate the finalization of the QUIC RFC in 2021 and
contain overlapping features which are also implemented through QUIC, such as connection
management over UDP or multistreaming (referred to as "Channels" in Enet). If a project relying on one of
these networking libraries wants to make use of QUIC features it has to switch
its entire networking stack to use a QUIC library. This is naturally a large
refactoring and is a limitation on how fast new networking features can propagate.
If a production ready TAPS-library had existed when the application was originally
designed, it could have abstracted away the actual underlying transport library desired
by the application, making switching to QUIC later essentially free. This comes in
addition to built-in fallbacks to the previous protocol should connection establishment fail.

This does not mean that CTaps is in a production ready state and should be immediately
switched to by any library currently utilizin ENet or GameNetworkingSockets. With
the large amount of complexity which is abstracted away by CTaps, it becomes a very
heavy dependency in and of itself. Applications may be wise in preffering to handle
their own networking stacks accepting future refactoring costs should they desire
to switch transport protocol. CTaps also has an issue of potential refactoring costs.
With the built in libuv event loop, CTaps forces your software to be written through a
async model which does its work through callbacks. This means that an application wanting
to use CTaps for underlying networking operations has to be designed with this in mind,
creating a very heavy reliancy on a single library and a single point of failure.

% Discuss QUIC under TAPS
Being an important research question it was important to find out if there
were large swaths of QUIC features which could not intuitively be presented
to an application through a TAPS system. While there are features which are
difficult to map, the absolute unavailable features are of a lesser importance.
The largest amount of missing features from QUIC in TAPS can be found in the
transport layer, such as with \mono{MAX_STREAM_DATA}, where this is not a
decision the application should make, but CTaps should make an informed decision.

The largest issues with creating a common abstraction for several different protocols like
TAPS is that there ends up being distinct concepts in each protocol which do
not necessarily map neatly into the shared abstraction layer concepts. This
is apparent in the differing behaviour of Connections in CTaps depending on
whether the underlying protocol is QUIC or not. with QUIC the Connection abstration
becomes the connection group and treating individual streams as connections
leads to unexpected behaviour, such as one-way closing by default except
when aborting the stream. This issue would likely be shared with any other
multistreaming protocol as well. This causes the application to have to interact
with TAPS in a way which does not reflect how they would interact with TCP,
for example always closing the entire connection group when done with a logical
connection to a server, to ensure that if the underlying protocl supports multistreaming
then the entire connection will be closed. While this is breaks slightly with 
with the principle of least surprised as described in RFC 9622, it provides
far greater flexibility and consistent behaviour to the downstream application
\cite{trammellAbstractApplicationProgramming2025}. It can also be added that
the only reason for this expectation of how to interact with the network
exists is due to TCP's long ubiquity as a transport protocol. This is
purely an appeal to tradition and should not be considered blocking in
situations where new useful features can be provided.

%discuss racing with UDP
When it comes to connection racing, it is a useful feature but can
be hard to map for non-connection oriented protocols leading to
unexpected or even undesirable behaviour. In CTaps this becomes an
issue with UDP. While the default preferences often lead users to TCP or
QUIC over UDP, if UDP is preferred but not required then there is no way for
racing to function properly. CTaps has chosen to assume that a UDP connection
is established as soon as local endpoint has been set up. This means that
if the remote is not actually genuinely UDP, the racing will tell the user
that the connection is ready, but the remote will be unreachable. An alternative
solution would be to send a test message and to wait for a reply, but
this relies on knowledge of the remote endpoint and that it will actually reply
to a ping. This risks designating legitimate endpoints as unresponsive. Since
the default preferences rank UDP lower than other supported protocols, it was
deemed that the risk of UDP being ranked higher than other protocols when
the endpoint does not support it was low enough that the simpler solution
of assuming an estbalished connection on opening the socket.
