\subsubsection{Software Tools} \label{sec:softwaretools}
Netem is a Linux kernel module which allows users to emulate network conditions such
as RTT packet loss and bandwidth limitations \cite{tc-netem}. 

As mentioned, all the benchmarks in this thesis were performed on the machine listed in
\cref{tab:machine-specs} on localhost. In order to provide network emulation the
Linux utility \mono{tc-netem} was used \cite{tc-netem}. This enabled the benchmarks
to emulate a link with delay, limited bandwidth and jitter.

Netem could be invoked programmatically
from the Python testing script, which allowed varying network conditions to be used for
a single benchmark, for example with the varying RTT times
in the handshake benchmark.

In the migration test, iptables \cite{iptables_man} was used in order to block a specific path based on
source IP, destination IP, protocol and destination port. The specific command
used when blocking a local path can be seen in \cref{lst:iptables-command}.
An equivalent command blocking
TCP data was also run, in order to illustrate the TCP connection failing after a path event.

This command adds a packet filter rule to the OUTPUT chain in the Linux kernel.
This chain is
run for locally generated packets \cite{iptables_man}, and when a packet
matches the assigned rule, netfilter rejects it and propagates \mono{-EPERM} back through the network
stack \cite{LinuxKernelNfHookSlow}. \mono{sendmsg} then returns -1 with
errno set to \mono{EPERM} according to POSIX conventions.

For CTaps this worked well, as it could
report any \mono{sendmsg()} failure directly to Picoquic.
Picoquic would then communicate back to CTaps that the path has become unavailable through
a callback, and CTaps could start probing new paths.

For the Picoquic client, a small patch had to be applied in order to achieve
the expected behavior in this specific benchmark. The Picoquic in packet loop only
notifies the QUIC engine of a path failure if the \mono{sendmsg()} fails with errno set to a value which implies
that the remote is unreachable, which means one of the following errors: \mono{EAFNOSUPPORT},
\mono{ECONNRESET}, \mono{EHOSTUNREACH}, \mono{ENETDOWN} or \mono{ENETUNREACH}
\cite{PicoquicImpliesUnreachable}. For the simulation, it was desirable for the Picoquic client
to treat \mono{EPERM} as implying that the remote is unreachable. Picoquic was therefore forked
for this benchmark and \mono{EPERM} was added to the list of socket errors which imply that
the remote is
unreachable \cite{PicobrigtImpliesUnreachable}. The forked version of Picoquic was only used
for the Picoquic client when benchmarking, all other instances of Picoquic,
including the one used in CTaps, were fetched from the main Picoquic repository.

\subsection{Results}

\subsubsection{Handshake Latency}
This benchmark has several intentions. It was meant to show that
a CTaps client can connect to multiple servers serving different protocols,
that there is no significant overhead between CTaps and the underlying
protocol handshake and show the cost of the staggered racing delay
as it stands now and as an area for future improvement. The handhake
was captured in the same benchmark as the small file transfer as
described in \cref{sec:smallfiletransfer}. The benchmark iterated
over 50, 100, 150, and \qty{200}{\milli\second} RTTs, to show the linear
growth between handshake time and the handshake time. While the Picoquic
and TCP client handshakes had to be measured by different clients compatible
with the given server protocol, the TAPS racing client remained the same.
This was intentional, to show the benefit of TAPS not requiring code change 
despite the protocol offered by the server changing. For each of the handshakes
the bandwidth was set to \qty{6.25}{\mega\byte\per\second}, although the bandwidth
does not have an effect on the time taken for the initial handshake.
The TCP server was a pure socket implementation, while the QUIC server
was pure picoquic, using the built in picoquic packet loop. This was done to
avoid any server-side issues with TAPS affecting the results and giving a
biased result.

\subsubsection{Connection Establishment with Slow and Fast Endpoint}
This benchmark intends to show the benefits of candidate racing, and
how it can mitigate the effects of for example a bad ordering of endpoints in
a DNS resolution. This benchmark was modelled after the theoretical
scenario described in \cref{fig:racingSeqDiagram} in \cref{chap:impl},
with a fast and a slow endpoint where the slow endpoint is attempted
first.

Two possible paths were used, with different remote endpoints.
The slow endpoint had a delay of \qty{150}{\milli\second}
while the fast endpoint had a delay of \qty{50}{\milli\second}.
Bandwidth was limited to \qty{50}{\mega\bit\per\second}, although
this has no effect on the handshake times.

Both the Picoquic client \todo{picoquic -> Picoquic}
and CTaps were given the slow endpoint as a first alternative.
Since CTaps supports racing, it was also given the fast endpoint
as a second alternative.

In order to gather a distribution of handshake times, the test was run 200 times, with a
jitter of 7 ms and a latency correlation of 25 \%.
The resulting graph can be seen in \cref{fig:handshake-kde}. While this
graph only shows the handshake times, candidate racing would in this case lead to not only shorter
handshake times, but also a significantly smaller delay throughout the entire connection lifetime.

The theoretical ideal handshake times for each endpoint are featured
in the graph. With a 250 ms racing delay, 2 RTTs for a full handshake and delays of 50 and 150
milliseconds, the theoretical minimum handshake times work out to 
$\qty{50}{\milli\second} \cdot 4 + \qty{250}{\milli\second} = \qty{450}{\milli\second}$
and
$\qty{150}{\milli\second} \cdot 4 = \qty{600}{\milli\second}$ respectively.

\subsubsection{Small File Transfer} \label{sec:smallfiletransfer}
For this benchmark, the intention was to show that a TAPS system could benefit
from QUIC multistreaming, while using a generic TAPS client
compatible with both TCP and QUIC, without considerable overhead when compared
to a non-TAPS QUIC implementation.

This benchmark shows an example of lost capacity when performing Connection Cloning
over TCP when compared to QUIC, as modelled in \cref{fig:connection-clone-gains} in 
\cref{chap:impl}.

The measurement consists of the time taken to transfer a file with 70 packets'
worth of data. For TCP this works out to
$70 \cdot 1460 \approx \qty{100}{\kilo\byte}$ after discounting the 20 byte IP and TCP
headers from a typical 1500 byte MTU. QUIC has slightly more overhead per packet,
but not enough to make a significant difference in the required file size.

Before the small file transfer was started, a large file of \qty{4.6}{\mega\byte} was
transferred in order to increase the CWND to be greater than the size of the small file.
This ensures that the small file transfer could be performed in a single RTT, if the
CWND is reused. This large file is the same as the one used in the migration benchmark
described in \cref{sec:path-migration}, which is why it is much larger than strictly
required for the small file transfer benchmark.

To verify that the small file transfer can actually be completed in
a single RTT, we need to check that the large file is large enough for the CWND to ramp up to at least 70 packets
and that the BDP of the link is large enough to carry the entire small file in a single RTT.

Assuming an initial CWND of 10 packets, the number of RTTs required for the CWND to grow to at least 70 during slow start is
$10 \cdot 2^{x-1} \geq 70 \Rightarrow x \geq 4 \text{ RTTs}$. During the minimum of 4 RTTs
$\sum_{n=1}^{4} 1460 \cdot 10 \cdot 2^{n-1} = \qty{219}{\kilo\byte}$ of data would
be transferred, which is well below the \qty{4.6}{\mega\byte} file used in the benchmark.

To verify that the BDP is large enough, we can calculate the minimum BDP for
70 packets, which works out to be
$70 \cdot 1500 \cdot 8 = \qty{840}{\kilo\bit}$.

In our benchmark, the shortest RTT is \qty{50}{\milli\second} and the bandwidth is
constant at \qty{50}{\mega\bit\per\second}. This gives a BDP of
$\qty{50}{\mega\bit\per\second} \cdot \qty{50}{\milli\second} = \qty{2.5}{\mega\bit}$.
Since the bandwidth is constant and BDP scales
with RTT we can therefore conclude that the BDP of the link is large enough
to carry the small file for all RTTs tested.

The benchmark contains four RTTs of 50, 100, 150 and 200 milliseconds. This was done to
show that the performance gap between TCP and QUIC grows with increasing RTTs.
The resulting transfer times of the small file can be seen in \cref{fig:small-file-bar}.

\subsubsection{Path Migration} \label{sec:path-migration}
The path migration benchmark is intended to show the benefits of connection migration,
that it is available with minimal code changes in CTaps, and the behaviour of TCP
when a local endpoint becomes unavailable.
In this benchmark, a \qty{4.6}{\mega\byte} file is transferred over an emulated
\qty{5}{\mega\bit\per\second} link with an RTT of \qty{50}{\milli\second}.

The combination of file size and bandwidth was chosen to give the transfer enough
time to reach its steady-state and to still have data left
to transfer after recovering from the path failure. For this test, the CTaps
client was given two local ip addresses \mono{127.0.0.1} and \mono{127.0.0.2}. It successfully connects
from \mono{127.0.0.1} and begins the transfer. After
five seconds the benchmark script blocks the local endpoint, as
described in \cref{sec:softwaretools}.

After the local endpoint is blocked both CTaps and the Picoquic client
detect this failure and
start the connection migration process, while TCP fails and cannot recover.

The resulting comparison of a local endpoint becoming unavailable for TCP, QUIC and CTaps
can be seen in \cref{fig:migration-throughput}. This figure shows how both CTaps and the picoquic
client recover after the local endpoint becomes unavailable, but TCP fails and does not complete
the transfer.

This benchmark was gathered with the CTaps and QUIC client described in \cref{sec:client-complexity},
showing that connection migration is made available through CTaps with minimal complexity
for the application.

\subsection{Client complexity} \label{sec:client-complexity}
While the advantages from QUIC made available through TAPS are clear, they
can also be achieved through using a QUIC library and writing custom fallback
logic to retreat to TCP on failing to establish a QUIC connection. This does
however introduce a lot of complexity in the client, and makes the application
layer responsible for what should be the responsibility of the transport layer.

One of the goals of TAPS is therefore to reduce complexity presented to the
application. This is achieved by front-loading complexity in the initialization
and context setup and abstracting away live connection management from the application.
This means that even more advanced features such as migration are achieved with
minimal code changes, simply by stating the intention to migrate if issues
arise.

\section{QUIC Under TAPS} \label{sec:quicUnderTaps}
TAPS has been designed with QUIC in mind from the very start,
but there are several aspects of QUIC which do not map directly
to a control flow presented to the application via the standardized
TAPS interface.

As the first TAPS implementation to feature QUIC as
an underlying protocol option, finding out exactly which featues of
QUIC are available through the default TAPS interface was one of the
key research questions for this thesis. Since TAPS is an interface to
a generic transport layer, TAPS concepts, terminology and intended functionality
naturally does not map one to one onto concepts, terminology and functionality
of QUIC.

The largest differences when it comes to the expected behavior when
for QUIC and TAPS stems from the
the Connection and the Connection Group. A TAPS Connection maps to an
individual transfer of data, which naturally maps to a single TCP connection
but maps to a QUIC stream. Since multistreaming is handled through connection
groups in TAPS, a single QUIC connection maps to a connection group in TAPS.

This has an especially large effect in when it comes to closing TAPS Connections,
since QUIC has no support for a graceful teardown of a single stream, without
either closing the entire Connection or waiting for the transfer to complete.
A QUIC stream is designed with HTTP/3 in mind, intended to transfer
a single object, with both the client and the server marking the transfer
from their respective ends as independently completed to close the stream.

While TCP connections can be closed one way, the typical behaviour is that
when it closed by one peer, then the other peer closes the connection
in conjunction, which differs from the expected one-way closing of QUIC
streams.

This means that a TAPS system has to make a choice for how to close individual
TAPS connections when using QUIC as the underlying protocol. Based on TCP, the 
expected behaviour
is that closing an individual Connection will close both the sending
and receiving side and start teardown. For QUIC, this can in theory be emulated by sending a \mono{FIN}
flag and a \mono{STOP_SENDING} frame. The peer then must respond with a \mono{RESET_STREAM}
frame if their stream is in a ready or sending state \cite{rfc9000}.
A \mono{RESET_STREAM} frame tears down a QUIC stream in the sending direction,
which in combination with the \mono{FIN} flag sent would completely close the frame.

Even though this handling of underlying QUIC streams in TAPS is superficially
closer to the expected behavior from TCP, \mono{STOP_SENDING} is not a graceful close and is
only
expected to be used on error conditions. For a graceful teardown of a single
stream only, the expected behaviour is both sides sending \mono{FIN} flags when
they are done sending.

In order to avoid unecessary error behaviour for QUIC the most natural mapping
for closing a TAPS Connection over QUIC is therefore to simply
send a \mono{FIN} flag in the sending direction and remain open in the receving direction.
This will appear to the appication in the form of it still having to monitor
the connection for incoming messages and it not receiving a closed()-callback until
after the peer has finished sending its data.

If a peer wants to gracefully close
a connection before all data has been received, the only way of handling it gracefully is to
close the entire QUIC connection. This naturally maps to the \mono{ConnectionGroup.close()}
functionality form RFC9622 \cite{rfc9622}.

If the application takes care of performing TAPS connection cloning in a consistent way,
it can constently use \mono{ConnectionGroup.close()} and achieve the same
expected behaviour no matter the underlying protocol in the TAPS system, this
is however an example of underlying functionality leaking through to how the application 
should interact with the API.

This mapping of QUIC connections to TAPS connection groups also affects how
CTaps handles the notification of path changes. In TAPS, the PathChange 
event is per-Connection \cite[Sec. 8.3.2]{rfc9622},
while QUIC migration happens for an entire Connection Group.

For CTaps this means that for each path change event every CTaps Connection is notified
separately for each change event, even though the event is always common for
the entire Connection Group. This could be mitigated by allowing registration
of callbacks on a per-Connection Group basis, but this does not match the current
TAPS API.

There are also features related to streams in QUIC which are not available
through a TAPS Connection.
One example of this is the one-way abort mentioned
earlier through \mono{STOP_SENDING}, which requests that the peer abort their
sending, and \mono{RESET_STREAM}
which aborts the local sending side \cite[Sec. 19.4, 19.5]{rfc9000}.
For aborting Connections, the TAPS API presents only the generic
\mono{Connection.abort()} \cite[Sec. 10]{rfc9622}.
A TAPS implementation therefore has no way of knowing whether the application
wants to abort sending, abort receiving, or both. The only natural conclusion is
therefore to abort both sending and receiving, leaving the granular QUIC control 
unavailable to the application.

Another stream functionality which is not exposed through CTaps is
initiating unidirectional streams. QUIC allows
applications to open streams for unidiractional sending \cite[Sec. 1]{rfc9000}.
A TAPS implementation has to assume that every new stream opened via \mono{Connection.clone()}
is bidirectional, although since the TAPS API is extensible, a parameter could
easily be added when Cloning a Connection to support this.

Despite not being able to inititate unidirectional sending, a TAPS implementation
is able to receive an unidirectional stream initiated by the peer as expected by QUIC.
It can infer from the stream ID that it is unidirectional \cite[Sec. 2.1]{rfc9000}
and mark it as closed for sending before presenting it to the appliaction.

Withing QUIC, there are also several transport parameters \cite[Sec. 18]{rfc9000} which are not
exposed through the TAPS API. TAPS does not feature a generic timeout value
wihch can be passed as \mono{max_idle_timeout} \cite[Sec. 10.1]{rfc9000}.
While TAPS does support timeouts for when sending data, for example through
the Timeout for Aborting Connection Connection Property \cite[Sec. 8.1.3]{rfc9622},
it does not feature a generic idle timeout for automatically closing a connection.

The TAPS API also does not support setting a maximum data to be received on either
a Connection Group or on a Connection, as supported on QUIC connections and streams
through the \mono{initial_max_data} and various \mono{initial_max_stream_data_*}
transport properties respectively.

In theory, an application
using TAPS could desire to set these, but they can also be assumed to
be handled by the underlying QUIC implementation and therefore not
necessarily be considered a gap in the TAPS API.
In CTaps' case, Picoquic provides
an optional \mono{picoquic_set_max_data_control} for
connection-level max data, but this is not surfaced through CTaps.
Picoquic does not provide a per-stream max data configuration.

CTaps does not expose any form of maximum data,
and as picoquic does not surface it as configuration options,
they are handled internally by the implementation

While TAPS does allow the application to control the number of Connections
in a Connection Group through the groupConnLimit Connection Property \cite[Sec. 8.1.9]{rfc9622},
it loses some granularity exposed through QUIC when controlling the number
of streams in a connection. QUIC allows control over both the maximum number
of both unidirectional and bidirectional streams through either the
\mono{initial_max_streams_uni} and \mono{initial_max_streams_bidi} or a
\mono{MAX_STREAMS} frame \cite[Sec. 18.2, 19.11]{rfc9000}. 

TAPS does also not feature a way to completely disable connection migration
as can be done in QUIC through the use of the \mono{disable_active_migration} transport
parameter \cite[Sec. 18.2]{rfc9000}. In TAPS, the only migration setting available to the application is
disabling active advertisment of local addresses through the \mono{advertisesAltaddr} Selection
Property \cite[Sec. 6.2.15]{rfc9622}.
The QUIC RFC specifically brings up the case where an application is behind
a simple load balancer, stating that a application should then disable
active migration \cite[Sec. 5.2.3]{rfc9000}, but this is not possible through
the TAPS API.

% TODO skriv om preferred_address, men bedre
On the server side, TAPS does not expose preferred addresses through its API.
These are used by QUIC when it wants to accept on a given address but wants
to migrate it to a new address shortly after the handshake \cite{rfc9000}.
Migration is however supported, so future support for preferred addresses can
easily be added to the API through a flag in a given local endpoint.

The remaining QUIC transport parameters which have no suitable mapping in TAPS
and cannot be assumed to be handled automatically are \mono{ack_delay_exponent},
\mono{max_ack_delay} and \mono{preferred_address} \cite[Sec. 18.2]{rfc9000}.

One way of handling this would be for the TAPS implementation to allow migration
to and from remote endpoints registered with the Connection. This would naturally
be quite limiting and would require you to know all remote interfaces of a
server before connecting to it, or discover them during the lifetime of the
Connection.

Certain features, such as stateless Reset are not exposed through the TAPS
API directly, but are expected to be handled by the underlying QUIC implementation,
so that there is no need to expose them in a TAPS system.

In the TAPS API ECN is marked as an UDP-lite specific feature, though this
is also useful for QUIC \cite{rfc9622}.

Additionally, when closing connections Picoquic exposes no way of passing
an error code to the peer.

\section{Discussion}
% Discuss handshake time
The handshake time shown in \cref{fig:racing-handshake} tells several stories.
Fist of all, it shows that there is little overhead in connection setup for
CTaps. Since the action is I/O-bound, the the network is the largest bottleneck
by far. It also shows a slightly biased comparison between the TCP and QUIC
handshakes as it does not include TLS, which would add another RTT onto
the TCP handshake and bring it in line with QUIC. It does however show that
there is a linear trend between all four combinations of server/client tried,
and that all of the handshake times grows in relation to their required number
of RTTs, nothing else. With the TCP implementation it also shows both a weakness
in the current implementation of CTaps, and a result of the desire to as much
of the benchmarking to a single CTaps client as possible, showing that modifying the
code is not necessary. For candidate racing CTaps currently uses a stagger delay
of 250 milliseconds between attempts as recommended by Happy Eyeballs V2, it does
however not implement interleaving as described in the same document
\cite{schinaziHappyEyeballsVersion2017}. 

As CTaps races local endpoints,
protocols, ALPN options and remote endpoints the consequences become
even more apparent for CTaps, as if it prefers an incompatible protocol
it will iterate through all possible combinations of local endpoints,
remote endpoints and ALPNs if applicable before trying the next protocol.
This means that if you are faced with a large number of endpoints, as you
might be after a hostname resolution for example, or when binding to an
ephemeral local endpoint, then you could face a multi second delay in CTaps
before even trying the correct protocol, if the highest ranked protocol is
not compatible to the server you are trying to connect to.

The reason for CTaps using ALPN values as separate protocol options is
two-pronged.
Primarily is done
in order to support 0-rtt. Since QUIC sesssion resumption relies on
resuming a previous session, one a single ALPN can be selected.
Therefore, each indivual ALPN value generates new racing Candidates
for each protocol which supports ALPN, although this is currently
only QUIC.

In theory, multiple ALPNs could be supported with a single candidate,
as you can select from a list of ALPNs after 1-RTT if you are not
using session resumption, but the current solution was selected since
it more followed the racing ideology of TAPS, as well as not having
to modify the racing flow to support 0-rtt. Unless 0-rtt is used,
having each ALPN create a new protocol candidate naturally creates
unecessary overhead in the form of unecessary candidates, which all
could be served by a single candidate, however the risk of this being
a significant overhead was deemed to be relatively small, since
applications are likely to not support a very large number of ALPNs,
and CTaps currently only supports a single protocol with ALPN support,
naturally limiting the number of generated combinations.

If this is found to be an unacceptable overhead in the future, optimizations
can be made, for example only generating a single candidate with several
ALPNs for each protocol if 0-rtt is not used by the application for the
current Connection, although this has been deemed to be out of scope for
CTaps.

If an attempt was to be made to make CTaps into a real-world production
ready system, then candidate interleaving would likely be one of the first
features to be implemeted. The biggest advantage of CTaps, that you can
access to several different transport layer protocols, without having to
implement racing and fallback logic yourself becomes moot if the racing
overhead is so large, that any other fallback system would be much faster.
It would have to be carefully considered exactly how these candidates were
to be interleaved. The applications does rank its most desired protocols,
so interleaving with no care as to the order could lead to a lower ranked
protocol being tried second after the most desirable one, to the application's
surprise. It would also be beneficial to include the local and remote endpoints
in the interleaving algorithm somehow. In the hypothetical situation that the
first candidates all try to connect to the same remote endpoint, but that endpoint
turns out to be unreachable, the current stagger delay would lead to a one
second delay before the next endpoint was tried.

In retrospect, the \qty{250}{\milli\second} delay recommended in Happy Eyeballs
V2 has a very specific background in that it is racing two protocol families.
While racing is one of the key features of TAPS, with the larger combination
of candidates where each one can represent an unreachable server or an incompatible
protocol, the staggering should likely be much shorter than \qty{250}{\milli\second}
for a TAPS system. This would allow CTaps to race its larger number of candidates
more efficiently, while still avoiding the undue strain on the network which
simultaneous racing can present.

Another mitagation against the tyranny of the stagger
delay would be to start connection attempts upon first resolution at least
one local and remote endpoint. Currently, CTaps waits until all local and remote
endpoints have been resolved before sorting the gathered candidates according
to the application preferences and then working its way down the list. Connection
attempts in CTaps can be made the same way as recommended in Happy Eyeballs V2, racing
as soon as a resolved endpoint comes back, with a \qty{50}{\milli\second} resolution delay
to give preferred addresses a chance to return before initiating an
attempt with a less desirable endpoint.

% Discuss candidate racing benchmark
The stagger delay does not show up only in the raw handshake measurement times,
but also in the candidate racing benchmark. Here the benchmark itself becomes
more contrived than needed because it needs to make the faster endpoint
preferred even with the long 250 ms stagger delay. This means making the
slow endpoint three times slower than the slow endpoint. While this
still serves to show the benefit gained on a suboptimal ordering from
a DNS resolution, it would become even more apparent if the delay
was lower.

% Discuss the small file transfer
While the above benchmarks show that CTaps achieves performance parity
with TCP sockets and Picoquic on small I/O bound applications, it does
not mean that there is no overhead at all with using CTaps. The small file
transfer is a constructed scenario used to show advantages with QUIC and
how CTaps makes them available through no extra cost to the user. In real-world
situations downloading large files like this is uncommon.
In 2025 Cloudflare
measured that the median number of packets sent to visitors to their service
was 12 \cite{sulemanahmadMeasuringCharacteristicsTCP2025}. A majority of 
connections never even leave slow start. However, it can also be argued that
thesee are exactly the connections which necessitate multistreaming. Since
such a large number of connections consist of only short data transfer, they
are never able to be scaled up to make use of the full available bandwidth.
Therefore, multiple streams using the same congestion control allows for a
far more efficient use of the network. This is also exactly what the small
file transfer benchmark shows, that the second transfer is so small that TCP
cannot compete with QUIC, since it has to go through slow start for the new data
transfer. This is a benefit of QUIC over TCP, not a benefit of TAPS vs a normal
QUIC implementation. However, the benefit of CTaps in this scenario is implicit,
as it shows that the very same client can connect to both a TCP and a QUIC server,
can be treated exactly the same, and the application does not have to care
whether the server it is connecting to supports QUIC or not.

% Discuss connection migration
\subsection{Connection Migration Benchmark} \label{sec:connection-mig-benchmark}
As mentioned above, one of the key features with QUIC is Connection Migration.
This is another aspect of QUIC hich helps utilize the avaiable bandwidth more
efficiently. If QUIC can successfully migrate from a dead path to a live one
without having to restart the connection and go through slow start again then
it leads to a much lower experienced latency by the user, for any transfer less
than the 10 packets sent during the initial RTT for TCP, in addition to avoiding
an additional handshake. It was therefore desirable to show that this feature
too was available through CTaps, and that there is minimal overhead. The results
are absolutely promising and show that the necessary complexity presented to the application
in order to achieve conneciton migration is very small. The graph shows familiar
sawtooth patterns of both CTaps, TCP and Picoquic having reached their steady-state
throughput. After the initiated path failure, TCP naturally fails, while Picoquic
and CTaps can both probe new paths, allowing them to continue until the transfer
completes.

When it comes to connection migration there is still a lot of functionality
which is not supported. This would not present extra complexity to the application,
but would lead to quite a bit of internal CTaps complexity. For example, if
the clients binds to a specific local endpoint with a specific port, there is no support
in the client currently for migrating away from this. This is passed to \mono{bind()}
and CTaps does not support opening new local sockets for migration. This naturally
limits the number of situations in which the client is actually able to migrate,
compared to what is desirable.

If adding this feature in the future, it should come as a part of a larger refactor,
removing the binding between individual connections and sockets in general. This would
also help listeners listen on several local endpoints, something which is not currently
supported in CTaps.


% Discuss performance in general
The discussion of the overhead of a networking library often becomes
quite moot if you are discussing situations in which the applications anyways
is I/O-bound. This is naturally the case with several of our benchmarks, where
the actual performance of the library does not matter much, if it will take a
minimum of \qty{50}{\milli\second} to set up a single QUIC handshake. This can
very clearly be seen in several ways in the handshake, small file transfer and
migration benchmarks. As with any library there is overhead to CTaps compared
to using purely picoquic, however this does not become apparent since those three
benchmarks are all I/O-bound, and do not place a load on the actual client or server.
Since both the Picoquic packet loop client and CTaps interact with the network in
a asynchronous manner, the system has time to perform any outstanding computation while
waiting for a response for the network. Since any computationally heavy work performed
on the data sent to or from the network would be in the domain of the client using CTaps,
the only way to truly show the overhead of CTaps would be to create a large amount of
concurrent clients and to transfer large amounts of data, in order to stress CTaps as
the interface between the network and the client. When it comes to high performance CTaps
is naturally limited by the single threaded event loop model of libuv and would therefore
not be able to fully make use of all the resources available to it on a host system.

% Discuss CTaps client complexity
The complexity of each benchmarking client as seen in \cref{tab:loc_complexity}
tells a very clear story. CTaps successfully abstracts away the complexity presented
to client applications interacting with the network. It does so not only when accessing
advanced features of QUIC such as connection migration, but also in the protocol
selection and the error handling itself. This is the primary aspect which sets
CTaps apart from other networking libraries, especially ones intending to help
make use of a single protocol such as QUIC. With CTaps, the complexity is moved
away from the application and presented in a generic way which can be reused
by any appliaction using the network. This does not only come with the added
simplicity, but also much easier migration to future protocols.

While CTaps is the most complex
individual client, it provides TCP and QUIC support, connection racing,
and several QUIC benefits at less than the combined complexity of the QUIC
and TCP clients. In order to develop a replacement of CTaps, the TCP and QUIC
clients would not only have to be combined, but would also have to implement
fallback logic, which is not accounted for in this table.

Several popular libraries exist which has significant feature overlap
with QUIC such as Valve's GameNetworkingSockets or ENet \cite{ValveGameNetworkingSockets}
\cite{Enet}. Both of these predate the finalization of the QUIC RFC in 2021 and
contain overlapping features which are also implemented through QUIC, such as connection
management over UDP or multistreaming (referred to as "Channels" in Enet). If a project relying on one of
these networking libraries wants to make use of QUIC features it has to switch
its entire networking stack to use a QUIC library. This is naturally a large
refactoring and is a limitation on how fast new networking features can propagate.
If a production ready TAPS-library had existed when the application was originally
designed, it could have abstracted away the actual underlying transport library desired
by the application, making switching to QUIC later essentially free. This comes in
addition to built-in fallbacks to the previous protocol should connection establishment fail.

This does not mean that CTaps is in a production ready state and should be immediately
switched to by any library currently utilizin ENet or GameNetworkingSockets. With
the large amount of complexity which is abstracted away by CTaps, it becomes a very
heavy dependency in and of itself. Applications may be wise in preffering to handle
their own networking stacks accepting future refactoring costs should they desire
to switch transport protocol. CTaps also has an issue of potential refactoring costs.
With the built in libuv event loop, CTaps forces your software to be written through a
async model which does its work through callbacks. This means that an application wanting
to use CTaps for underlying networking operations has to be designed with this in mind,
creating a very heavy reliancy on a single library and a single point of failure.

% Discuss QUIC under TAPS
Being an important research question it was important to find out if there
were large swaths of QUIC features which could not intuitively be presented
to an application through a TAPS system. While there are features which are
difficult to map, the absolute unavailable features are of a lesser importance.
The largest amount of missing features from QUIC in TAPS can be found in the
transport layer, such as with \mono{MAX_STREAM_DATA}, where this is not a
decision the application should make, but CTaps should make an informed decision.

The largest issues with creating a common abstraction for several different protocols like
TAPS is that there ends up being distinct concepts in each protocol which do
not necessarily map neatly into the shared abstraction layer concepts. This
is apparent in the differing behaviour of Connections in CTaps depending on
whether the underlying protocol is QUIC or not. with QUIC the Connection abstration
becomes the connection group and treating individual streams as connections
leads to unexpected behaviour, such as one-way closing by default except
when aborting the stream. This issue would likely be shared with any other
multistreaming protocol as well. This causes the application to have to interact
with TAPS in a way which does not reflect how they would interact with TCP,
for example always closing the entire connection group when done with a logical
connection to a server, to ensure that if the underlying protocl supports multistreaming
then the entire connection will be closed. While this is breaks slightly with 
with the principle of least surprised as described in RFC 9622, it provides
far greater flexibility and consistent behaviour to the downstream application
\cite{rfc9622}. It can also be added that
the only reason for this expectation of how to interact with the network
exists is due to TCP's long ubiquity as a transport protocol. This is
purely an appeal to tradition and should not be considered blocking in
situations where new useful features can be provided.

%discuss racing with UDP
When it comes to connection racing, it is a useful feature but can
be hard to map for non-connection oriented protocols leading to
unexpected or even undesirable behaviour. In CTaps this becomes an
issue with UDP. While the default preferences often lead users to TCP or
QUIC over UDP, if UDP is preferred but not required then there is no way for
racing to function properly. CTaps has chosen to assume that a UDP connection
is established as soon as local endpoint has been set up. This means that
if the remote is not actually genuinely UDP, the racing will tell the user
that the connection is ready, but the remote will be unreachable. An alternative
solution would be to send a test message and to wait for a reply, but
this relies on knowledge of the remote endpoint and that it will actually reply
to a ping. This risks designating legitimate endpoints as unresponsive. Since
the default preferences rank UDP lower than other supported protocols, it was
deemed that the risk of UDP being ranked higher than other protocols when
the endpoint does not support it was low enough that the simpler solution
of assuming an estbalished connection on opening the socket.

\section{Security}
The security of a TAPS system is reliant on the secuirity of the underlying protocols.
While there are several key security measures which should be taken before
CTaps could be used in a security-sensitive production context, such as adding support
for TLS, it also shows that it is possible to achieve equivalent security benefits
in a TAPS system as when using the underlying protocols directly.

% Handling of underlying credentials
Despite TAPS being designed before the official release of QUIC, it supports
the same security settings due to QUIC using TLS under the hood. It supports
both 0-RTT handshakes and the necesary client and server certicates for both
clients and servers using TLS. In addition it supports the required ALPNs and
and makes them accessible through Security Parameters. Security Parameters as
designed in TAPS and implemented by CTaps are by nature extensible, without
introducing breaking API changes.

Since TAPS provides a common API to several different protocols, there may
of course be situations where TAPS places a limitation on the security of the
underlying protocol, although this is likely to be more related to the specific
implementation than to TAPS specifically. One can for example imagine a scenario
where the key files should specifically be kept in memory and cannot be referred
to via a file name, then that could not be supported by CTaps as of the current
implementation. The TAPS API does however support this since it is generic
and only specifies server and client certificates through the use of
certificate bundles, but does not mandate the form which these bundles should take.
CTaps has chosen the form of referrring to files on disk, but there is no reason why
it could not be adopted into taking the memory aeddress of a buffer instead.

%TAPS downgrading security
The user of a TAPS system is also reliant on TAPS not silently downgrading
the required security of the application. Here TAPS needs to follow sane defaults
which default to mandating security, otherwise TAPS would not only be liable
to downgrade security, but the unsecure variants of protocols would be at
an advantage during candidate racing, since they would require fewer RTTs
to set up completely. For this case, TAPS would be wise in assuming that
there are few situations where it both secure and insecure protocols can
be used over the same link and should be raced together.

A secure connection
should be mandated by default and TAPS should only allow racing with only
secure protocols or with only insecure protocls. Security parameters should
then naturally be mandatory by default, only allowing insecure protocols if
explicitly enabled by the user. This is the ideology taken by TAPS as can be
seen in the Allowed Security Protocols security parameter, which mandates
that the default be \say{Implementation-specific best available security protocols}
\cite{rfc9622}.

In addition to mandating that TAPS implementations set the default allowed
security protocols to be the best available, RFC 9622 also mentions that applications
should use common safe defaults for security parameters
\cite{rfc9622}. This means that the
default should be requiring an encrypted connection. RFC 9622 also describe
the difference between disabled Security Parameters and opportunistic
Security Parameters. Disabled Security Parameters are an option in order
to support compatability with endpoints that do not support transport security
protocols, such as a pure HTTP website, while opportunistic Security Parameters
allow for the coexistence of secure and insecure transport layer protocols.
It is natural to assume that in cases where both insecure and secure transport
protocols are viable, any TAPS implementation would choose to prioritize the
secure protocol if performing candidate racing. For this case, the TAPS implementation
should also take the increased number of RTTs for the secure protocol into account
when calculating the stagger delay for candidate racing, in order to not give
the unencrypted protocols undue advantages.

While these considerations obviously should be made, and will likely be
arrived at independently by any serious production-ready TAPS implementations,
they add both complexity and risk for downstream applications of TAPS implementation,
as overlooked features or logic errors could lead to an insecure connection
being established even when not necessary.

These concerns are however very specific to the case of opportunistic
Security Parameters, where both insecure and secure transport protocols have
to be supported at the same time. This is likely a fairly small proportion
of use cases and most applications will be well served by mandating Security
Parameters and secure transport protocols as should be default according
to TAPS.

The concerns about potential differing behavior in different TAPS implementations
does however raise a real concern for security aware applications when using
a TAPS system. When introducing a large library such as TAPS which ends up forming
the entire flow and architecture of the application, the surface area for potential
attacks and errors becomes much larger. With CTaps, which is an incomplete TAPS implementation,
sitting at over 13 000 LoC for only implementation code, downstream applications cannot
be expected to comb through the source code with security experts in order to look
for potential vulnerabilities.

The most useful mitigation for issues like this would be for a community
effort to develop a true fully featured open source implementation of TAPS
which achieves genuine real world usage. This would create a real incentive
to find and patch as many vulnerabilities as possible in the library. This naturally
leads to a bit of an chicken and the egg problem, as no one wants to utilize
a library if no-one has verified the security of it, while no-one has any incentive
to verify the security of a library that no-one uses.

This purposefuly rather vague phrasing is typical of TAPS future-proofing, as 
it allows future implementations to update their defaults, if new security
protocols have arrived which are considered more secure than for example TLS 1.3
which is at the forefront at the time of writing.

Although TAPS aims to be as secure as possible by default, there are of course
tradeoffs to be made between the default configuration of an application
when it comes to feature-richness and security. With the relative complexity
and large number of configurations available within TAPS there is a genuine risk
of unknown unknowns. Security minded applications may not have combed through all
available configurations in TAPS, and therefore not be aware of the security risk.
An example of a potential unknown unknown is the potential of isolating sessions
through Security Parameters. By default this is set to false, allowing a TAPS implementation
to use an internal cache to optimize established connections. Using this cache
could lead to a leak of connection linkability, as observers over the wire could
observe different connections exhibiting the same behavior, and thereby induce that
they are made by the same client.

This issue with a potentially large configuration space is naturally present in all
software libraries, and is the reason behind sane defaults being extremely important
for open source libraries such as TAPS. TAPS strikes a balance between being extremely
cautious and being secure by default by allowing certain configurations to be the most
performant by default instead of necessarily the most secure. For most applications,
avoiding accidentally using unencrypted communication is extremely imporant, while
potential client fingreprinting is not as important. Since a choice has to be made
for TAPS, it can be assumed that extremely security conscious applications where
issues like fingerprinting are absolutely unacceptable will perform a careful comb
through of the security parameters, making the most cautious decision at every step.
The same cannot be assumed of a more casual application, having to comb through
Security Parameter configurations in order to find relatively large performance
gains to be made with relatively small security trade-offs.

%Racing reveal who uses TAPS? Other revealing options?
For applications where fingerprinting are absolutely unacceptable there are
also other issues related to TAPS, which cannot necessarily be traded away.
First of all, using a networking library which standardizes the interaction
with the network to a certain degree would naturally lead to potential fingreprinting
of clients. This is not something which can be configured away and is an
inherent property of software libraries whose behavior are exposed to the internet.
Without a large and diverse enough userbase, fairly accurate assumptions can
be made about the users behind a connection, if the underlying library being
used can be ascertained. This can be seen in real world examples such as with TOR,
where one of the key motivations behind it being open is the crowd effect.
If only certain groups were using TOR, then TOR would provide no real anonymity,
in that anyone seeing a TOR user, could reliably ascertain that they for
example were a CIA agent or a whistleblower \cite{collier2025torhistory}.

This fingerprinting is likely a potential for users of TAPS, especially
in the more unique factors of TAPS systems such as connection racing.
While racing in the format of Happy Eyeballs would be common, TAPS extended
racing of all combinations of candidates could be imagined to be a source
of client fingerprinting. As with TOR, security from this would come from
a wider user base of TAPS. Assuming that individual TAPS implementations
do not differ enough to be fingerprinted on their own, this risk would
grow smaller and smaller as not only the actual use of TAPS implementations
grow, but also as the concepts contained within TAPS become more common,
such as racing both endpoints and protocols, in addition to the IP addresses
as contained in Happy Eyeballs.

\section{Scope and Limitations}
CTaps is an exploratory work, intended to show how an implementation of a TAPS
system might look in a real world context, but is not intended to be used
in a production context.

The nature of CTaps being an exploratory work naturally brings with it
certain limitations unique to this kind of work. First of all, CTaps
has throughout had a larger focus on exposing as much as possible of
the TAPS api, without necessarily implementing the internal logic associated with
the actual API settings. Since CTaps is the first known TAPS implementation
since the RFCs were finalized, this was done to see exactly how the designed TAPS
RFC would actually map into code and that the different settings could actually be
presented well. Although this has proven successful in providing a public
API to reason about and guidance for future TAPS implementations, this is
not the ideal starting location when considering future real-world usage of
CTaps.

Applications coming across CTaps would likely find the API to be fairly confusing.
It looks like a more complete implementation than it actually is, and several
attempts to configure connections with unsupported settings, for example the multipath
policy connection property, would lead to no changed behavior from the applications point of view.

If CTaps were to be developed into a fully featured TAPS library, it would likely
be wise to first strip away the parts of the external API which currently have
no support in the internal implementation. This would allow a gradual addition of
features as they were supported, making it more clear to downstream applications exactly
which parts of the external library is mature enough to be used. This would also allow
future development to focus on more imporant improvements, instead of implementing
as much of the API as possible in order to make the features exposed by the API actually
real. The highest priority future work is described in \cref{sec:future-work}.

Another consequence of this push to make CTaps present as much of the API of TAPS
as possible is that most testing and development has been focused on the various
happy paths of the features, while not enough time or testing has been devoted
to testing various error paths, both for errors stemming from errors in user configuration
and from network errors.

Another key limitation with the current implementation of CTaps is that several key
features have been deemed to be out of scope, and is not presented in the API at
all. The largest of these are peer to peer connection establishment. This is however
the only major group of TAPS features which is totally absent, while there are
several other smaller features and quality of life pieces of the API not featured in
CTaps. Among the actual features, which present an actual lacking feature are that CTaps
cannot add endpoints in during the lifetime of a connection, CTaps does not feature
listening on more than a single connection and inaccuracies like that sent-callbacks
does not represent when a packet has been actully transeferred to the network,
but instead when a packet has been handed over to picoquic if QUIC is the underlying
protocol. QUIC may then cache the packets internally and the application
cannot therefore use the sent callbacks to reliably get the state of the send buffers.

Not being able to add endpoints to an active Connection places a real limitation on
a Connetion's ability to migrate, as there is no way to dynamically administer the
endpoints which can be migrated to or from. All Local and Remote endpoints has to be
passed to the Preconnection. QUIC connections can still migrate to new remote endpoints,
if migration is initiated by the peer, but there is no way to for example dynamically
add a new endpoint. It is not difficult to imagine a scenario where being able to add a
new endpoint dynamically would be useful for an application using CTaps. For example,
the underlying device could have connected to a wi-fi network during the lifetime of
the connection.
