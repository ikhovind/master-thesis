\subsubsection{Software Tools} \label{sec:softwaretools}
When performing benchmarks, the Linux utility \mono{tc-netem} was used~\cite{tc-netem} to emulate delay, bandwidth limitations
and potential jitter.

Netem could be invoked programmatically
from the Python testing script, which allowed a series network conditions to be used in
a single benchmark, for example the four RTTs
in the handshake benchmark.

In the migration test, iptables~\cite{iptables_man} was used in order to block a specific path based on
source IP, destination IP, protocol and destination port. The specific command
used when blocking a local path can be seen in \cref{lst:iptables-command}.
An equivalent command blocking
TCP data was also run, in order to illustrate the TCP connection failing after a path event.

This command adds a packet filter rule to the OUTPUT chain in the Linux kernel.
This chain is
run for locally generated packets~\cite{iptables_man}, and when a packet
matches the assigned rule, netfilter rejects it and propagates \mono{-EPERM} back through the network
stack~\cite{LinuxKernelNfHookSlow}. \mono{sendmsg()} then returns -1 with
errno set to \mono{EPERM} according to POSIX conventions.

For CTaps this works well as it could
report any \mono{sendmsg()} failure directly to Picoquic.
Picoquic would then communicate back to CTaps that the path has become unavailable through
a callback and CTaps could start probing new paths.

For the Picoquic client a small patch had to be applied in order to achieve
the same migration behavior. The Picoquic packet loop only
notifies the QUIC engine of a path failure if \mono{sendmsg()} fails with errno set to a value which implies
that the remote is unreachable. Internally in Picoquic this means one of the following errors: \mono{EAFNOSUPPORT},
\mono{ECONNRESET}, \mono{EHOSTUNREACH}, \mono{ENETDOWN} or \mono{ENETUNREACH}
\cite{PicoquicImpliesUnreachable}.

For the simulation it was desirable for the Picoquic client
to also treat \mono{EPERM} as implying that the remote is unreachable. Picoquic was therefore forked
and \mono{EPERM} was added to the list of socket errors which imply that
the remote is
unreachable~\cite{PicobrigtImpliesUnreachable}. The forked version of Picoquic was only used
for the Picoquic client when benchmarking. All other instances of Picoquic,
including the one used in CTaps, were fetched from the main Picoquic repository.

\begin{listing}[htbp]
    \begin{minted}[fontsize=\small]{bash}
iptables -A OUTPUT -s 127.0.0.1 -d 127.0.0.1 -p udp --dport 8080 -j REJECT
\end{minted}
\caption[iptables command forcing path migration.]{iptables command to block UDP data from local endpoint to file server on localhost with port 8080.}
\label{lst:iptables-command}
\end{listing}


\subsubsection{Measurement Approach}
Benchmarking data was gathered through a common benchmarking Python script.
This script is invoked with a specific set of command line arguments for each
benchmark and these invocations were stored in dedicated per-benchmark shell
scripts for the sake of reproducibility \footnote{The associated shell scripts
can be found in the \mono{benchmark/scripts/} folder of the main CTaps repository,
together with a README on how to reproduce the benchmarks.}.

\subsection{Results}

\subsubsection{Single Endpoint Handshake}

This benchmark can be seen in \cref{fig:racing-handshake} and
intends to show that a single CTaps client
can connect to multiple servers serving different protocols and
that there is no significant overhead in CTaps when connecting to a server.
It also shows the cost of a staggered racing delay as it exists now in CTaps.

This benchmark iterates
over delays of 25, 50, 75, and \qty{100}{\milli\second} to show the linear
growth between network delay and the handshake time.
The handshake times are very consistent since no jitter was added to the emulated
network. The graph is therefore based on a single measurement
for each data point. Since CTaps does not
support TLS, the TCP handshakes do not feature encryption,
and therefore require only one RTT to set up compared to the two RTTs required
for QUIC.

Because the handshake client used for CTaps here is the same as
the client used in \cref{sec:handshake-kde}, the client has a preference
for QUIC and is given two remote endpoints. This is the reason for the
\qty{500}{\milli\second} stagger delay between the CTaps TCP and QUIC
handshake times, since CTaps tries two remote endpoints for QUIC before trying TCP
and the stagger delay is a static \qty{250}{\milli\second} for each attempt.
The implications of this delay and possible improvements are discussed further
in \cref{sec:discussion}.

For each of the handshakes
the bandwidth was set to \qty{6.25}{\mega\byte\per\second}, although the bandwidth
does not have an effect on the time taken for the initial handshake.

The associated TCP server was a pure socket implementation and the QUIC server
used the Picoquic packet loop.

    \label{fig:racing-handshake}
\end{figure}

\subsubsection{Slow and Fast Endpoint Handshake} \label{sec:handshake-kde}
This benchmark can be seen in \cref{fig:handshake-kde} and intends to
show the benefits of candidate racing and how it helps find the
endpoint with the smallest delay.
This benchmark was modeled after the theoretical
scenario described in \cref{fig:racingSeqDiagram} in \cref{chap:impl}.

Two possible paths were used with different remote endpoints.
The slow endpoint had a delay of \qty{150}{\milli\second}
while the fast endpoint had a delay of \qty{50}{\milli\second}.
Bandwidth was limited to \qty{50}{\mega\bit\per\second}, although
this has no effect on the handshake times.

Both the Picoquic
and CTaps clients were given the slow endpoint as a first alternative.
Since CTaps supports candidate racing it was also given the fast endpoint
as a second alternative. This is intended to emulate the scenario after
for example a DNS resolution where multiple endpoints are available but
the client does not know which one is best.

In order to gather a distribution of handshake times, the test was run 200 times with a
jitter of 7 ms and a latency correlation of 25 \%.
In the scenario that this benchmark models,
candidate racing would lead to not only shorter
handshake times, but also a significantly smaller delay throughout the entire connection lifetime.

The theoretical ideal handshake times for each endpoint are featured
in the graph. With a 250 ms racing delay, 2 RTTs for a full handshake and delays of 50 and 150
milliseconds, the theoretical minimum handshake times work out to 
$\qty{50}{\milli\second} \cdot 4 + \qty{250}{\milli\second} = \qty{450}{\milli\second}$
and
$\qty{150}{\milli\second} \cdot 4 = \qty{600}{\milli\second}$ respectively.

\end{figure}

\subsubsection{Small File Transfer} \label{sec:smallfiletransfer}
This benchmark can be seen in \cref{fig:small-file-bar} and the
intention is to show that a TAPS system can benefit
from QUIC multistreaming while using a generic TAPS client
compatible with both TCP and QUIC.

This benchmark shows an example of lost capacity when performing Connection Cloning
over TCP when compared to QUIC, as modeled in \cref{fig:connection-clone-gains} in 
\cref{chap:impl}.
In order to show that the performance gap between TCP and QUIC grows with increasing delay
the benchmark sweeps across four delays of
25, 50, 75 and \qty{100}{\milli\second}.

The measurement consists of the time taken to transfer a file with 70 packets'
worth of data. For TCP this works out to
$70 \cdot 1460 \approx \qty{100}{\kilo\byte}$ after discounting the 20 byte IP and TCP
headers from a typical 1500 byte MTU. QUIC has slightly more overhead per packet,
but not enough to make a significant difference in the required file size.

Before the small file transfer was started, a large file of \qty{4.6}{\mega\byte} was
transferred in order to increase the CWND to be greater than the size of the small file.
This ensures that the small file transfer could be performed in a single RTT if the
CWND is reused. This large file is the same as the one used in the migration benchmark
described in \cref{sec:path-migration}.

To verify that the small file transfer can actually be completed in
a single RTT we need to check that the large file is large enough for the CWND to ramp up to at least 70 packets
and that the BDP of the link is large enough to carry the entire small file in a single RTT.

Assuming an initial CWND of 10 packets, the number of RTTs required for the CWND to grow to at least 70 during slow start is
$10 \cdot 2^{x-1} \geq 70 \Rightarrow x \geq 4 \text{ RTTs}$. During the minimum of 4 RTTs
$\sum_{n=1}^{4} 1460 \cdot 10 \cdot 2^{n-1} = \qty{219}{\kilo\byte}$ of data would
be transferred, which is well below the \qty{4.6}{\mega\byte} file used in the benchmark.

To verify that the BDP is large enough, we can calculate the minimum BDP for
70 packets which is
$70 \cdot 1500 \cdot 8 = \qty{840}{\kilo\bit}$.

In our benchmark, the smallest delay is \qty{25}{\milli\second}
giving an RTT of \qty{50}{\milli\second}.
The bandwidth is constant at \qty{50}{\mega\bit\per\second} which gives a minimum BDP of
$\qty{50}{\mega\bit\per\second} \cdot \qty{50}{\milli\second} = \qty{2.5}{\mega\bit}$.
Since the bandwidth is constant and BDP scales
with RTT we can therefore conclude that the BDP of the link is large enough
to carry the small file for all tested delay values.

  \label{fig:small-file-bar}
\end{figure}

\subsubsection{Path Migration} \label{sec:path-migration}
This benchmark can be seen in \cref{fig:migration-throughput} and intends
to show the benefits of connection migration
and that it is available with minimal code changes in CTaps.

In this benchmark, a \qty{4.6}{\mega\byte} file is transferred over an emulated
\qty{5}{\mega\bit\per\second} link with a delay of \qty{25}{\milli\second}.

The combination of file size and bandwidth was chosen to give the transfer enough
time to reach its steady-state and to still have data left
to transfer after recovering from the path failure.

This test uses two local IP addresses consisting of \mono{127.0.0.1} and \mono{127.0.0.2}.
All client successfully connect through \mono{127.0.0.1} and start the transfer,
but after five seconds the local endpoint is blocked through iptables as
described in \cref{sec:softwaretools}.

After the local endpoint is blocked both CTaps and the Picoquic client
detect this failure and
start the connection migration process, while TCP fails and cannot recover.

This benchmark was gathered with the CTaps and QUIC client described in \cref{sec:client-complexity},
showing that connection migration is made available through CTaps with minimal complexity
for the application.

\section{QUIC Under TAPS} \label{sec:quicUnderTaps}
As the first TAPS implementation to feature QUIC as
an underlying protocol option, finding out exactly which features of
QUIC are available through the default TAPS interface was one of the
key research questions for this thesis. Since TAPS is an interface to
a generic transport layer, TAPS concepts, terminology and intended functionality
naturally does not map one to one onto concepts, terminology and functionality
of QUIC.

The largest differences when it comes to the expected behavior when
for QUIC and TAPS stems from the
the Connection and the Connection Group.
Since multistreaming is handled through Connection
Groups in TAPS, this means that a single QUIC connection maps to a Connection Group
while a TAPS Connection maps to a QUIC stream.

This has an especially large effect in when it comes to closing TAPS Connections
since QUIC has no support for a graceful teardown of a single stream.
A QUIC stream is intended to transfer
a single piece of data, with both the client and the server marking the transfer
from their respective ends as independently completed to close the stream.

While TCP connections can be closed one way, the typical behavior is that
when it closed by one peer, then the other peer closes the connection
in conjunction, which differs from the expected one-way closing of QUIC
streams.

This means that a TAPS system has to make a choice for how to close individual
TAPS connections when using QUIC as the underlying protocol. Based on TCP, the 
expected behavior
is that closing an individual Connection will close both the sending
and receiving side and start teardown. For QUIC, this can in theory be emulated by sending a \mono{FIN}
flag and a \mono{STOP_SENDING} frame. The peer then must respond with a \mono{RESET_STREAM}
frame if their stream is in a ready or sending state~\cite{rfc9000}.
A \mono{RESET_STREAM} frame tears down a QUIC stream in the sending direction,
which in combination with the \mono{FIN} flag sent would completely close the stream.

Even though this handling of underlying QUIC streams in TAPS is superficially
closer to the expected behavior from TCP, \mono{STOP_SENDING} is not a graceful close and is
only
expected to be used on error conditions. For a graceful teardown of a single
stream only, the expected behavior is both sides sending \mono{FIN} flags when
they are done sending.

In order to avoid unnecessary error behavior for QUIC the most natural mapping
for closing a TAPS Connection over QUIC is therefore to simply
send a \mono{FIN} flag in the sending direction and remain open in the receiving direction.
This will appear to the application in the form of it still having to monitor
the connection for incoming messages and it not receiving a closed()-callback until
after the peer has finished sending its data.

If a peer wants to gracefully close
a connection before all data has been received, the only way of handling it gracefully is to
close the entire QUIC connection. This naturally maps to the \mono{ConnectionGroup.close()}
functionality from RFC9622~\cite{rfc9622}.

If the application takes care of performing TAPS connection cloning in a consistent way,
it can consistently use \mono{ConnectionGroup.close()} and achieve the same
expected behavior irrespective of the underlying protocol in the TAPS system, this
is however an example of underlying functionality leaking through to how the application 
should interact with the API.

This mapping of QUIC connections to TAPS connection groups also affects how
CTaps handles the notification of path changes. In TAPS, the PathChange 
event is per-Connection~\cite[Sec. 8.3.2]{rfc9622},
while QUIC migrations is per-Connection Group.

For CTaps this means that for each path change event every CTaps Connection is notified
separately, even though the event is always common to an
entire Connection Group. This could be mitigated by allowing registration
of callbacks on a per-Connection Group basis, but this not featured
in the current TAPS API.

There are also features related to streams in QUIC which are not available
through a TAPS Connection.
One example of this is the one-way abort mentioned
earlier through \mono{STOP_SENDING}, which requests that the peer abort their
sending, and \mono{RESET_STREAM}
which aborts the local sending side~\cite[Sec. 19.4, 19.5]{rfc9000}.
For aborting Connections, the TAPS API presents only the generic
\mono{Connection.abort()}~\cite[Sec. 10]{rfc9622}.
A TAPS implementation therefore has no way of knowing whether the application
wants to abort sending, abort receiving, or both. The only natural conclusion is
therefore to abort both sending and receiving, leaving the granular QUIC control 
unavailable to the application.

Another stream functionality which is not exposed through CTaps is
initiating unidirectional streams~\cite[Sec. 1]{rfc9000}.
Currently TAPS implementations have to assume that every new stream opened via \mono{Connection.clone()}
is bidirectional, although since the TAPS API is extensible a parameter could
easily be added when Cloning a Connection to support this.

Despite not being able to initiate unidirectional sending, a TAPS implementation
is able to receive an unidirectional stream initiated by the peer as expected by QUIC.
It can infer from the stream ID that it is unidirectional~\cite[Sec. 2.1]{rfc9000}
and mark the resulting Connection as closed for sending before presenting it to the application.

Withing QUIC, there are also several transport parameters~\cite[Sec. 18]{rfc9000} which are not
exposed through the TAPS API. TAPS does not feature a generic timeout value
which can be passed as \mono{max_idle_timeout}~\cite[Sec. 10.1]{rfc9000}.
While TAPS does support timeouts for when sending data, for example through
the Timeout for Aborting Connection Connection Property~\cite[Sec. 8.1.3]{rfc9622},
it does not feature a generic idle timeout for automatically closing a connection.

The TAPS API also does not support setting a maximum data to be received on either
a Connection Group or on a Connection, as supported on QUIC connections and streams
through the \mono{initial_max_data} and various \mono{initial_max_stream_data_*}
transport properties respectively.

In theory, an application
using TAPS could desire to set these, but they can also be assumed to
be handled by the underlying QUIC implementation and therefore not
necessarily be considered a gap in the TAPS API.
In CTaps' case, Picoquic provides
an optional \mono{picoquic_set_max_data_control} for
connection-level max data, but this is not surfaced through CTaps.
Picoquic does not provide a per-stream max data configuration.

While TAPS does allow the application to control the number of Connections
in a Connection Group through the groupConnLimit Connection Property~\cite[Sec. 8.1.9]{rfc9622},
it loses some granularity exposed through QUIC when controlling the number
of streams in a connection. QUIC allows control over both the maximum number
of both unidirectional and bidirectional streams through either the
\mono{initial_max_streams_uni} and \mono{initial_max_streams_bidi} or a
\mono{MAX_STREAMS} frame~\cite[Sec. 18.2, 19.11]{rfc9000}. 

QUIC path migration is supported in TAPS, and the migration
policy can be controlled through the
\mono{multipath} Selection Property.
For QUIC servers it can be used to set the
\mono{disable_active_migration} transport parameter~\cite[Sec. 18.2]{rfc9000},
while for clients it can be used to simply disable active migration.

The combination of \mono{multipath} and the \mono{multipathPolicy} 
Connection Property~\cite[Sec. 8.1.7]{rfc9622} gives
a more granular control of the expected behavior when using
multipath than is featured in QUIC.
This is expected from TAPS
and it is clearly documented in RFC~9622 that Connection Properties
will only have an effect if they can actually be mapped onto the
underlying connection~\cite[Sec. 5]{rfc9622}. 

An overview of the
expected behavior according to RFC~9622 can be seen in
\cref{tab:multipath-combinations}.
The \mono{interactive} and \mono{aggregate} multipath policies require
simultaneous use of multiple paths, which is not supported in RFC~9000 QUIC.
The IETF QUIC multipath extension~\cite{ietf-quic-multipath-21} is currently
under development and would enable these policies to be fully supported in a
future version of CTaps.

On the server side, TAPS does not expose the \mono{preferred_address} 
transport parameter~\cite[Sec. 18.2]{rfc9000} directly.
This parameter is used by a QUIC server when it wants to accept on a given address but wants
to migrate it to a new address shortly after the handshake~\cite[Sec. 9.6]{rfc9000}.
A workaround could be made by setting \mono{advertisesAltaddr} and
interpreting a second Local Endpoint as a preferred address for a Listener, but this
would be a fairly large break with the expected behavior from other protocols.

The remaining QUIC transport parameters which have no suitable mapping in TAPS
and cannot be assumed to be handled automatically are \mono{ack_delay_exponent} 
and \mono{max_ack_delay}~\cite[Sec. 18.2]{rfc9000}, which both can be
assumed to be reasonably handled through the underlying QUIC implementation.

There are also other features such as stateless reset which are not exposed through the TAPS
API directly, but are expected to be handled by the underlying QUIC implementation,
so that there is no need to expose them in a TAPS system.

and how they may into CTaps and TAPS in general can be seen in
\cref{tab:quic-taps-mapping}.

\section{Discussion} \label{sec:discussion}
\subsection{Single Endpoint Handshake Benchmark} \label{sec:single-endpoint-handshake-discussion}
When it comes to CTaps, the results in \cref{fig:racing-handshake} are promising
but also illustrate some weaknesses.
The benchmark shows that 
there is little overhead in connection setup for
CTaps and that a single CTaps client can successfully connect to multiple
servers with different protocols, due to candidate racing.

The benchmark does however also show that candidate racing
is not implemented optimally in CTaps.
Currently, CTaps uses static stagger delay
of 250 milliseconds between attempts as recommended by Happy Eyeballs V2~\cite{schinaziHappyEyeballsVersion2017}, but it does
not implement interleaving as described in the same document.
This means that all
combinations involving the most desirable protocol are attempted
before a new protocol is attempted.

This easily becomes a very significant delay, as the number of
raced parameters easily becomes very high when racing both
path combinations and protocol options.

If even a simple form of interleaving was implemented, then the
stagger delay present for TCP could be halved from \qty{500}{\milli\second}
to \qty{250}{\milli\second}.

This means that if an application is faced with a large number of Endpoints,
as it for example might be after a remote hostname resolution,
then it could face a multi second delay in CTaps
if the most desirable protocol is
not compatible to the server it is trying to connect to.

This lack of protocol interleaving is made worse by CTaps treating
each individual ALPN value as a separate protocol option, as this
inflates the number of candidates for racing for all protocols which
support ALPN. This design choice was however necessary 
in order to support 0-RTT as session resumption relies on
using the same ALPN as the previous session. In theory it
would be possible to only generate candidates for each ALPN like
this if 0-RTT was supported according to the Transport Properties
and Security Parameters set by the user, but this was deemed to be out
of scope for CTaps.

Having access to several different transport layer protocols
becomes far less appealing if the racing
delays introduce multi-second handshake delays, limiting
the current appeal of CTaps.

The \qty{250}{\milli\second} delay recommended in Happy Eyeballs
V2 has a very specific background in that it is racing two protocol families
on a single local path.
With the larger number of
possible combinations of candidates, a TAPS implementation should heed the
more closely follow the recommendations for racing on multihomed hosts as
described in~\cite{rfc8421}.

Other potential mitigations for this problem with interleaving is discussed further
in \cref{sec:future-work}.

In addition to the stagger delay, the connection establishment time of
a CTaps client is also vulnerable to slow DNS resolutions,
although this does not show in the handshake time benchmark.
Currently, CTaps waits until all local and remote
endpoints have been resolved before sorting the gathered candidates according
to the application preferences and then working its way down the list.
This could be optimized by integrating candidate gathering and candidate
racing more tightly, utilizing a Resolution Delay similar to the one described
in Happy Eyeballs~\cite{schinaziHappyEyeballsVersion2017}.

% Discuss candidate racing benchmark
\subsection{Slow and Fast Endpoint Handshake Benchmark}
The stagger shows up not only in the raw handshake measurement times,
but also in the candidate racing benchmark. Here the benchmark itself becomes
more contrived than needed because it needs to make the faster endpoint
preferred even with the long 250 ms stagger delay. This means making the
delay for the slow endpoint three times larger than for the fast
endpoint. While this
still serves to show the benefit gained on a suboptimal ordering from
a DNS resolution, the difference in the two endpoints has to be
unrealistically large due to the CTaps-specific stagger delay.

% Discuss the small file transfer
\subsection{Small File Transfer Benchmark}
The small file benchmark shows that CTaps can utilize Connection
Cloning to make use of QUIC multistreaming and achieve performance
parity with a Picoquic client. This is however on a 
small I/O bound applications and it does
not mean that there is no overhead with using CTaps.

The small file
transfer is a constructed scenario and in real-world
situations downloading large files like required for this benchmark this is uncommon.

In 2025 Cloudflare
measured that the median number of packets sent to visitors to their service
was 12~\cite{sulemanahmadMeasuringCharacteristicsTCP2025}. A majority of 
connections never even leave slow start.

However, it can also be argued that
these are exactly the connections which necessitate multistreaming. Since
such a large number of connections consist of only short data transfer, they
are never able to be scaled up to make use of the full available bandwidth.
Therefore, multiple streams using the same congestion control allows for a
far more efficient use of the network.

However, the benefit of CTaps over a normal QUIC client is more implicit
than explicit in this scenario,
as it shows that the very same client can connect to both a TCP and a QUIC server,
can be treated exactly the same, and an application does not have to care
whether the server it is connecting to supports QUIC or not.

% Discuss connection migration
\subsection{Connection Migration Benchmark} \label{sec:connection-mig-benchmark}
This benchmark shows that connection migration is available through
CTaps and that CTaps can make use of another QUIC-specific advantage
from a client that also supports TCP.

While the benchmark shows that connection migration is currently supported
with little overhead, there are however still many scenarios which are not
supported by CTaps. These would not present extra complexity to the
application, but would lead to quite a bit of internal CTaps complexity.

As discussed in \cref{sec:connection-migration-impl},
the current implementation is a workaround, and there is currently no support
for a CTaps client to migrate from a socket that is bound to a non-ephemeral IP.
This naturally limits the number of situations in which the client is actually able to migrate.

% Discuss performance in general
\subsection{Performance}
The discussion of the overhead of a networking library becomes
unproductive in situations where applications anyways
are I/O-bound, which is naturally the case in the CTaps benchmarks.
They can therefore be used to draw the conclusion that there
is no significant overhead between CTaps and the underlying protocols,
but they cannot for example be used that a CTaps client would perform
under load as a pure QUIC or TCP client.

Since both the Picoquic packet loop client and CTaps interact with the network in
a asynchronous manner, the system has time to perform any outstanding computation while
waiting for a response for the network. Since any computationally heavy work performed
on the data sent to or from the network would be in the domain of the application using CTaps,
the only way to truly show the overhead of CTaps would be to create a large amount of
concurrent clients and to transfer large amounts of data in order to stress CTaps as
the interface between the network and the client. When it comes to high performance CTaps
is naturally limited by the single threaded event loop model of libuv and would therefore
not be able to fully make use of all the resources available to it on a host system.

% Discuss CTaps client complexity
\subsection{Client Complexity}
The complexity of each benchmarking client as seen in \cref{tab:loc_complexity}
tells a very clear story and CTaps successfully abstracts away the complexity presented
to client applications interacting with the network. It does so not only when accessing
advanced features of QUIC such as connection migration, but also in the protocol
selection and the error handling itself. This is the primary aspect which sets
CTaps apart from other networking libraries, especially ones intending to help
make use of a single protocol such as QUIC. With CTaps, the complexity is moved
away from the application and presented in a generic way which can be reused
by any application using the network. This does not only come with the added
simplicity, but also much easier migration to future protocols.

While CTaps is the most complex
individual client by lines of code, it provides TCP and QUIC support, connection racing,
and several QUIC benefits at less than the combined complexity of the QUIC
and TCP clients. A combined TCP and QUIC client would likely feature
some shared logic, so the naive sum in \cref{tab:loc_complexity} can
be viewed as an upper bound for the non-CTaps clients. 

However, in order to develop a replacement of CTaps the TCP and QUIC
clients would not only have to be combined, but would also have to implement
fallback logic which is not accounted for in this table.

It is clear that the functionality present in QUIC has been desired
for a long time, and
several popular libraries exist which have significant feature overlap
with QUIC. Two examples are Valve's GameNetworkingSockets~\cite{ValveGameNetworkingSockets} and ENet
\cite{Enet}. Both of these predate the finalization of the QUIC RFC~in 2021 and
contain overlapping features which are also implemented through QUIC, such as connection
management over UDP or multistreaming (referred to as "Channels" in Enet).

If an application relying on one of
these networking libraries wants to make use of QUIC it has to switch
its entire networking stack to use a QUIC library. This is naturally a large
refactoring and is a limitation on how fast new networking features can propagate.
If a production ready TAPS-library had existed when this hypothetical application was originally
designed it could have abstracted away the actual underlying transport library and
made a later switch to a newer protocol such as QUIC essentially free. This comes in
addition to built-in fallbacks to the previous protocol should connection establishment fail
for QUIC.

With
the large amount of complexity which is abstracted away by CTaps, it becomes a very
heavy dependency and therefore quite a large risk for downstream applications
which become heavily reliant on the correctness of CTaps.
With the built in libuv event loop, CTaps forces your software to be written through an
async model which does its work through callbacks. This means that an application wanting
to use CTaps for underlying networking operations has to be designed with this in mind,
from the very beginning, creating a barrier for adoption in existing libraries.

% Discuss QUIC under TAPS
\subsection{QUIC under TAPS}
Since the TAPS RFCs were written before the QUIC design was finalized
in RFC~9000~\cite{rfc9000}, part of the motivation for developing CTaps
was to find out which QUIC features could not be presented through the
TAPS API.

The largest issues with creating a common abstraction for several different protocols like
TAPS is that there ends up being distinct concepts in each protocol which do
not necessarily map neatly into the shared abstraction layer concepts. This
is apparent in the differing behavior of Connections in CTaps depending on
whether the underlying protocol is QUIC or not.

With QUIC the Connection abstraction
becomes the connection group and treating individual streams as connections
leads to unexpected behavior, such as one-way closing by default except
when aborting the stream. This causes the application to have to interact
with TAPS in a way which does not reflect how they would interact with TCP,
for example always closing the entire connection group when done with a logical
connection to a server, to ensure that if the underlying protocol supports multistreaming
then the entire connection will be closed.

While this breaks slightly with 
with the principle of least surprise as described in RFC~9622~\cite{rfc9622}, it provides
greater flexibility and consistent behavior to the downstream application.

\subsection{Racing with UDP}
When it comes to connection racing, it is a useful feature but can
be hard to map for non-connection oriented protocols such as UDP, leading to
unexpected or even undesirable behavior.

While the default preferences in CTaps prefers TCP or
QUIC over UDP, if UDP is preferred but not required then there is no way for
racing to function properly.

Since UDP has no handshake to complete, CTaps has chosen to assume that a UDP connection
is established as soon as local endpoint has been set up. This means that
if the remote is does not support UDP, the racing will be unable to detect
this and UDP can potentially win the candidate racing,
but the remote will be unreachable.

An alternative
solution would be to send a test message and to wait for a reply, but
this relies on the remote peer having implemented this functionality and
therefore  risks designating legitimate endpoints as unresponsive. Since
the default preferences rank UDP lower than other supported protocols, it was
deemed that for CTaps the risk of UDP being ranked higher than other protocols when
the endpoint does not support it is quite low. The simple solution of
of assuming an established connection on opening the socket is adequate was therefore
chosen, for lack of a better option.

\section{Security}
The security of a TAPS system is reliant on the security of the underlying protocols.
While there are several key security measures which should be taken before
CTaps could be used in a security-sensitive production context, CTaps
also shows that it is possible to achieve equivalent security benefits
in a TAPS system as when using the underlying protocols directly.

% Handling of underlying credentials
Despite TAPS being designed before the official release of QUIC, it supports
the same security settings due to QUIC using TLS under the hood. It supports
both 0-RTT handshakes and the necessary client and server certificates for both
clients and servers using TLS. In addition it supports the required ALPNs and
and makes them accessible through Security Parameters. Security Parameters as
designed in TAPS and implemented by CTaps are by nature extensible, without
introducing breaking API changes.

Since TAPS provides a common API to several different protocols, there may
of course be situations where TAPS places a limitation on the security of a system.
This is however likely to be more related to the specific
TAPS implementation than to TAPS in general. One can for example imagine a scenario
where the key files should specifically be kept in memory and cannot be referred
to via a file name, then that could not be supported by CTaps as of the current
implementation. The TAPS API does however support this since it is generic
and only specifies server and client certificates through the use of
certificate bundles, but does not mandate the form which these bundles should take.
CTaps has chosen the form of referring to files on disk, but there is no reason why
it could not be adopted into taking the memory address of a buffer instead.

%TAPS downgrading security
The user of a TAPS system is also reliant on TAPS not silently downgrading
the required security of the application. Here TAPS needs to follow sane defaults
which default to mandating security, otherwise TAPS would be liable
to downgrade security since the insecure variants of protocols are at
an advantage during candidate racing since they require fewer RTTs.

In addition to mandating that TAPS implementations set the default allowed
security protocols to be the best available, RFC~9622 also mentions that applications
should use common safe defaults for security parameters
\cite{rfc9622}. This means that the
default should be requiring an encrypted connection. RFC~9622 also describe
the difference between disabled Security Parameters and opportunistic
Security Parameters.

Disabled Security Parameters are an option in order
to support compatibility with endpoints that do not support transport security
protocols, such as a pure HTTP website, while opportunistic Security Parameters
allow for the coexistence of secure and insecure transport layer protocols.
It is natural to assume that in cases where both insecure and secure transport
protocols are viable, any TAPS implementation would choose to prioritize the
secure protocol if performing candidate racing. For this case, the TAPS implementation
should also take the increased number of RTTs for the secure protocol into account
when calculating the stagger delay for candidate racing, in order to not give
the unencrypted protocols undue advantages.


Another concern with TAPS is that when applications utilize a large library
like a full TAPS implementation would be, the surface area for potential
attacks and errors becomes much larger.

The most useful mitigation for issues like this would be for a community
effort to develop a true fully featured open source implementation of TAPS
which achieves genuine real world usage. This would create a real incentive
to find and patch as many vulnerabilities as possible in the library.

Although TAPS aims to be as secure as possible by default, there are of course
trade-offs to be made between the default configuration of a TAPS system 
when it comes to feature-richness and security. With the relative complexity
and large number of configurations available within TAPS there is a genuine risk that
security minded applications may not have combed through all
available configurations in TAPS and therefore not be aware of a security risk.

An example of a potential unknown unknown is the potential of isolating sessions
through Security Parameters. By default this is set to false, allowing a TAPS implementation
to use an internal cache to optimize established connections. Using this cache
could lead to a leak of connection linkability, as observers over the wire could
observe different connections exhibiting the same behavior, and thereby induce that
they are made by the same client.

TAPS strikes a balance between being extremely
cautious and being secure by default by allowing certain configurations to be the most
performant by default instead of necessarily the most secure. Since a choice has to be made
for TAPS it can be assumed that extremely security conscious applications where
issues like fingerprinting are absolutely unacceptable will perform a careful comb
through of the security parameters, making the most cautious decision at every step.
The same cannot be assumed of a more casual application, having to comb through
Security Parameter configurations in order to find relatively large performance
gains to be made with relatively small security trade-offs.

%Racing reveal who uses TAPS? Other revealing options?
For applications where fingerprinting are absolutely unacceptable there are
also other issues related to TAPS, which cannot necessarily be traded away.
First of all, using a networking library which standardizes the interaction
with the network to a certain degree would naturally lead to potential fingerprinting
of clients. This is not something which can be configured away and is an
inherent property of software libraries whose behavior are exposed to the internet.
Without a large and diverse enough user base, fairly accurate assumptions can
be made about the users behind a connection, if the underlying library being
used can be ascertained. This becomes a larger risk if the behavior
if the underlying library is fairly unique, such as the extensive
candidate racing in TAPS.

\section{Scope and Limitations}
CTaps is an exploratory work, intended to show how an implementation of a TAPS
system might look in a real world context, but is not intended to be used
in a production context.

CTaps has throughout had a larger focus on exposing as much as possible of
the TAPS API, without necessarily implementing the internal logic associated with
the actual API settings. 
This was done to see exactly how the designed TAPS
RFC~would actually map into code and that the different settings could actually be
presented well. Although this has proven successful in providing a public
API to reason about and guidance for future TAPS implementations, this is
not the ideal starting location when considering future real-world usage of
CTaps.

Applications coming across CTaps would likely find the API to be fairly confusing.
It looks like a more complete implementation than it actually is, and several
attempts to configure connections with unsupported settings, for example the multipath
policy connection property, would lead to no changed behavior from the applications point of view.

Another consequence of this push to make CTaps present as much of the API of TAPS
as possible is that most testing and development has been focused on the various
happy paths of the features, while not enough time or testing has been devoted
to testing various error paths, both for errors stemming from errors in user configuration
and from network errors.

Another key limitation with the current implementation of CTaps is that several key
features have been deemed to be out of scope, and is not presented in the API at
all. The largest of these are peer to peer connection establishment. This is
the only major group of TAPS features which is totally absent from CTaps.

Among the smaller features not present in CTaps a prominent examples are that CTaps
does not support adding endpoints in during the lifetime of a connection and that
CTaps does not feature
listening on more than a single connection.

Not being able to add endpoints to an active Connection places a real limitation on
a Connection's ability to migrate, as there is no way to dynamically administer the
endpoints which can be migrated to or from. All Local and Remote endpoints has to be
passed to the Preconnection. QUIC connections can still migrate to new remote endpoints,
if migration is initiated by the peer, but there is no way to for example dynamically
add a new endpoint.

CTaps also contain inaccuracies which deviate from expected behavior, for example
that \mono{sent()}-callbacks
does not represent when a packet has been actually transferred to the network,
but instead when a packet has been handed over to Picoquic if QUIC is the underlying
protocol. QUIC may then cache the packets internally and the application
cannot therefore use the sent callbacks to reliably get the state of the send buffers.

\section{Alternative QUIC Backends} \todo{This section is valuable but needs to be tightened by a lot}
Since QUIC is an exploratory work an prioritized simplicity over raw performance
when selecting the QUIC library to use there are likely large performance gains
to be made by selecting a more performance oriented library than Picoquic. While
CTaps maintains a generic interface to the underlying protocols and the switch
would therefore in theory consist of only rewriting the two QUIC files themselves,
This requires that the QUIC library would be compatible with the "QUIC engine"
mode of operation as epected by CTaps, in order to integrate it into the libuv
event loop.

An alternative more performance-oriented QUIC library in C is for example
be MsQuic~\cite{MsQuic}. MsQuic is a cross-platform C implementation of QUIC
developed by Microsoft. With MsQuic being a cross-platform library it would
further enable CTaps to become cross-platform as well, since the core dependencies
of Libuv and MsQuic would be cross-platform. For CTaps, the QUIC
implementation of the protocol interface is already by far the most complicated
of the current protocol implementations as seen in \cref{tab:loc_complexity}.
MsQuic is a far larger library than Picoquic at $\sim135~000$ LoC compared to
Picoquic's $\sim84~000$. While this means that more features and performance
is available, this would also likely lead to a larger amount of complexity
in the configuration from CTaps' point of view, inflating the difference
in the complexity in the various protocol implementations even further.

In normal execution mode MsQuic owns the event loop, notifying
the application of updates and asking for input through registered
callbacks. This mirrors how CTaps interacts with its downstream
applications. In theory this execution mode could be integrated
into CTaps, but it would require using async handles in libuv [cite], since
MsQuic specifically require that the callback functions do not
introduce significant delay~\cite{msquic_execution}. This would
be a significant architectural change in CTaps and introduce
a large amount of complexity.

MsQuic also features an alternative execution mode where the
application, in this case CTaps, creates the threads it will
run on and periodically call into MsQuic to perform thhe necessary
actions. This mirrors the timed wakeups required by Picoquic
and could be integrated with the existing functionality in
CTaps. The largest difference with the current implementation
is that CTaps would use create a event queue object
through picoquic instead of using a timer to wake up at set times.
CTaps would then have to monitor this event queue object with \mono{uv_poll_t}
instead of using a \mono{uv_timer_t} handle as with Picoquic.
Examples from the MsQuic documentation creating a event
queue object and draining it can be seen in \cref{lst:msquicCustomEpoll}
and \cref{lst:msquicCustomLoop}.

While this shows that CTaps is not very tightly bound
to Picoquic itself, it is very tightly bound to libuv and
requires any additional libraries to be able to integrate
well with an existing event loop. For example if 
MsQuic did not feature the alternative execution mode
then integrating it with CTaps would be a large undertaking,
likely changing the entire internal architecture to
make the updates form QUIC flow through libuv async
handles.
