\chapter{Design} \label{chap:design}
This chapter will cover the design decisions which has affected
CTaps, both internally and for the external API. The overarching
design goal was to conform to the API described in RFC~9622~\cite{rfc9622},
while exploring how to best expose as much QUIC functionality as possible
through the TAPS API. The chapter first describes the overall scope
of CTaps, before expanding on the chosen dependencies, internal architecture
and the external API presented to applications.

\section{Choice of Scope}
One of the key research questions motivating the development of CTaps is exactly
how well the TAPS API as it exists currently can be presented programmatically.
CTaps therefore aims to present as much of the TAPS API as possible, despite
implementing several features as no-ops.

As it stands now, CTaps implements a large subset of the TAPS API and presents
several of the most beneficial features of the TAPS API such as candidate gathering
and candidate racing, as well as support for TCP, UDP and QUIC. In addition
it presents all of the core objects of the TAPS API, such as Connections, Listeners and
Connection Groups. It allows the application to set all Transport Properties listed
in RFC~9622, although not all properties are supported internally and setting them has no effect.
When performing
racing the most desirable protocols are attempted first and staggered racing
is performed for less desirable protocols, as described in RFC~9623
\cite{brunstromImplementingInterfacesTransport2025}.
\todo{Check in on this later, describe how large of a subset}

The parts of the TAPS API which were prioritized for CTaps are common features related
to establishing client and server connections, transferring data and managing
the lifetime of Connections. The highest prioritized internal TAPS features
were the most unique parts of the TAPS API, which only a software library
presenting a generic interface to the transport layer could present. This
is naturally candidate gathering and candidate racing, which allows a single
client to be able to connect to multiple different endpoints using different
protocols.

Certain parts of the TAPS API are not present in CTaps, the
largest of which is the peer to peer connection functionality.
Implementing this was deemed to be out of scope, while presenting
the API as a no-op was deemed to be of little value, since CTaps would not be able to predict
how the internal rendezvous implementation would affect the external API.

Other functionality which is not present in the CTaps API is, among others,
being able to add endpoints to an active Connection.
This is omitted because an actual implementation was deemed
out of scope, and presenting it as a no-op was deemed to be of little value.

Overall, CTaps serves as a reference
to show the architectural and design decisions which are necessary when
implementing the TAPS API, as well as to show which features of QUIC can
be exposed through TAPS.

\section{Choice of dependencies}
\subsection{libuv} \label{sec:dependency-libuv}
Since TAPS is an asynchronous API at its core, CTaps needed to
reflect this. This could have been achieved in several ways,
with the primary alternatives being internal threading or an event loop. For
CTaps, the event loop was found to have several advantages,
primarily avoiding race conditions and not having to guard shared
state, since a single threaded event loop ensures that state
is never accessed concurrently.

When selecting a specific event loop library libuv~\cite{Libuv2025} presented
several advantages such as built-in protocol handles for TCP
and UDP, in addition to being a mature library which is already
used in production systems such as Node.js~\cite{nodejsEventLoop}.
Libuv is also the event loop engine behind
NEAT, which influenced the decision of CTaps~\cite{neatCode}.

Internally in CTaps, libuv provides the asynchronous interface
which CTaps uses to interact with the network. It also provides
a timer, which is used to query Picoquic at given intervals
for new packets to send.

\subsection{Picoquic}
CTaps has been developed on Linux kernel version 6.1.0, which
features no kernel-level QUIC support.
Since QUIC is also not supported natively by libuv an external library
was needed to add QUIC support to CTaps.
For this the largest priority was ease of use, in order
to assist in exploring how to QUIC features map onto TAPS.

Picoquic~\cite{christianhuitemaPicoquic} was therefore chosen because
of its simplicity and was
prioritized over larger and more performance-oriented libraries such as
MsQuic~\cite{MsQuic} from Microsoft.

A large advantage of Picoquic is that it features two modes of operation.
It can either be used independently with its own event loop, or be
integrated with an existing one by querying at given timer intervals.
The first mode allowed the benchmarking to be performed with a Picoquic
client completely independent of CTaps, while the second mode allowed Picoquic
to integrate with libuv without having to manage two event loops.

\subsection{GLib} \label{sec:glib-design-reason}
CTaps uses GLib~\cite{GLibGitLab} for implementations of
hash maps, linked lists, trees and arrays.

GLib was added as a dependency since C has a small standard library
which does not feature any of the data structures mentioned.
If CTaps were to implement all the data structures internally
it would create a large surface area liable for bugs requiring
extensive testing, increasing the scope of CTaps.

GLib itself was chosen because it is a large and mature library
which is featured in several other large well known software libraries,
such as GTK~\cite{gtk_website}.

\subsection{log.c}
In order to create structured logs which can
easily be filtered by level
CTaps uses log.c~\cite{logc}.
log.c was specifically chosen because of its simplicity
and pure C implementation, making it easy to integrate
with CTaps.

\section{Architecture of CTaps} \label{sec:ctaps-arch-design}
As mentioned, CTaps aims to conform to the asynchronous TAPS API described
in RFC~9622~\cite{rfc9622}. Since CTaps is asynchronous it has
to rely on the result of any asynchronous operation being passed via
callback, which affects the look of the external API and
the internal architecture.

CTaps features a shared protocol API internally and is therefore extensible
when it comes to adding support for new protocols.
In order to keep the implementations as small as possible, common logic such as
state tracking
is extracted to the Socket Manager, which is an internal data structure
responsible for network sockets.

An overview of the architecture of CTaps can be seen in \cref{fig:ctaps-arch}.
Here, all the core aspects of CTaps can be seen, with the Socket Manager
acting as an intermediary between the CTaps API and the
protocol interface, which provides a common API to the different
protocol implementations. The unique consideration of QUIC can
also be seen, requiring Picoquic as an external library due to lack
of support in libuv.

This section will now expand on these key concepts which have shaped the
architecture of CTaps and the reasoning behind each of them.

\subsection{Mapping QUIC to TAPS} \label{sec:quic-arch-mapping}
While the TAPS RFCs were designed with QUIC in mind, they were written before the
QUIC design was finalized in RFC~9000~\cite{rfc9000}. This means that RFC~9623 does not
feature any specific transport protocol considerations for mapping QUIC into
a TAPS system~\cite{brunstromImplementingInterfacesTransport2025}.
There exists a draft for mapping Transport Services to QUIC, but this has
not made it into the official RFC~\cite{transportServicesMappingForQuic}.

The most important design decision made in CTaps in regards
to QUIC was to map a QUIC Connection
to a Connection Group instead of to a Connection. Since
Connection Groups are populated by Connection cloning, and RFC
9622 says \say{if the underlying protocol supports multi-streaming, it is natural to use this functionality to
implement \mono{Clone}}~\cite[Sec. 7.4]{rfc9622}, the Connection Group
becomes the natural model for the QUIC Connection.

CTaps could
map QUIC connections to a TAPS Connection, but this would make
multistreaming unavailable through the TAPS API, since TAPS
features no equivalent concept of a stream within a Connection.
Multistreaming is a core benefit of QUIC and allows for more
efficient bandwidth usage, as showcased in \cref{sec:smallfiletransfer}.

The consequence of this mapping is that the behavior 
when closing a single Connection depends on the underlying
protocol. For TCP, closing a CTaps Connection will initiate
teardown of the entire connection, and no more data will be
delivered after the \mono{closed()} callback has been invoked.

Since QUIC features no way of gracefully closing
a single stream other than both peers sending a FIN flag,
CTaps has to let a Connection remain open to receiving data
after an application has invoked \mono{closed()} until the
peer has indicated that it has finished writing, which breaks
with the expected behavior from TCP.

This differing behavior can be mitigated by only interacting
with Connection Groups when wanting to shut down a logical connection,
which is compatible with how TCP support is implemented.
This is discussed in detail in \cref{sec:quicUnderTaps}.

\subsection{Socket Manager} \label{sec:socket-manager-design}
The Socket Manager is an internal structure in CTaps responsible
for acting as an asynchronous intermediary between the API and the protocol implementations,
as well as managing the lifetimes of the underlying sockets.

The Socket Manager has to be compatible with both multiple unrelated
Connection Groups using a single Socket, such as when listening
on an UDP socket, as well as Connections in the same Connection
Group using different sockets, such as when cloning TCP connections.
Therefore, the Socket Manager is
referenced on the Connection level and a single shared Socket Manager is created
for each used Socket.
Since UDP Connections in CTaps can share a single socket the 
Socket Manager is also responsible for demultiplexing incoming messages,
as UDP is not connection-oriented.

Since several functions within the various protocol implementations
are asynchronous, communication between the protocol implementation
and the Socket Manager has to be asynchronous as well. The Socket
Manager features a structure with internal callbacks which are invoked by the protocol
implementation at appropriate times, for example to notify
the Socket Manager that a Connection has been closed after teardown
has completed. This callback structure can be seen in \cref{lst:socket-manager-callbacks}.

This pattern of the protocol implementation exclusively speaking
to the Socket Manager only applies to memory-critical paths, in order
to mitigate
potential memory leaks.

An example of such a memory-critical internal flow in CTaps can be seen in
\cref{fig:async-close-flow}, showing the asynchronous flow of
graceful connection teardown for a Connection using a shared
Socket Manager. Here, a single Connection in a Connection Group
is closed, and the Socket Manager invokes the application callback
asynchronously.

The figure also shows that crucially the Socket Manager keeps
the socket open, so that it can be used by the remaining Connections.

\subsection{Protocol Interface}
In order to make the CTaps design as modular as possible,
CTaps uses an internal interface to interact with the underlying protocol.
This interface abstracts away the implementation details of the individual
protocols and allows the Socket Manager to not have to implement protocol
specific logic, making it far easier to extend support to more protocols in
the future.

An excerpt of the protocol interface struct can be seen in
\cref{lst:protocol-impl-struct}. Here the \mono{supports_alpn} flag
and the Selection Properties can be seen. These are used
to encode the capabilities of the protocol when defining an
instance of the interface. The encoded capabilities 
allows CTaps to filter and sort protocols according to the
Transport Services desired by the application, while
the implemented function pointers present a generic
interface to protocol-specific behavior.

\section{API Design}
The external API of CTaps is designed according to the TAPS API
described in RFC~9622~\cite{rfc9622}.
The TAPS API is object oriented and asynchronous, which are the core
aspects of CTaps as well. In addition, CTaps hides
internal state from the user and only exposes getters and setters
to fields which are meant to be public.

These concerns, in addition to a consistent ownership model which helps
prevent memory leaks, are the largest influences on the external
CTaps API. This section will describe these design decisions,
while a full API overview, which objects are available and how they
map to TAPS can be seen in the attached Doxygen output in the
appendix. \todo{Add to appendix}

\subsection{Object Orientation}
Since TAPS itself is object oriented CTaps aims to emulate this.
This affects how CTaps presents data structures to external application
and how it structures them internally. An example can be 
seen with the Remote Endpoint object in \cref{lst:remote-endpoint}.
This object contains all the context related to remote endpoints, which then is used
in Connections and Listeners. The object has dedicated functions to operate
on it, and therefore encapsulates all responsibilities related to Remote Endpoints.

CTaps aims to present objects as consistently as possible to the application.
An example of this is that 
in CTaps every Connection belongs to a Connection Group,
even if it has not been used for cloning. This
was done in order to both present a unified interface to applications using
CTaps. This also allows CTaps to avoid internal checks for whether a Connection was
part of a Connection Group or not.

In order to hide internal state from applications, the public CTaps API only
features forward declarations of objects. This means that the application
cannot interact directly with the struct members and instead must
interact with the objects through a list of functions exposed by CTaps,
which are used to allocate a struct on the heap and to modify the object using
its pointer, a pattern which can be seen in \cref{lst:using-remote-endpoint}.

This pattern was chosen in order to make it more clear exactly which fields it is
allowed to interact with, and which fields are only used internally in CTaps. This
also allowed CTaps to differentiate between read-only Connection Properties and writable
Connection Properties as described in the TAPS API~\cite[Sec. 8.1.11]{rfc9622}.

\subsection{Asynchrony} \label{asynchrony}
Being asynchronous and built around an event loop has
several implications for the external CTaps API, with the most important being
that most CTaps functions return immediately, with the results given 
through a callback invoked later.

When registering callbacks in CTaps this is done as close
to the relevant event as possible, for example Connection-related callbacks 
are registered when the Preconnection is initiated, an example of flow
and the registration of callbacks can be seen in \cref{lst:preconnection-init-flow}.
Not all callbacks have to be registered, although not registering important
callbacks such as the \mono{ready()} or \mono{closed()} would lead 
to the application missing important events that it should react to.

Since the results of most operations in CTaps is available through callbacks,
the application needs a way to connect these results to state external to the
callback, for example the filename to transfer when the \mono{ready()} callback is invoked.
The application could achieve this through global variables, but CTaps 
allows the user to pass a void pointer when registering callbacks so that
downstream applications can avoid global variables as much as possible.

When initiating a Connection, the void pointer will be stored in the Connection
structure and it can be fetched when the Connection is for example passed
to the \mono{ready()} callback.

CTaps uses callbacks to create Connection and Listener
objects themselves. Both of these objects require asynchronous operations
to instantiate, and therefore require the resulting object to be passed
through the application through callback.
The Connection is asynchronous both
because Candidate Gathering is asynchronous, but
also because it is only created after an actual connection has
been created over the network, in the case of connection-oriented
protocols. The Listener is asynchronous because it uses the
same candidate gathering as normal Connection initiation, which
is asynchronous in CTaps.

CTaps has two ways of notifying an application of errors, either
through a return code or an error callback.
Return codes are used for synchronous error while callbacks
are used for asynchronous errors.
Error codes in CTaps are
negative values matching
an errno corresponding to the given error. This matches the
error convention used by libuv~\cite{libuv_errors}.

If a function 
can fail asynchronously then it also takes error callbacks as
parameters, which can for example be invoked if a remote endpoint
rejects an connection attempt.

In theory, error handling could be unified into using callbacks
for all errors, but CTaps splits it into two categories in order
to fail as early as possible and to differentiate simple errors
such as illegal parameters from more advanced asynchronous problems,
which often may stem from things outside the application's control.

The asynchronous operations described above are all reliant on
the event loop provided by libuv as described in
\cref{sec:dependency-libuv}. CTaps
uses a single global event loop from libuv internally which has to
be initialized before operation can be performed.

This initialization is done through the
setup function \mono{ct_initialize()}, and teardown has to
be performed through \mono{ct_close()}.

After initializing CTaps, the application must perform any setup
required and start a Listener or initiate a Connection in order to
start interacting with the network. After this has been performed the
application must start the event loop with \mono{ctaps_start_event_loop()}.
This event loop will block the invoking thread until there are 
no more open handles performing work.

CTaps is not designed with multi-threading in mind and does not make use
of the threading API available through libuv~\cite{libuv_threading}. This
means that there is no internal synchronization and it expects applications
to only interact with CTaps through a single thread, which eventually will
start executing the event loop. Multi-threading is avoided in order to
minimize complexity and it would not add much value to the nature of CTaps
as a reference implementation of TAPS.

\subsection{Ownership Model}
Because all objects in CTaps are allocated on the heap in CTaps using \mono{ct_*_new()}
functions, it is important that the client frees the allocated object at the appropriate time.
In order to make this as intuitive and consistent as possible, CTaps has a clear ownership model
where the application retains ownership of any allocated objects and is responsible for freeing them.
In addition to the objects allocated synchronously using a \mono{ct_*_new()} function, applications
also gain ownership of Connection and Listener objects which are created asynchronously.

CTaps passes ownership to the created objects, even in the cases where
the application has not registered the
correct callbacks for receiving the object.
This will lead to a memory leak if an application is registered without
a \mono{ready()} callback, since the Connection will never be freed.

When freeing object which require teardown, the application must first
invoke the associated \mono{close()} function, as the object
will only be safe to free after the associated \mono{closed()}
callback has been invoked by CTaps.

Connection data and any waiting messages can still be read from a connection after it has
closed, but the data is invalidated on free.

The only location where the application does not gain ownership over an object is in
the receive callbacks passed to CTaps when an application wants to receive a message. 
Here, the received Message object and Message Context only remain valid for the duration
of the callback, and must be deep copied if they are to be referenced outside of this scope.
CTaps decided to deviate from the standard model here because of the frequency
of receiving Messages and that they would very rarely have to be stored for long term use.

An alternative to deep copying objects would be for CTaps
to implement reference counting for all objects, but this was decided to be unnecessarily complex
when compared to deep copying.
