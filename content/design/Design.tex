\section{Choice of dependencies}

\subsection{libuv}
CTaps is reliant on several
external libraries. The
and most essential of these are libuv \cite{Libuv2025}. Libuv provides the central
event loop which allows CTaps to be asynchronous. Libuv was chosen
to drive the event loop within CTaps because it is a well-established
and mature library, which offers both an event loop and networking
utility functionality, which allows CTaps to avoid having to write
custom epoll listeners. Libuv is also the event loop engine behind
NEAT, which also influenced the decision of CTaps \cite{neatCode}.

\subsection{Picoquic}
Since QUIC is not currently supported by libuv or by the Linux kernel
an additional external library was needed in order to add QUIC support
to CTaps. Since CTaps is primarily a thesis project and is not
designed to be production ready, Picoquic \cite{christianhuitemaPicoquic} was chosen over other, larger
C QUIC implementations, such as MsQuic \cite{MsQuic} from Microsoft or XQUIC \cite{XQUIC} from Alibaba,
because of its relative simplicity. Picoquic also features the benefit
of allowing the use of a built in event loop in benchmarking code,
making the QUIC comparisons when benchmarking completely independent of CTaps.

\subsection{GLib} \label{sec:glib-design-reason}
CTaps uses GLib for internal data structures. This was chosen over
re-implementing the necessary data structures internally in order
to limit the scope of CTaps, as well as to avoid risking introducing
bugs, which are more likely in custom implementation than in a mature
library such as GLib.

\subsection{log.c}
CTaps uses log.c, in order to create structured logs
which can be easily filtered by level \cite{logc}.
This library was chosen because of its simplicity
and pure C implementation, making it easy to integrate
with CTaps.

\section{Scope}
Since one of the key reseach questions motivating the development of CTaps is exactly
how well the TAPS API as it exists currently can be presented programmatically,
it follows that CTaps aims to be a as complete an implementation of the TAPS
API as possible.

While presenting a complete functional implementation of TAPS where
all Transport Properties are respected is naturally out of scope, it was however determined
that CTaps should present as much of the TAPS API described in RFC 9622 \cite{rfc9622} as
possible, even if most of the options presented were implemented as a no-op in CTaps.

Despite the desire to present as much of the API as possible, certain sections of
the TAPS API are not present in CTaps, with the only major feature not available
through the CTaps API being peer to peer connections. This was deemed to be
too orthogonal to the CTaps internal functionality, and presenting this as
a no-op would present little value, since CTaps would not be able to predict
if any internal functionality of the rendezvouz functionality would cause changes
in the TAPS API. Other functionality which is not present is primarily some
forms of error error handling such as soft errors in addition certain 
functionality such as being able to add endpoints to an active Connection.
These are excluded because implementing
them would not drive any research question forward,
and were therefore deemed to be out of scope.

\section{Architecture}
The overarching architecture of CTaps is primarily affected by it being
an asyncrhonous library and the desire to utilize a common protocol API
in order to make additions of future protocols as simple as possible.

An overview of the architecture of CTaps can be seen in \cref{fig:ctaps-arch}.
Here, all the core aspects of CTaps can be seen, with the Socket Manager

\subsection{Protocol Interface}
In order to make the CTaps design as modular as possible,
CTaps uses an internal interface to interact with the underlying protocol.
This interface abstracts away the implementation details of the individual
protocols and allows the upper layers to interact with the network in an
asynchronous way. This interface consists of the protocol name and certain
protocol properties, a list of selection properties which is used
when filtering candidates during Candidate Gathering, as well as
a list of function pointers invoked when interacting with the protocol.
The full definition of the protocol interface struct can be seen in
\cref{lst:protocol-impl-struct}.

A single protocol is implemented as a single implementation of the interface.
Together with providing implementations of the interface protocols, it also
has to register a list of settings for each selection protocol. These settings
indicate which properties the given protocol can fulfill, which is used 
when filtering candidates during candidate gathering.

\subsection{Socket Manager} \label{sec:socket-manager-design}
The architecture of CTaps has been designed to fit the restrictions
placed on socket lifetimes by the underlying protocols of TCP, UDP
and QUIC. Because the application can close and free connections
at any point, CTaps has to ensure that all required memory remains
valid until the final external resource using it has been freed.

CTaps therefore makes use of an internal structure called a
Socket Manager which is described in detail in \cref{sec:socketManager}.

Because UDP mandates that multiple unrelated Connections must
be able to share a socket, TCP mandates that Connections in the
same Connection Group must be able to have different sockets Socket
Managers are referenced on the Connection level. The application
interacts only with the connection
which in turn interacts with the Socket Manager. The Socket Manager
owns the protocol implementation and invokes one or more of the interface
function pointers, according to the will of the application.

Since several functions within the various protocol implementations
are asynchronous, communication between the protocol implementation
and the socket manager has to be asynchronous as well. The Socket
Manager has a structure with internal callbacks which are invoked by the protocol
implementation at appropriate times, for example to notify
the Socket Manager that a Connection has been closed when teardown
has completed.

This closed-layer model only applies to memory-critical paths, in order
to mitigate
potential memory leaks. Some remaining flows, such as messages being passed
from Connection to protocol do not currently flow through a dedicated
function in the Socket Manager. 

An example of the internal flow in CTaps can be seen in
\cref{fig:async-close-flow}, showing the asynchronous flow of
graceful connection teardown for a Connection using a shared
Socket Manager. Here, a single Connection in a Connection Group
is closed.

This Connection uses a Socket Manager which is shared
with not only its Connection Group, but also with a Connection
in a completely separate Connection Group. This can for example
happen when listening for Connections over UDP, as all received
Connections will share a socket but belong to different Connection
Groups. When the Connection is closed, the Socket Manager invokes
the associated function pointer for the underlying protocol. This
asynchronously performs any necessary connection teardown over the
wire, and then notifies the Socket Manager when necessary.

Crucially,
it does not close the socket until notified by the Socket Manager,
since the socket needs to stay open until all Connections using
it have been closed. This is not the case in the example flow,
so the Socket Manager simply updates the state of the Connection
and notifies the application of the Connection closing through the
\mono{closed()} callback.

\subsection{QUIC to TAPS mapping} \label{sec:quic-arch-mapping}
While the TAPS RFCs were designed with QUIC in mind, it was written before the
QUIC design was finalized in RFC 9000 \cite{rfc9000}. This means that RFC 9623 does not
feature any specific transport protocol consideratoins for mapping QUIC into
a TAPS system \cite{brunstromImplementingInterfacesTransport2025}.
There exists a draft for mapping Transport Services to QUIC, but this has
not made it into the official RFC \cite{transportServicesMappingForQuic}.

Because of RFC 9623 not featuring a mapping of transport services
onto QUIC, CTaps has had to make certain design decisions in order
to expose as much QUIC functionality as possible through the TAPS API.

In this process making as many QUIC features available
as possible through TAPS was prioritized above an absolute
uniformity of the behavior of the API no matter what underlying
protocol was selected. This meant that it was deemed acceptable
for any downstream applications to have to make certain
design decisions based on the underlying protocol, or to have
to handle Connections in a slightly unintuitive way, if it
wanted to write CTaps code common for all underlying protocols.

The most important design decision was to map a QUIC Connection
to a TAPS Connection Group instead of to a Connection. Since
Connection Groups are populated by Connection cloning, and RFC
9622 \say{if the underlying protocol supports multi-streaming, it is natural to use this functionality to
implement \mono{Clone}} \cite[Sec. 7.4]{rfc9622}. Because
Connection cloning creates separate Connection objects, the most
intuitive way to allow an application to interact with a single
stream is by having a single Connection represent a single
Connection, when this is applicable.

The model of a single Connection only mapping to a single
stream fits with TCP, as each TCP Connection only presents
a single stream of data. The model does however break down
slightly when introducing Connection cloning. Since Connections
utilizing TCP can also be cloned in TAPS, this allows the application
to create a Connection Group containing several TCP connections.
From the point of view of TCP there is no relation between the
TCP connections in this Connection Group, and within TAPS, each
of them can be torn down individually. This differs from the
QUIC behavior, where individual QUIC streams are only torn
down gracefully when both sides indicate that they are finished
sending.

As part of this mapping of TAPS Connections to QUIC streams,
closing a Connection which uses QUIC in CTaps only sends a FIN
on the associated stream instead of tearing it down completely,
unless the Connection was already closed for receiving, and it
was the last open Connection in the Connection Group.
This is discussed in detail in \cref{sec:quicUnderTaps}.

\subsection{Internal Error Handling} \label{sec:internal-error-handling}
Since CTaps is asynchronous it has two primary forms of errors,
synchronous and asynchronous, both of which need to be propagated
throughout the system. While all error states of asynchronous functions
could in theory be handled asynchronously, CTaps has chosen to differentiate
between synchronous errors which are propagated through return value,
and asynchronous errors which are propagataed through callbacks.

CTaps expects the parameters passed to internal functions
to be valid, and therefore does not consider functions where the only
error condition to be invalid input parameter to be fallible, as long
as the external application is not responsible for the paramaters.
If a function has a genuine failure path then it will return an
value which will reflect this.

If the function would return
a pointer in its success path then it returns a NULL pointer in
the failure path. Otherwise if the function can fail then it
returns an error code. CTaps follows the convention of libuv
of using negative numbered constants matching the errno set by
Linux \cite{libuv_errors}. If an error condition is encountered
which does not not an errno then a matching negative errno is
selected manually by CTaps.

If a function returns a synchronous error then any error callback
will never be invoked.

In certain situations errors cannot be discovered until after some
asynchronous handling has completed. In these cases,
CTaps invokes a callback in order to notify of the error. Internally
in CTaps, the same callback is used for both success and error states,
with the value of the parameters indicating whether the operation
completed successfully or not, usually through passing a NULL parameter
on error where a value is otherwise expected.


\section{API Design}
\subsection{Object Orientation}
In oder to present a consistent API to downstream application,
CTaps follows a design philosophy to match the design
described in RFC 9622 as closely as possible \cite{rfc9622}
The abstract interface described in this RFC is object
oriented and to match this, CTaps emulates object
orientation as described in \cref{sec:object-orientation-in-c}.

An example of this
is the struct representing Remote Endpoints which can be seen in \cref{lst:remote-endpoint}.
This object contains all the context related to remote endpoints, which then is used
in Connections and Listeners. The object has dedicated functions to operate
on it, and therefore encapsulates all responsibilities related to Remote Endpoints.

The CTaps API consists of a single header file, and all CTaps objects are contained
as forward declarations within this file
The application then has to interact with the objects through a list of functions
which are use to allocate a struct on the heap and to modify the object using
its pointer.
This pattern can be seen in \cref{lst:using-remote-endpoint}.

Not exposing the definitions of the sturcts 
to the application allows CTaps defines which fields are supposed to be external,
by controlling getters and setters in the external API.
This also helps future-proof the API, as renaming and adding or
removing internal members of a struct does not affect downstream applications.

Because all objects in CTaps are allocated on the heap in CTaps using \mono{\*\_new()}
functions it is important that the client frees the allocated object at the appropriate time.
In order to make this as intuitive and consistent as possible CTaps has taken the deliberate
design decision to never take ownership over any of these objects when they are passed as parameters
to a CTaps function. CTaps is then responsible for freeing its internal copies when appropriate.

This also means that if the application
knows that the object it has allocated will not be used again, then it can
be freed after the last
function taking it as a parameters returns.

All CTaps objects have
dedicated destructors and it would therefore be be possible for each object
to contain an internal reference count ensuring the lifetime of the passed
objects. Deep copying parameters was however chosen over reference counting
since the objects in the external CTaps API are relatively small, so the simplicity
of deep copying was deemed more important than the efficiency of reference
counting.

QUIC also affected the architecture and ownership model of CTaps, as it
was the only supported protocol which where the per-Connection Group
state differs from the per-Connection state, it forced CTaps to introduce
a clearer separation of conecerns with a separate void pointers to be
used for internal protocol state in the Connection, Connection Group and
Socket Manager structures. It also caused CTaps to make every Connection a
part of a Connection Group, even if it has not been used for cloning. This
was done in order to both present a unified interface to applications using
CTaps, as well as avoiding internal checks for whether a Connection was
part of a Connection Group or not.

\subsection{External Error Handling}
The error handling presented to the application largely follows
the same structure as described in \cref{sec:internal-error-handling}
with two major differences. Firstly, CTaps verifies the input parameters
of functions presented in the external API, meaning that they return an
error value, usually -EINVAL, if a NULL pointer or othervise invalid value
is passed to a mandatory parameter. The second change is that in the
external API CTaps features dedicated error callbacks.

This is done to present a more intuitive API to the application and
to fail early, making it easier to detect for example when a mandatory parameter
is omitted. Having dedicated error callbacks also allows the application to not
have to write error handling code in the success callbacks.

\subsection{Library Lifetime}
Internally, CTaps uses a global event loop, which has to be initialized
before operation can be performed. The application has to do this through the
setup function \mono{ct_initialize()}. This allocates memory for the event loop,
initiates it and sets a default log level.

After initializing CTaps, the event loop can be
started using \mono{ctaps_start_event_loop()}. This event loop will run until
there are no more open handles performing work. Any set up of Connections
which should be used during the lifetime of a given event loop has to
be performed before starting the event loop. The event loop takes over
execution of the main thread, and will block until execution of the underlying
handles completes, as described above.

This means that if the application invokes \mono{ct_preconnection_initiate()},
then CTaps will attempt to connect to the specified endpoint only when the event
loop is started.

Any behavior that has to happen after an event loop is created has to be
specified in the callbacks provided by CTaps. For example, if a Connection
has to immediately send a message after successfully connecting to an endpoint,
then this behaviour has to be defined in the \mono{ready()} callback.

Because CTaps allocates memory for the global event loop this also has to
be freed in order to prevent memory leaks. This is done through the
CTaps teardown function \mono{ct_close()}. This closes the event loop
and frees it synchronously.

\subsection{Single-threaded async model} \label{asynchrony}
Because CTaps uses libuv to interact with the network, the external API
of CTaps also has to be callback-based. The application can register callbacks
with CTaps at key locations in the API.

When registering callbacks in CTaps this is done
as close
to the relevant event as possible. Connection callbacks 
are registered when the Preconnection is initiated, Listener
callbacks are registered when invoking \mono{ct_preconnection_listen()} etc.

In order to allow the user to pass data into the callback, all callback structures
contain a void pointer which can be set by the user. This void pointer
is then passed as a parameter when CTaps invokes the callback. This allows the user
implementation of the callback to dereference the void pointer
and pass any data to the callback, to make it accessible in
the body of the callback itself.

CTaps is not designed for multi-threading in mind and does not make use
of the threading API available through libuv \cite{libuv_threading}. This
means that there is no internal synchronization and it expects applications
to only interact with CTaps through a single thread, which eventually will
start executing the event loop.

\subsection{\texttt{ct\_preconnection\_t}}
In CTaps, all connections are started through a Preconnection. Preconnections
contain basic parameters such as desired Transport Properties as well as Local
and Remote Endpoints. Preconnections can either be used to initiate a Connection
or to start a Listener, in order to receive incoming Connections.

\subsection{\texttt{ct\_connection\_t}}
In TAPS, Connections are created asynchronously through initiating
a connection to a remote endpoint through a Preconnection.
If the Connection attempt succeeds then the Connection is
passed back to the application through the registered \mono{ready()}
callback.

If the Connection cannot be established
then the \mono{establishment_error()} callback
is invoked instead of the \mono{ready()} callback.

\subsection{\texttt{ct\_listener\_t}}
The Listener is the object in TAPS responsible for receiving
incoming connections and is created asynchronously through
invoking \mono{listen()} on a Preconnection. Listeners are
created asynchronously because they share the Candidate Gathering
process as normal Connection initiation.

\subsection{Cloning} \label{sec:managing-cloning}
In CTaps, cloning a Connection will create a new Connection and add it to the
source Connection's Connection Group and pass the new Connection back to the
Application.
The client operates on each individual

Invoking \mono{connection_clone} can lead to either a \mono{Ready} event or
a \mono{CloneError} event.
For protocols which support multistreaming Connection cloning can be
used to utilize multiple streams over a single Connection.

For underlying protocols which do not support multistreaming, each cloned
Connection becomes functionally independent, but still belong to the
same Connection Group.

\subsection{Data Transfer}
Due to the asynchronous nature of the library, users can request to
receive messages at any time, and CTaps can receive messages
over the network and cache them at any time.

If no message has been received by the library, then the callback
is cached and is then invoked the next time a message
is received. Each invocation only receives
a single message, and will always receive the first message in
the message queue.

When sending messages the message is not cached, instead
immediately passed to the libuv event loop or Picoquic.

\subsection{\texttt{ct\_framer\_impl\_t}}
Framer interface consists of a structure of function pointers,
which are expected to be implemented by the application.

Since the Connection takes a deep copy of passed Messages the
Message data itself is owned by the protocol stack. Therfore
Framer can modify the message in place and does not have to allocate new
memory unless it is extending the length of the message.

The Framer interface as defined in CTaps can be seen in \cref{lst:framer-struct-def},
while an example implementation from the CTaps framing tests can be seen in \cref{lst:framingTestImpl}.

  // Implementation should call the callback when encoding is complete

                        ct_message_t* message,

  // Implementation should call ct_connection_deliver_to_app() for each complete message

\begin{listing}[tbp]
  \begin{minted}{C}
static void length_prepend_encode(ct_connection_t* connection,
                                   ct_message_t* message,
                                   ct_message_context_t* context,
                                   ct_framer_done_encoding_callback callback) {
    // Prepend the message length as a single ascii character
    // First byte is the length
    message->content = (char*)realloc(message->content, message->length + 1);
    memmove(message->content + 1, message->content, message->length);
    // Get ascii representation of length (as long as it is <= 9 bytes)
    message->content[0] = '0' + (char)(message->length);
    message->length += 1;

    callback(connection, message, context);
}

static void passthrough_decode(struct ct_connection_s* connection,
                     const void* data,
                     size_t len,
                     ct_framer_done_decoding_callback callback) {
    // Just pass through - deliver as-is
    ct_message_t* msg = (ct_message_t*)malloc(sizeof(ct_message_t));
    msg->content = (char*)malloc(len);
    msg->length = len;
    memcpy(msg->content, data, len);

    callback(connection, msg, NULL);
}

static ct_framer_impl_t length_prepend_framer = {
    .encode_message = length_prepend_encode,
    .decode_data = passthrough_decode
};

\subsection{Connection Termination}
Since both aborting and closing Connections may involve over-the-wire teardown,
all Connection termination is treated asynchronously by CTaps, even when it only involves
closing a socket synchronously.
Connection data and any waiting messages can still be read from a connection after it has
closed, but the data is invalidated on free.

Since a QUIC Connection maps onto a TAPS Connection Group, the underlying
behavior when closing a Connection which uses QUIC as its underlying protocol
may be surprising as discussed in \cref{sec:quic-arch-mapping}. 

If the application wants to close
the entire logical Connection it is recommended to close the entire Connection Group.
If the application wants to avoid having unique Connection handling based on the
underlying protocol then it can rely on using only \mono{ct_connection_close_group()}
in all situations.
