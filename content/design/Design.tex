\section{Object Orientation in C}
The abstract interface described by the TAPS WG is very clearly object
oriented, C is a procedural programming language, and does not feature
the concept of classes or objects, which would most neatly map to the
TAPS interface. Instead, the object orientation and division of responsibility
featured in an object-oriented language has to be emulated through the use of
structs, and dedicated functions which operate on them. An example of this
would be the struct representing remote endpoints which can be seen in \cref{lst:remote-endpoint}.
This contains all the
context related to remote endpoints, which then is used
in Connections and Listeners.

\begin{listing}[tbp]
\begin{minted}{C}
typedef struct ct_remote_endpoint_s {
    uint16_t port;  ///< Port number set by user
    char* service;  ///< Service name (e.g., "https") or NULL
    char* hostname; ///< Hostname for DNS resolution or NULL
    struct sockaddr_storage resolved_address; ///< Resolved socket address
} ct_remote_endpoint_t;
\end{minted}
\caption{Remote endpoint as represented in CTaps}
\label{lst:remote-endpoint}
\end{listing}

In the external API CTaps only contains forward declarations for all its structures.
The application then has to interact with the structure through a list of functions
which are use to allocate a structure on the heap and to modify an existing object.
This pattern can be seen in \cref{lst:using-remote-endpoint}.

\begin{listing}[tbp]
\begin{minted}{C}
    // Step one allocate object on the heap
    ct_remote_endpoint_t* remote_endpoint = ct_remote_endpoint_new();
    // Check for allocation error
    if (!remote_endpoint) {
      return -1;
    }

    // Then modify it
    ct_remote_endpoint_with_ipv4(remote_endpoint, inet_addr("127.0.0.1"));
    ct_remote_endpoint_with_port(remote_endpoint, 1234); // example port

    // ... Then use it in a preconnection for example...

    // And finally free it when done
    ct_remote_endpoint_free(remote_endpoint);
\end{minted} 
\caption{Remote endpoint lifetime example}
\label{lst:using-remote-endpoint}
\end{listing}

By defining structs and dedicated functions to work on the structs like
above, we can to a very large degree emulate the encapsulation of responsibilites
as seen in a object oriented language. By not exposing the definitions of the structs
to the application, we can also define which fields are supposed to be settable and
gettable externally. This also helps future-proof the API, as renaming and adding or
removing internal members of structures does not affect downstream applications.
This API with heap-allocated objects is also as close as possible to the object oriented
API as described in RFC9622 \cite{trammellAbstractApplicationProgramming2025}.

Because all objects in CTaps are allocated on the heap in CTaps using \mono{\*\_new()}
functions it is important that the client frees the allocated structures at the appropriate time.
In order to make this as intuitive and consistent as possible CTaps has taken the deliberate
design decision to never take ownership over any of these objects when they are passed as parameters
to a CTaps function. CTaps will always perform a deep copy of this object and then be
responsible for freeing its own copy. This also means that if the application
knows that the allocated object will not be used again, it can be freed as the last
function taking it as a parameters returns.

\section{Library Lifetime}
Internally the library uses a global event loop, which has to be initialized
before operation can be performed. The application has to do this through the
setup function \mono{ct_initialize()}. This allocates memory for the event loop,
initiates it and sets a default log level.

After initializing CTaps, the event loop can be
started using \mono{ctaps_start_event_loop()}. This event loop will run until
there are no more open handles performing work. Any set up of Connections
which should be used during the lifetime of a given event loop has to
be performed before starting the event loop. The event loop takes over
execution of the main thread, and will block until execution of the underlying
handles completes, as described above.

Because CTaps allocates memory for the global event loop this also has to
be freed in order to prevent memory leaks. This is done through the
CTaps teardown function \mono{ct_close()}. This closes the event loop
and frees is synchronously.

\section{Asynchrony} \label{asynchrony} \todo{Can I write some numbers here, how much is I/O bound as a percentage?}
Most modern interactions with networking applications are asynchronous
and the TAPS API reflects this. CTaps is therefore callback based. Any
interactions with the network or other I/O-bound activities are loaded
performed by the central event loop, which is managed by libuv. 
Any asynchronous interaction with CTaps is handled via callbacks.

When it comes to registering callbacks, CTaps has taken
the design philosophy of registering the callback as close
to the relevant event as possible. Connection Ready-callbacks 
are registered when the Preconnection is initiated, listener
are registered when invoking \mono{ct_preconnection_listen()} etc.

In order to allow the user to pass data into the callback, all callback structures
contain a void pointer which can be set by the user. This void pointer
is then passed as a parameter when CTaps invokes the callback. This allows the user
implementation of the callback to dereference the void pointer
and pass any data to the callback, to make it accessible in
the body of the callback itself.

\section{Preestablishment Phase}
\subsection{\texttt{ct\_preconnection\_t}}
As described in TAPS, Preconnections are used as sources to both servers
and clients in taps. They are created synchronously via \mono{ct_preconnection_new()},
where the user can pass local endpoints, remote endpoints, transport properties and
security parameters which will affect all connections and listeners spawned from
the given preconnection.

\subsection{\texttt{ct\_transport\_properties\_t}}
Whenever the user creates a new Preconnection they have the option of passing
a Transport Propertie\mono{ct_transport_properties_t}s object. Internally, this object consists of Connection
Properties and Selection Properties. In a slight deviation from RFC 9622
CTaps does not feature generic setters or getters, instead all selection
and transport properties are set through dedicated function in the format
\mono{ct_transport_properties_set_*} \cite{trammellAbstractApplicationProgramming2025}.
This is a deliberate design choice, made to make it possible to move away from
the current enum-based properties in the future, and also ensure type safety on
all \mono{get()} and \mono{set()} operations.

\subsection{\texttt{ct\_security\_parameters\_t}}
When setting up a connection using QUIC, TLS is required. The application
therefore has to pass one or more ALPN values as well as a set of certificates for a connection.
An example of setting up security parameters for a client can be seen in \cref{lst:using-sec-params-client}.

or remote hostnames. The addresses will then be resolved when invoking
\mono{ct_preconnection_initiate()} to create a client connection or
\mono{ct_preconnection_listen()} to create a server.

\section{Managing Connections}

\subsection{Cloning}
Cloning a Connection in TAPS yields a Connection Group. A Connection Group
is represented as an array of connections. The client operates on each individual
Connection, with certain operations, such as aborting/closing all being able
to operate on all connections in a connection group. For CTaps,
all Connections
have an associated Connection Group, with each Connection Group
only containing a single Connection until the original Connection has been cloned.

Invoking \mono{connection_clone} can lead to either a \mono{Ready} event or
a \mono{CloneError} event. Connection Cloning is primarily used to create
multiple logical Connections, and how this maps internally depends on the
underlying protocol being used. For protocols which support multistreaming,
such as QUIC, the connection group comes to represent a single overarching
Connection, while the individual connections in the group represents single
streams. The overarching connection can either be closed by a single operation
on the connection group or by closing each Connection individually.

For underlying protocols which do not support multistreaming, each cloned
Connection becomes functionally independent, while still supporting the
operations on Connection Groups. For a TCP Connection an independent Connection
will be established from an ephemeral local port, while UDP connections will
start listening on an ephemeral local port, immediately generate a \mono{Ready}
event, being ready to send messages from this ephemeral port.

\subsection{Connection Properties}
In a full feature TAPS system the application would be able to
dynamically set Connection Properties on Connections. This would
involve for example setting the capacity profile in order to
manipulate the underlying DSCP values in the IP header
\cite{trammellAbstractApplicationProgramming2025}. CTaps
has very limited support for this, with the only Connection Property
that can be dynamically set being Connection Priority. This decides,
if applicable, the relative priority of Connections within a Connection
Group, and is only applicable if the underlying protocol is QUIC and
the application is using mulistreaming.

\section{Data Transfer}
Due to the asynchronous nature of the library, users can request to
receive messages at any time, and the library can receive messages
over the network and cache them at any time. When a user requests
to receive a message they pass a callback to the application, if a
message is available, that is, the library has received a message
over the network and has cached it, the callback is invoked immediately,
passing the first cached message as a parameter to the callback.
If no message has been received by the library, then the callback
itself is cached. The callback is then invoked the next time a message
is received over the network. This way, the user does not have to
spin while waiting for messages and is perform other work while
waiting for incoming messages. Each invocation only receives
a single message, and will always receive the first message in
the message queue.

When sending messages the message is not cached, instead
immediately passed to the libuv event loop. The message
may queue here for a while, depending on the number of tasks
in the loop.

\subsection{\texttt{ct\_message\_t}}
Messages are created using constructors following the same pattern as other
CTaps objects. Here too does the application keep ownership over the created
Message object, even after passing it to CTaps. The message must therefore
be freed, and can be freed immediately after \mono{send_message()} returns.

\subsection{\texttt{ct\_message\_properties\_t}}
Message properties can be passed when sending a message to influence the
underlying behaviour of the mssage. Several API functions are available
through CTaps, but currently only \mono{final} and \mono{safely_replayable}
are supported, where final is used to close sending on a connection and
safely replayable is required for 0-rtt.

\subsection{\texttt{ct\_message\_context\_t}}
In TAPS Message Context is additional information which is passed alongside
the received message when receiving data. This can be used to fetch the
Local and Remote Endpoint associated with the message as well as read the Message Properites
of the incoming message.

\subsection{\texttt{ct\_framer\_impl\_t}}
In CTaps, the Framer interface consists of a structure of function pointers,
which are implemented by the application. These are then passed to a preconnection
and are invoked at appropriate times during the lifetime of the Connection.
Along with the Connection, Message and MessageContext, the framer is also
passed a callback, which has to be invoked when the Framer is done modifying
the message. Since the Connection takes a deep copy of the message, the
Message data itself is owned by the protocol stack. This means that the
framer can modify the message in place and does not have to allocate new
memory, unless it is extending the length of the message.

An example
implementation from the CTaps framing tests can be seen in \cref{fig:framingTestImpl}.

                                   ct_message_t* message,
                                   ct_message_context_t* context,
                                   ct_framer_done_encoding_callback callback) {
    // Prepend the message length as a single ascii character
    // First byte is the length
    message->content = (char*)realloc(message->content, message->length + 1);
    memmove(message->content + 1, message->content, message->length);
    // Get ascii representation of length (as long as it is <= 9 bytes)
    message->content[0] = '0' + (char)(message->length);
    message->length += 1;

\section{Connection Termination}
Since both aborting and closing Connections may involve over-the-wire teardown,
all Connection termination is treated asynchronously by CTaps, even when it only involves
closing a socket synchrnously as with UDP. Since the application has ownership over
the Connections it has received, it must be careful to free them when appropriate.
Connections cannot be received until the underlying state has been successfully closed,
which is indicated through the application through the \mono{closed()} callback.
It is recommended that applications register this callback at all times, freeing
the Connection passed to it to avoid memory leaks. It would be possible for CTaps
to automatically free the closed Connections when the \mono{closed()} callback
has not ben registered as a quality of life improvement, but this has not been
implemented at this time.

Connection data and any waiting messages can still be read from a connection after it has
closed, but the data is invalidated on free.


\subsection{Closing QUIC connections}
Since a QUIC Connection maps onto a TAPS Connection Group, the underlying
behavior when closing a Connection which uses QUIC as its underlying protocol
may be surprising to some users. QUIC does not have functionality to gracefully
close a stream both ways without waiting for the peer to send its last data.
This means that when calling \mono{ct_connection_close()} on a CTaps Connection
which uses QUIC, it will simply send a \mono{FIN} flag and wait for the peer
to finish sending before closing the connection. If the application wants to close
the entire logical Connection it is recommended to close the entire Connection Group.
If the application wants to avoid having unique Connection handling based on the
underlying protocol then it can rely on using only \mono{ct_connection_close_group()}
in all situations. This differing behavior and the reasoning behind it is discussed
in more detail in \cref{sec:quicUnderTaps}.
