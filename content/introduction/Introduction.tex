\section{Motivation}\label{sec:motivation}
Network Ossification is the process of a network becoming
resistant to change.
This has become an issue at the transport layer, where
network middleboxes present a challenge to any new IP
packets containing a protocol number which does not match
either either the Transmission Control Protocol (TCP)
or the User Datagram Protocol (UDP). In addition to limiting innovation in
the form of new protocols, network ossification also limits
the possible innovations for existing protocols such as TCP.
This can be caused by implementations havin bugs which cause
them to not support rarely used features, or by network
middleboxes not supporting changes to the wire image of
a protocol~\cite{rfc9170}.

Mitigations against this network ossification does exist,
with two examples being using UDP as a substrate 
for new transport layer protocol and encrypting packet headers
to minimize the amount of protocol data which middleboxes
can interact with on.

In addition issues with middleboxes and bugs in protocol
implementations, another issue with utilizing new innovations on the
transport layer is the network effect required in order to
start utilizing new protocols. Servers will not spend
time adding support for a new transport layer protocol
if no applications are using it, and no applications
will utilize a new protcol if no servers support it.
If a fallback to TCP is always needed, then all applications
can avoid a great deal of complexity by simply utilizing
TCP.

Together, network ossification and the network
effect required for new transport layer protocols combine
to make it hard for applications to make use of genuine innovations.

Transport Services (TAPS) is a series of IETF RFCs which aim
to tackle this issue of network ossification. TAPS presents
a common API which abstracts away protocol specific features, instead
allowing the application to specify which services it needs from the
underlying application such as reliable data transfer or in-order delivery.
By creating a common interface which is based on the application's
intent instead of the concrete action of the transport protocol,
Transport Services allows the application to be served by any
protocol which fulfills the requirements stated by the application.

For a TAPS system the transport protocol fallback is built into the libary
itself and presents no extra complexity to the application. Because
TAPS is based on intentions as well, it allows the applications to state
their ideal intentions when interactin with the network, but still have
it be compatible no matter which underlying protocol is used. This means
that an application utilizing multistreaming and connection migration
when QUIC is available does not have to change or update its interaction
with TAPS in cases where QUIC is not available and the underlying protocol
falls back to TCP.

\section{Problem Statement}\label{sec:prob-statement}
While open source implementations of TAPS exist, such as NEAT~\cite{neatCode}
and PyTAPS~\cite{frankePythonasynciotaps2025}, none support QUIC, and none have
had active development since the TAPS RFCs were published in
2025~\cite{rfc9621, rfc9622, rfc9623}.
RFC~9623~\cite{rfc9623} also provides no guidance on mapping QUIC to the TAPS API,
though an open draft exists~\cite{transportServicesMappingForQuic}. Together, these knowledge
gaps form the basis of the research questions addressed in this thesis.

\section{Contributions}
The contributions made by this thesis to the TAPS ecosystem is CTaps.
CTaps is an open source C implementation of transport services with support
for QUIC, TCP and UDP. CTaps is an exploratory work primarily
focused on how QUIC maps through the TAPS api and which
architectural concerns are encountered when implementing TAPS.

CTaps has support for several key QUIC features such as multistreaming 
and connection migration and candidate racing.
Applications utilizing these features through CTaps can be written independently
of the underlying protocol, allowing a single client to be
compatible with multiple protoocls and still make 
use of new transport layer features when available.

In addition, CTaps also supports key TAPS
features such as candidate racing. This allows applications to attempt the most
desirable protocol first, but it gracefully fall back
if a connection cannot be established. Applications
can identify features and selection properties it wants to make
use of and make connection attempts based on that, instead of
having to manually maintain a fallback hierarchy for the connections
it wants to try.

The development of CTaps has led to several insights
on exactly how QUIC features map to TAPS and what
considerations should be taken when implementing a TAPS
system.

\subsection{Scope}
TAPS is a large standard and the scope of CTaps is
naturally limited. While most of the TAPS API is present
in CTaps, it lacks support for rendezvous, adding endpoints
to an active Connection and several Transport Properties are
implemented as no-ops.

CTaps has also focused on making as many features as possible
available through its API. This means that the potential error
paths for these features have not been as extensively tested and
may feature bugs which would arise in a real-world environment.

\section{Thesis Structure}
\textbf{\Cref{chap:background}} lays out the necessary background knowledge
to understand the basis of TAPS and CTaps. It explains the current
state of transport protocols as well as TAPS and its advantages.
Finally it explains the necessary background for understanding CTaps, with
the central concepts and dependencies used in CTaps.

\textbf{\Cref{chap:related-work}} lists other TAPS implementations. It
explains their features, backgrounds and current state of development.
Finally it compares them to the current state of CTaps and reflects on
why CTaps was still needed with the current state of the ecosystem.

\textbf{\Cref{chap:design}} explains central decisions made when designing 
CTaps. It explains exactly why CTaps has the public interface that it does
and explains choice of dependencies. Additionally it discusses how well QUIC
features map through the TAPS API, weaknesses in the design of CTaps as well
as finally discussing security in a TAPS system.


\textbf{\Cref{chap:impl}} explains how the features of CTaps
actually were implemented internally in CTaps.

\textbf{\Cref{chap:eval}} contains a series of benchmarks performed to highlight
the various parts of CTaps. It compares CTaps with TCP and Picoquic clients, discussing
the advantages in terms of features and simplicity when utilizing CTaps over
direct protocol implementations.

\textbf{\Cref{chap:conc}} summarizes the conclusions of this thesis
and answers the posed research questions. It proposes future work
for CTaps and reflects on the broader significance of CTaps as a whole.
