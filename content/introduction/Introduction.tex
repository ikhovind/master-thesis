\section{Motivation}\label{sec:motivation}
Network ossification is the process of a network growing
resistant to change.
This has become an issue at the transport layer, where
network middleboxes present a challenge to any IP
packets containing a protocol number which does not match
either Transmission Control Protocol (TCP)
or User Datagram Protocol (UDP). In addition to limiting innovation in
the form of new protocols, network ossification also limits
the possible innovations for existing protocols such as TCP.
This can be caused by implementations featuring bugs in rarely used features, or by network
middleboxes not supporting changes to the wire image~\cite{rfc8546} of a protocol.

Another issue with utilizing new innovations on the
transport layer is the coordination required in order to
start utilizing new protocols. Developer of server applications
will not spend time adding support for a new transport layer protocol
if no applications are using it, and no applications
will utilize a new protocol if no servers support it.
If a fallback to TCP is always needed, then all applications
can avoid a great deal of complexity by only utilizing TCP.

Together, network ossification and the network
effect required for new transport layer protocols combine
to make it hard for applications to make use of genuine innovations.

Transport Services (TAPS) is a series of IETF RFCs which aim
to tackle this issue of network ossification. TAPS presents
a common API which abstracts away protocol specific features, instead
allowing the application to specify which services it needs from the
underlying protocol, such as reliable data transfer or in-order delivery.
By creating a common interface which is based on the application's
intent instead of the concrete action of a transport protocol,
Transport Services allows the application to be served by any
protocol which fulfills the requirements stated by the application.

For a TAPS System the transport protocol fallback is built into the library
itself and presents no extra complexity to the application. Because
TAPS is based on intentions as well, it allows the applications to state
their ideal intentions when interacting with the network, but still have
it be compatible no matter which underlying protocol is used. This means
that an application utilizing multistreaming and connection migration
when QUIC is available does not have to change or update its interaction
with TAPS in cases where QUIC is not available, and the underlying protocol
falls back to TCP.

\section{Problem Statement}\label{sec:prob-statement}
The TAPS working group concluded in 2025 after RFC~9621--9623~\mbox{\cite{rfc9621, rfc9622, rfc9623}}
were published. While several open-source TAPS implementations have been developed previously,
none have had actively maintained and none support QUIC. In addition, RFC~9623~\cite{rfc9623} does not provide any
guidance on how to map QUIC to the TAPS API.

This means that the only known implementation of TAPS, which supports both QUIC and
is actively maintained, is Apple's closed-source Network.framework~\cite{networkFramework}.
This leaves two open questions: how to implement the current TAPS standard in practice,
and how QUIC features are expressed under the TAPS API.
These form the basis for the research questions addressed in this thesis.

\section{Contributions}
The contributions of this thesis are CTaps, the insight that implementing
it has brought and a survey on how QUIC features can be mapped
to the TAPS API.
CTaps is an open-source C implementation of Transport Services with support
for QUIC, TCP and UDP. It is an exploratory work primarily
focused on how QUIC maps to the TAPS API and which
architectural concerns are encountered when implementing TAPS.

CTaps supports key QUIC features such as multistreaming, connection migration and session resumption.
Applications can utilize these features while remaining compatible with other
transport layer protocols. CTaps also supports key TAPS features, such as candidate
gathering and candidate racing. This allows CTaps to race protocols and endpoints when
establishing connections.

The development of CTaps has led to several insights
on exactly how QUIC features map to TAPS and what
considerations should be taken when implementing a TAPS
system.

\subsection{Scope}
All major elements of the TAPS API are available in CTaps,
except for rendezvous and adding endpoints
to an active Connection. All Transport Properties are
available to the application, but several are implemented as no-ops.

Because CTaps is a research artifact, development has prioritijzed feature coverage over robustness
and error handling has not been as thoroughly tested as the core functionality.

\section{Thesis Structure}
\textbf{\Cref{chap:background}} lays out the necessary background knowledge
to understand the basis of TAPS and CTaps. It explains the current
state of transport protocols as well as TAPS and its advantages.
Finally it explains the necessary background for understanding CTaps, with
the central concepts and dependencies used in CTaps.

\textbf{\Cref{chap:related-work}} lists other TAPS implementations. It
explains their features, backgrounds and current state of development.
Finally it compares them to the current state of CTaps and reflects on
why CTaps was still needed with the current state of the ecosystem.

\textbf{\Cref{chap:design}} discusses the design of TAPS Systems
in general and CTaps in specific. Additionally, it discusses how well QUIC
features map through the TAPS API, weaknesses in the design of CTaps as well
as finally discussing security in a TAPS System.

\textbf{\Cref{chap:impl}} explains how the features
were implemented internally.

\textbf{\Cref{chap:eval}} contains a series of experiments performed to highlight
the various parts of CTaps. It compares CTaps with TCP and Picoquic clients, discussing
the advantages in terms of features and simplicity when utilizing CTaps over
direct protocol implementations.

\textbf{\Cref{chap:conc}} summarizes the conclusions of this thesis
and answers the posed research questions. It proposes future work
for CTaps and reflects on the broader significance of CTaps as a whole.
