\section{Candidate Gathering} \label{sec:candidateGathering}
CTaps follows the recommended procedure for Candidate Gathering from RFC 9623
very closely. CTaps represents the candidates in a tree, using a GNode struct from
GLib to represent a N-ary tree. CTaps first builds the entire tree, resolving
all possible local interfaces, all protocols and all possible resolutions of
remote hostnames respectively. If the client has specified multiple remote endpoints,
then they are treated the same way as each invidvidual resolution from a hostname
endpoint. After the entire tree has been built, CTaps prunes the branches which
are not compatible with the required or prohibited selection properties. After pruning,
the tree is sorted according to the stated selection properties, as described in RFC 9623 
\cite{brunstromImplementingInterfacesTransport2025}.

\section{Candidate Racing} \label{sec:candidateRacing}
After a list of candidates has been gathered when initiating a preconnection, CTaps
will race them according to the specification from RFC 9623 \cite{brunstromImplementingInterfacesTransport2025}.
The goal of candidate racing is to successfully negotiate a protocol stack, local
and remote endpoints based on the user's specification. Candidate racing in a TAPS
system follows the same procedure as specified in Happy Eyeballs V2 \cite{schinaziHappyEyeballsVersion2017},
which was originally designed to increase IPv6 adoption, see \cref{happyEyeballs}.
TAPS takes racing a step further by not only racing IP-address families, but also
racing potential protocols and endpoints. A TAPS system primarily follows a staggered
approach, first sorting the gathered candidates by preference and performing a connection
attempt for each, with a delay between each candidate. All outstanding racing attempts
are cancelled upon the first success.

\section{Protocol Implementations}

\subsection{Protocol Interface}
In order to make the CTaps design as modular as possible and ease the
process of adding more protocols, CTaps uses an internal interface for a protocol,
which abstracts away the implementation details. This interface consists of
a struct containing the name of the protocols, and function pointers for
the necessary functions for the wider CTaps library to interact with the protocol.
Each protocol implementation is this way kept as small as possible, and is
isolated from the rest of the library.

Implementations of the interface also register
a list of the applicable selection properties, allowing the library
to later prune and sort the Candidate Tree protocols, when selecting
candidates.

  int (*listen)(struct SocketManager* socket_manager);
  int (*stop_listen)(struct SocketManager*);

Adding the protocols is done at runtime in order to allow any
application using CTaps to add support for custom protocols if
desired. Because all Connections using the same protocol are associated
with the same protocol structure, the protocol structure itself
is naturally stateless, providing only read-only values and function
pointers where the necessary state has to be passed in as paramaters.
How a client can add custom protocols is discussed further in \cref{sec:addingNewProtocols}.


Each Connection is associated with a single protocol, which
is selected through the process of candidate gathering and candidate
racing as described in \cref{sec:candidateGathering} and \cref{sec:candidateRacing}.
This means, that whenever a Connection wants to interact with the network, it
will invoke a function pointer from its associated protocol:

  return connection->protocol.send(connection, message, NULL);
}

\subsection{Supported protocols}
Currently, CTaps natively supports three transportation layer protocols:
UDP, TCP and QUIC. TCP and UDP were added because of their ubiquity and
relative ease of implementation due to native support in libuv. QUIC was
added since having easier access to QUIC was one of the primary motivations
behind the thesis as a whole. Using CTaps, an application can get access to QUIC,
without having to implement custom fallback-logic or error handling. This
greatly eases the implementation complexity and could help QUIC adoption
on both the server and client side.

\subsubsection{UDP} \label{sec:udpImpl}
UDP is a connectionless, message-based unreliable protocol first designed
in 1980 \cite{postelUserDatagramProtocol1980}. It has since become one
of the two primary transport layer protocols used today and has native
support in libuv through the libuv \texttt{udp\_udp\_t} handle \cite{libuvLibuvUDPHandle}.
Due to its simplicity, there is fairly little logic required to integrate
UDP into CTaps, which can be seen by it being the protocol requiring the
fewest lines of code to implement, as seen in \cref{fig:protocolLoc}.

UDP does however not fit perfectly into the model assumed by
TAPS, which is that of a Connection-based system. There is therefore
several pieces of logic in the CTaps UDP implementation which adapt
the underlying UDP layer to this assumption. Opening a client is
fairly simple, as this simply creates a new UDP handle through
libuv, binds it to either the local endpoint required or an ephemeral one
before finally starting to receive data over that handle, and
invoking the \texttt{Ready}-callback for the associated Connection.
This does not actually transfer any data to the remote endpoint
before the clients sends its first message, and any data received
on the local handle has to be filtered based on remote endpoint,
before being passed to the associated Connection. \todo{Write
about how this filtering happens after impelmenting it}
A \texttt{Ready}-callback does therefore not necessarily
mean that the remote endpoint is actually reachable.

Another piece of logic required to adapt UDP to the structure
of a TAPS library is that surrounding the Listener. A Listener
listens on a single \todo{correct?} port, creating a new 
Connection each time a new client connects. With UDP not
featuring a handshake, the Listener instead has to keep track
of which remote endpoints it has received messages from, creating
a new Connection whenever it receives a message from a remote endpoint
it has not seen before. This is similar to how TCP connections
are identified by a four-tuple of Source IP address, Source port,
Destination IP address and Destination port. Instead of the
demultiplexing to per-Connection happening as part of the
operating system stack as with TCP, it has to happen as
part of the CTaps custom UDP logic. After a message has
been received by a Listener, the Connection associated with
the sender is found, and any callback for received messages
for this connection is invoked, if registered by the client.
When sending data from a Connection created by a Listener, the
Connection itself contains the destination remote endpoint,
meaning that the Listener does not have to look up any state
in this case. \todo{Listeners should receive on seeveral protocols according to RFC9623}

For a TAPS system, Listeners are only responsible for receiving new
Connections, while the created Connections are independent
of the listener. This means that the created Connections should
remain active, even if the Listener is stopped. In order to ensure
the validity of the underlying UDP handles and other data structures
used, even if the Listener is stopped and freed by the client, CTaps
uses an underlying structure called a Socket Manager to actually
handle contain the libuv handle and to handle the demultiplexing of
messages. The Socket Manager contains everything necessary for keeping
track of the received Connections and for transferring data, while the
Listener contains all state necessary for spawning new Connections.
When a Listener is closed and freed, the Socket Manager remains active,
still receiving messages and passing them along to the associated Connection.
If a message from a new remote endpoint is received after closing a
Listener, the associated Socket Manager will simply discard it. The
Socket Manager needs to remain active as long as there is at least
one active associated Connection. It therefore has an internal counter,
keeping track of the associated Listener and the number of active Connections.
When a Connection is closed, it checks whether or not it was created from a Listener
and if it was, then it informs its associated Socket Manager,
which then internally checks whether there are any remaining associated
objects. If there are none, the socket manager closes the associated socket
through the associated protocol, and then finally frees itself.

\subsubsection{TCP}
TCP is a connection-oriented, stream-based reliable protocol first designed
in the 1970s \cite{cerfSpecificationInternetTransmission1974}. TCP forms the
basis of much of the modern internet and is the most commonly used transport
layer protocol. TCP is therefore naturally a requirement for any networking
library and is supported in CTaps. TCP has native support in libuv through the
\mono{uv_tcp_t} handle \cite{libuvLibuvTCPHandle}.

The TAPS API aims to reflect the most common ways of interacting with the network,
providing the defaults which have the broadest applicability and are least
dangerous in terms of what a normal client expects from the network. Since
most modern applications which use the network are built on top of TCP,
this naturally refelcts in how most moern applications actually use the
network, for example expecting a reliable transfer of data. This is however
not the case with every aspect of TCP, as most application level protocols
are not stream-based, but build their own custom framing logic on top
of TCP.

Apart from the stream-based logic, TCP maps relatively well to the concepts
provided by TAPS. When creating a Connection, a socket is created and
a handshake is initiated. If this handshake succeeds, then the
\mono{ready} callback associated with the Connection is invoked and
the Connection is ready to send or receive data. If the handshake fails
then the Connection could not be created and the client can be alerted
of this immediately, without trying to send data itself.
\todo{Candidate racing might mean that this kinda doesn't matter}

When listening, the demultiplexing required in UDP, as discussed in
\cref{sec:udpImpl}, is handled by the kernel TCP stack itself.
Since TCP is also connection-based, there are also expectations
that you have to listen for and spawn new connections from a given
endpoint on a server, instead of purely receiving data as with UDP.
This means that there is already built-in logic to handle the creation
of new connections in this case, which can be utilized in order to spawn
new Connection objects for the sake of a TAPS system. For example, libuv
has the built-in function \mono{uv_listen}, which invokes a callback whenever
a new connection is received. Since the whole protocol is connection-oriented,
it also means that the logic of only closing the receiving socket when
the last connection closes is built-in, and does not have to be implemented
by CTaps. CTaps still uses a Socket Manager to keep track of the open TCP
handle, however, for TCP this only tracks the single listener, and can be
closed immediately when the listener is closed. The new TCP handles which
have been spawned from this handle will not be closed, as this logic is handled
by libuv.

\subsubsection{QUIC} \label{sec:quicImpl}
Out of the protocols natively supported by CTaps, the logic related to QUIC 
is by far the most complex, which can for example be seen by the required
number of lines of code in \cref{fig:protocolLoc}. QUIC is a relatively
new protocol, with RFC 9000 being published in 2021 \cite{iyengarQUICUDPBasedMultiplexed2021}. As QUIC and TAPS was developed relatively concurrently, the TAPS
RFCs feature no protocol considerations or implementations hints like is featured
for several other protocols, despite access to QUIC being one of the
major value propositions of a TAPS system \cite{brunstromImplementingInterfacesTransport2025}.

With the recent publication of the QUIC RFCs, this also means that
none of the current implementations of TAPS found during research for this
thesis, such as NEAT or PyTaps, feature QUIC as one of the underlying protocols
\cite{khademiNEATPlatformProtocolIndependent2017}\cite{frankePythonasynciotaps2025}.
As mentioned previously, this is one of the key motivations behind devloping CTaps,
and is the biggest value proposition behind CTaps.

QUIC is a large protocol and is very configurable. In the development of CTaps,
effort has been made to make this configurability as available as possible to
a client using CTaps, however, since CTaps aims to be a implementation of the
TAPS API, the decision has been made to not expose to many QUIC-specific configuration 
options. \todo{List some of the options not exposed}
For these configurations options, CTaps has tried to follow the principle of
least surprise, choosing the sanest defaults as possible. Other configuration
options are available through the TAPS Transport Properties, Security Parameters,
other configuration options, or can be inferred by a combination of properties.
For example, Application-Layer Protocol Negitiation (ALPN), which is mandatory for QUIC,
is available through a dedicated ALPN Security Parameter. Some custom configuration
was found to be necessary, for example the key and certificate files, which are required
for creating a QUIC server and are set globally for each QUIC context have to be
set through a global state, and are passed to when first initializing the CTaps
context \todo{Rephrase after refactoring from global to per-instance}.

Since libuv does not feature native QUIC support like it does with TCP and UDP,
CTaps uses Picoquic as an external library, working as a "QUIC engine",
while performing the actual transfer of data over an UDP socket provided by libuv.
More details on why exactly picoquic was chosen over other QUIC implementations
can be seen in \cref{sec:picoquic}.

Picoquic features two primary types of state used within CTaps. The global picoquic
context is contained within a \mono{picoquic_quic_t} structure, and the per-connection
context is contained within a \mono{picoquic_cnx_t} context \cite{huitemachristianPicoquicInternalStructures2020}.

Picoquic has two modes of operation, either it can provide the entire networking
stack, providing its own event loop and socket handling by invoking the \mono{picoquic_packet_loop},
or it can serve as a "QUIC engine", with the client being responsible with reading
data from and sending data to the networking, with picoquic formatting the QUIC packets
to send over UDP. Simple clients of picoquic which only need to use QUIC can use the
built-in event loop, while more complex use cases such as CTaps can use the second
use case, supplying their own event loop. Data which is read over the socket associated
with QUIC is passed to picoquic through the \mono{picoquic_incoming_packet_ex} function
and new QUIC packets to send over the network can be fetched through \mono{picoquic_prepare_next_packet}.
When functioning as a "QUIC engine" \todo{Figure out better term for quic engine lol}
all the data received over the UDP socket is passed directly to picoquic, and picoquic
is polled at regular intervals for new messages which are ready to send.
The maximum time until the next mandatory polling
can be fetched using the \mono{picoquic_get_next_wake_delay} function. This function needs
to be reset every time picoquic receives new data, as there may be a need to respond
within a set time interval. This function returns wakeup times common to a whole QUIC Context,
so a single
timer will work across several QUIC connections. Internally in CTaps, the timer
functionality is handled by a single instance of a libuv \mono{uv_timer_t} handle \cite{libuvLibuvTimerHandle}.

Initiating a Connection over QUIC is fairly straight-forward, as QUIC is connection-oriented
by default \todo{But it can be connection-less??}. First, a single UDP socket is created
for the desired local endpoint or an ephemeral one, as with a normal UDP socket, see \cref{sec:udpImpl},
then, \mono{picoquic_create_cnx} is invoked. This makes picoquic start to create a connection
and generate the necessary packets for performing a handshake and when the timer fires,
the generated data is transferred. Picoquic informs the client about the state of the
connection through a callback supplied when first building the QUIC context. In this
callback, CTaps can see when a given Connection is ready, and invoke its own client's
callbacks. Data can be transferred as by the CTaps client as normal, and CTaps will
add it to a given QUIC stream using \mono{picoquic_add_to_stream}, causing picoquic
to build a new message with the given data.

Opening a Listener is even simpler. All CTaps has to do is open a UDP socket listening
on the desired local endpoint or an ephemeral one, and when data is received is is passed
to picoquic through \mono{picoquic_incoming_packet_ex}. If picoquic has not seen the
given endpoint before, it will create a new picoquic connection and pass this to CTaps.
CTaps can then use this picoquic connection to create an inernal Connection, and invoke
the \mono{ConnectionReady} callback from the Listener. CTaps keeps track of the created
Connections according to their associated remote endpoints \todo{implement this, or is it in socket manager?}.
This is
needed because as with UDP, when closing the listener CTaps cannot automatically close the
receiving UDP handle, since the created Connections remain active and they all receive and
transfer data over the same UDP socket. When closing a Connection, picoquic is informed
through \mono{picoquic_close} and it will wind down the connection. When all the created
connections have been closed, the UDP socket will be closed as well.

As mentioned previously ALPN is mandatory for QUIC and is supplied to picoquic
by CTaps through a configurable Security Parameter. ALPN is a negotiation
between a QUIC server and client, therefore, both a Listener using QUIC and
a client using QUIC has to take this parameter into account. For a client this
is fairly straight forward, CTaps creates a picoquic connection and when
necessary picoquic requests a list of protocols for ALPN through a callback
to CTaps. However, for a Listener the ALPN implementation in picoquic does
not perfectly line up with the expected behaviour of CTaps. Since each Listener
represents a server, each Listener has a unique set of Security Parameters and
can therefore potentially supports varying protocols for ALPN. However, for
picoquic the ALPN negotiation of an incoming connection to a server while
picoquic is in the process of building the connection. Picoquic lets the
client supply a ALPN selection callback through the function
\mono{picoquic_set_alpn_select_fn}. This callback is only supplied the
global picoquic context and picoquic offers no functionality for fetching
the connection which is currently being created or to which local endpoint
the conneciton currently undergoing ALPN is trying to connect to. There is
therefore no way of associating an incoming connection with a Listener
and therefore way of knowing exactly which protocols are supported.
ALPN therefore becomes more complicated and CTaps has to make a sacrifice
in order to be able to perform it successfully. During development three
primary alternatives were considered, creating a global context per-Listener
instead of a single global one, accept any ALPNs supported by at least one
Listener and redirect if it does not match the desired endpoint, or finally
to dissalow different ALPNs between listener associated with a single
context. \todo{Is this actually true, maybe the second option is the expected handling}

The first option was rejected as it does not match the expected usage
of picoquic, and degrades performance while introducing unecessary complexity.
While the second option of redirecting would be the most client-friendly
in terms that it would lead to the largest number of successfull connections
to a server created by CTaps, the third option was chosen for several reasons.
First of all, third option follows the principles of failing early and offering
the least surprise to a server implemented in CTaps and to the client trying to
connect. If a client has requested a specific endpoint, it would be both surprising
and difficult to debug exactly why it was redirected if it knows that a server
is running on a given endpoint. It would also be possible that the client
supports more ALPN protocols than the server it is trying to connect to. If
one of the protocols supported by another Listener is selected on connection,
the client could be redirected even if it would accept an ALPN protocol which
is supported by the Listener on the endpoint it is trying to connect to.
By failing during setup of the server, this limitation of picoquic becomes
immediately apparent to a developer setting up a Listener in CTaps, and
necessary workarounds can be made of this is truly a problem.

\subsection{Adding new protocols} \label{sec:addingNewProtocols}

implementation of the struct, filling out the function pointers
with custom behaviour for the given protocol. Libuv features
designated handles for TCP and UDP support, while generic
handles can be used for other protocols. 
With the abstraction of the protocol interface, adding new protocols
becomes fairly simple, depending on the complexity of the protocol
itself.

New protocols can naturally be added to the core CTaps library,
making the underlying protocol available to all consumers of CTaps

\section{Testing} \label{testing}
A core tenet since the very start of development was that the library should be
as robust as possible. This would to a very large degree achieved primarily through
the use of automated testing, and continious integration. By writing automated tests
which run on every commit to the repository, it would be easy to see exactly which change
has lead to a bug, making it far easier to fix. A well-tested library is also an absolute
necessity when trying to attract external dependents, or to integrate it into a larger software
library. This is especially important for large libraries such as TAPS, which abstract away
large amounts of logic away from the application programmer, making it harder to detect bugs
and know exactly where the problem in their application lies.

Writing tests for an asynchronous networking library is not necessarily straight-forward.
Both networking and asynchronous programming is not necessarily deterministic, which makes
it harder to write reliable tests which can verify the library's expected behaviour. Another
challenge is that there are not very many testing libraries available in pure C. Early on,
it was therefore decided to write the tests in C++, which allows for interoperability with the C
library, but would also allow for using a C++ testing library such as Google test. \cite{GoogleTest2025}
This allows for a richer feature set than a simpler library or manually written tests in C,
and allows for a richer set of assertions, verifying for example that the library fails in the expected
manner when given invalid data.

Tests are first separated into type and then into responsibility. When separating
by type, the main types are integration tests and unit tests. Integration tests perform a larger check on the
whole library and often actually interact with the network by sending packets or creating
connections to a local server. \todo{Talk about how this is done, maybe via docker?}
Unit tests perform a smaller check on individual functions, attempting to verify
all logic and that differing branches are hit when expected, based on the input data.
When dividing by responsibility, the testing stack largerly follows the same model
as the main library, with test files separated by responsibility of the original
file or struct that it is performing tests on.

A central concept when writing good tests is to isolate the component being tested from
external libraries. This is important for several reasons. First of all, we are not interested
in verifying the behavior of our external dependencies, as we have to assume that they
work well in order to limit the scope of our testing. Secondly, we need to know that if
an external library \textit{does} fail, we need to know that our library will handle it
in the correct manner. Therefore we have to simulate that an external library fails and
verify our applications behaviour in this case. All of these concerns are best met by
mocking. Mocking is the concept of defining a custom behaviour for an external library
or dependency when testing, instead of invoking the real library. By using this we can
simulate the full range of behaviours from the external library and verify that our application
reacts correctly, while also avoiding time-consuming API calls, making unit tests
significantly faster. For mocking, CTaps decided on FFF, which is a small open-source
framework for creating fake C functions when testing. \cite{longFakeFunctionFramework2025}
FFF can either mock the external library functions, such as \texttt{uv\_getaddrinfo},
which is the built in support for DNS lookup in libuv, or mock a call to a different
function, which either contains a call to an external library, or which we are simply
not interested in testing.

\section{Build System}
CTaps is built using CMake. CMake is used for both building the library
and all tests.
CTaps mimics other open source libraries such as libuv as much as possible,
in order to make the structure as familiar as possible. This includes
having the API consist of a single header file, \mono{ctaps.h} which
is included to expose all functionality intended to be used by an Application.
CTaps is also compiled with visibility of all symbols set to hidden by default,
with only the symbols marked as \mono{CT_EXTERN} being available in the final
shared library file. This is done to hide internal state from external Applications,
making it less complex to make use of CTaps for an external Application.

CTaps is built as a shared library. This means that the library is compiled
into a single shared object file. When compiling an Application using CTaps,
this shared object file is not included in the compiled binary, instead the
compiled binary looks for a shared object file at a given location at
run time. This means that any Application depending on CTaps will by default
have to include CTaps in its own binary, leading to a smaller file size. This
is naturally dependent on the build configuration of the Application. 
The largest downside of a a dynamically
linked Application is that it requires its users to have CTaps installed on their
systems, in order for the Application find the correct symbols at run time.

In addition to the smaller binary size, the largest upside of a dynamically
linked application is that it is far easier to update its dependencies.
If a new version of CTaps was released then an Application could update
its dependency by swapping the local shared object file, without having
to recompile the entire application. If the Application was statically linked
then it would have to update its dependency and recompile. This would also mean
that the entire Application would have to be redistributed, instead of simply
having users update their CTaps installation whenever needed. This becomes
very noticable in situations with critical updates, especially if multiple
applications rely on CTaps. If a criticial security error was found in CTaps
and it was statically linked
then every user of downstream Application would have to update every Application
using CTaps. If it was dynamically linked, then the user could simple update
a single shared object file.

As a limitation of scope in the original thesis description, CTaps is only
targeted towards Linux. However, since I/O and other system calls are handled
by libuv which does support windows, CTaps is likely very close to being
Windows-compatible, but this has not been tested \cite{Libuv2025}.
Certain aspects of the build system are Linux-specific, such as GCC's
\mono{__attribute__((visibility("default")))}, but this could quite
easily be adapted to a Windows build, as has been done with for example
libuv.

The library is distributed through GitHub.
