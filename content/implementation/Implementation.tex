This chapter will cover how the design decisions described
in~\cref{chap:design} are implemented in CTaps.
This chapter will focus on how the internal architecture and
internal TAPS functionality were realized, in addition

\section{Internal Architecture}
This section will describe the key implementations
for achieving the internal architecture of
CTaps as described in~\cref{sec:ctaps-arch-design}.
This includes the Socket Manager as an intermediary
between the Connections and the underlying protocol implementations, the protocol interface
which provides a common API for the Socket Manager to interact with, and
the internal communication and error handling in CTaps.

\subsection{Socket Manager} \label{sec:socketManager}
The need for a Socket Manager was described in~\cref{sec:socket-manager-design},
with the most important responsibilities being managing the lifetimes
of sockets as well as acting as an intermediary between Connections
and the various protocol implementations.

Since a single socket can be shared between many Connections,
the Socket Manager keeps track of the number of open Connections
in order to ensure that the underlying socket is only closed when
the last Connection using it is closed. The underlying protocol
implementation notifies the Socket Manager that a Connection has
been closed asynchronously, at which point the Socket Manager
considers whether the socket should be closed or not, as seen
in the callback in
\cref{lst:socket-manager-close-callback}.

Separately from keeping track of the number of open Connections,
the Socket Manager is also reference counted in order to ensure
that the internal state remains valid as long as at least one associated
object remains valid, as can be seen in~\cref{lst:socketManagerRefCount}.

\begin{listing}[htbp]
  \begin{minted}{C}
void socket_manager_decrement_ref(SocketManager* socket_manager) {
  socket_manager->ref_count--;
  log_debug("Decremented socket manager reference count, updated count: %d", socket_manager->ref_count);
  if (socket_manager->ref_count == 0) {
    int rc = socket_manager->protocol_impl.stop_listen(socket_manager);
    if (rc < 0) {
      log_error("Error stopping socket manager listen: %d", rc);
    }
    free(socket_manager);
  }
}
  \end{minted}
  \caption{Internal Socket Manager counter decrementation}
  \label{lst:socketManagerRefCount}
\end{listing}

\begin{listing}[htbp]
  \begin{minted}{C}
void ct_socket_manager_listener_closed_cb(ct_socket_manager_t* socket_manager) {
    log_debug("Socket manager listener closed callback invoked");
    ct_listener_t* listener = socket_manager->listener;
    int num_open = ct_socket_manager_get_num_open_dependents(socket_manager);

    if (num_open == 1) {
        log_debug("Socket manager now has no open connections, closing entire socket manager");
        socket_manager->close_reason = CT_CLOSE_TYPE_GRACEFUL;
        ct_socket_manager_close(socket_manager);
        return;
    } else {
        log_debug("Socket manager has %d open connections after stopping listener, not closing "
                  "socket manager",
                  num_open);
    }

    ct_listener_mark_as_closed(listener);
    ct_socket_manager_notify_of_listener_close(socket_manager, listener);
}
  \end{minted}
  \caption[Socket Manager connection closed callback]
  {Socket Manager callback invoked my protocol implementation on connection close, showing how it only closes the underlying socket when the last Connection closes.}
  \label{lst:socket-manager-close-callback}
\end{listing}

\subsection{Protocol Interface}
As mentioned in~\cref{sec:ctaps-arch-design} the protocol interface
in CTaps provides a common API for the Socket Manager to communicate with the
underlying protocol. This interface itself is implemented as a struct
which defines the required properties and function properties which a
protocol implementation must implement.

The concrete protocols themselves are then defined as global instances
of this struct and are stored in a global array of currently supported protocols.
An example declaration can be seen in~\cref{lst:udp-protocol-decl}, where
the global UDP protocol implementation is declared in \mono{udp.h}. Since
the protocol implementations themselves have no state, it is safe to use a
single global instance of each protocol among multiple Socket Managers.

\subsection{Error Handling} \label{sec:internal-error-handling}
Internal error handling in CTaps matches the external
handling, except that CTaps does not verify that parameters
are legal in internal functions except where strictly necessary
and that internally, CTaps uses the same function for success paths
and failure paths, instead passing a NULL pointer as a result when operations fail.

Similar to libuv, if a CTaps function returns a synchronous error then any error callback
will never be invoked.

\section{Connection Lifetime}
When an application initiates a Connection in CTaps it flows through
a series of distinct phases, from first gathering candidates for the
Connection and racing these candidates to finally transferring data
and being torn down. This section will describe how each phase is implemented
internally within CTaps.

\subsection{Candidate Gathering} \label{sec:candidateGathering}
Before a Connection can be established, a list of possible Local and Remote Endpoints
needs to be generated based on the data passed to the associated Preconnection.
In CTaps this candidate gathering follows the procedure described in RFC~9623
very closely.

Using N-ary tree as described in RFC~9623~\cite{rfc9623},
CTaps first builds the entire tree and resolves
all possible local interfaces, all protocols, specified ALPNs and all possible resolutions of
remote hostnames respectively. Each level in the tree corresponds to a specific
one of these options and the tree branches whenever there are multiple options for a specific level.
The final leaves of the tree, which are a combination of a specific Local Endpoint, a specific protocol
stack and a specific Remote Endpoint, are referred to as a candidate.

After the entire tree has been built, CTaps prunes the branches which
are not compatible with the required or prohibited selection properties. After pruning,
the tree is sorted according to the stated selection properties, as described in RFC~9623 
\cite{rfc9623}. Finally, the leaves of the tree
are extracted into an array of candidates which is then used for candidate racing.

\subsection{Candidate Racing} \label{sec:candidateRacing}
After a list of candidates has been gathered, CTaps
will race them according to the specification from RFC~9623
\cite{rfc9623}. 

The goal of candidate racing is to successfully negotiate a protocol stack, local
and remote endpoints based on the user's specification. Candidate racing in a TAPS
system follows the same procedure as specified in Happy Eyeballs V2~\cite{rfc8305}.
TAPS takes racing a step further by not only racing IP-address families, but also
racing potential protocol configurations and endpoints.

CTaps follows a staggered
approach, first sorting the gathered candidates by preference and performing a connection
attempt for each, with a delay between each candidate. All outstanding racing attempts
are canceled upon the first success.

Since CTaps has no internal cache and therefore no support for RTT estimation, the delay
between each candidate is set to a fixed \qty{250}{\milli\second} which is the recommended value
in Happy Eyeballs V2~\cite{rfc8305}. In a real-world
use case this is a major limitation, which is discussed further in~\cref{sec:discussion}.

An example flow of candidate racing can be seen in~\cref{fig:racingSeqDiagram}.
Here, DNS resolution resolves two endpoints with delays of
RTT of \qty{300}{\milli\second} and \qty{100}{\milli\second} respectively. The slower endpoint is attempted first,
but the faster RTT leads to the second endpoint winning the race and being
presented to the application.
In the example the DNS resolution is set to take \qty{100}{\milli\second}
which is a arbitrary value with no influence on the following candidate racing.

\subsection{Data Transfer}
In CTaps, applications can request to receive data on a Connection
at any time. CTaps will then either cache the passed \mono{receive()}
callback until a message is received, or invoke it immediately with
a cached message.
Each invocation only receives
a single message, and will always receive the first message in
the message queue.

When sending messages the message is not cached in CTaps, instead it is
immediately passed to the libuv protocol handles or to Picoquic, depending
on the underlying protocol. The message
may queue here for a while, depending on the number of tasks
in the loop.

\subsection{Connection Teardown}
Since both aborting and closing Connections may involve over-the-wire teardown,
all Connection termination is treated asynchronously by CTaps, but sometimes
the callbacks may be invoked immediately, such as with UDP which does not
require any asynchronous teardown.

\section{Connection Cloning}
CTaps support connection cloning through the \mono{ct_connection_clone()}
function, which allows applications to create a new Connection to the same
endpoint, either through normal connection establishment or through
multistreaming if supported by the underlying protocol.

After cloning a Connection, the resulting clone is passed to the application
through the \mono{ready()} callback registered with the source Connection.
The clone will be organized in the same Connection Group as the source Connection,
and will therefore share most Connection Properties with the source Connection,
and will be affected by Connection Group operations such as \mono{ct_connection_close_group()}. 

The Connection Properties that are not shared are individual Connection state, the Connection
priority and the Connection's \mono{can_send} and \mono{can_receive} flags. The
internal Connection state track whether the Connection is initiating, open or closed.
The Connection priority is used to prioritize data transfer between the Connections
in the same Connection Group, while the \mono{can_send} and \mono{can_receive}
are used to keep track of whether the Connection is unidirectional.

Connection cloning is naturally the most useful when the underlying protocol can make
use of multistreaming, since this allows the clone to reuse the same underlying connection.
Connection cloning
can however still be performed no matter what the underlying protocol is. This allows
TAPS to be protocol-agnostic, attempting to make use of benefits
like multistreaming when available, but being functional even if the underlying protocol
does not support it.

The implications of the application having to make an effort to write
protocol-agnostic code when using Connection cloning in CTaps is discussed
in~\cref{sec:quicUnderTaps}.

A theoretical example displaying the benefits of Connection cloning and multistreaming
in general can be seen in~\cref{fig:connection-clone-gains}. Here, two
transfers are performed sequentially. After the first transfer completes successfully,
the application makes use of connection cloning in CTaps and starts a second transfer.

If the underlying protocol is TCP then CTaps will initiate a new connection and transfer
successfully, but if the underlying protocol is QUIC then it will simply open a new
stream and therefore benefit from stream multiplexing.

The second transfer
has significantly higher throughput over QUIC than TCP, since TCP has to go through
slow start again for the second transfer. In CTaps a single client can use both
TCP and QUIC this way, supporting connecting to a TCP endpoint, but making use
of QUIC benefits when available.

A benchmark showing the measured benefits of utilizing Connection cloning in
CTaps over QUIC vs. TCP can be seen in~\cref{sec:smallfiletransfer}.

\section{Connection Migration} \label{sec:connection-migration-impl}
Connection migration is supported in CTaps by default, but multipath
is by default disabled for clients in RFC~9622~\cite[Sec. 6.2.14]{rfc9622}.

If multipath is set to active
then CTaps will probe all Local Endpoints registered with the Connection through
Picoquic when Picoquic notifies CTaps that a path has become unavailable.

Since Picoquic does not control the underlying sockets itself it relies
on CTaps notifying it when the socket error implies that a path is unreachable.
CTaps does this on any error code from \mono{sendmsg()}.

While the current support for Connection Migration in CTaps
is useful for showcasing Connection Migration as a concept,
it is also fairly limited by the internal CTaps architecture.
Currently a Socket Manager can only be associated with a
single socket, this socket is opened on initiation of
the first Connection using it and it cannot be updated
for the lifetime of the Socket Manager. There is also
no support for a Connection switching to a different Socket Manager.

For TCP and UDP
this is not a large issue for normal usage, as they are tightly bound to their original
socket. QUIC is however limited by being able to use only a single local.
In CTaps this is seen through
the requirements for connection migration to function as expected.

With how candidate gathering is implemented in CTaps,
the first Local Endpoint passed to a Preconnection is likely
the one that will be attempted first in candidate racing, and
therefore also likely the one that the final socket will be bound to for a 
successful connection.

Since the Linux kernel only delivers incoming packets with a
dest address which matches the exact bound IP of the specific socket,
CTaps will likely not be able to make use of multiple local
IP addresses if an application states the first local IP address
explicitly, even if the application passes multiple Local Endpoints
to a Preconnection.

If an application states
an wildcard address as the second Local Endpoint to a Preconnection,
then CTaps could in theory make use if this instead, but this
is unlikely due to how ordering is implemented in candidate gathering.

Lacking this feature was deemed acceptable since CTaps is
a research project and a workaround exists, allowing applications
to utilize path migration as seen in the benchmark in
\cref{sec:path-migration}. Improvements to achieve
a more robust connection migration without having to
perform CTaps-specific workarounds are discussed in~\cref{sec:future-work}.

The described workaround is to explicitly bind to an wildcard endpoint,
while still passing alternative paths to the Preconnection when first
initiating a Connection.
CTaps only probes paths explicitly passed to the Preconnection,
and therefore alternative paths must be passed to the Preconnection,
even if binding to an wildcard endpoint.

An example of utilizing
this can be seen in the excerpt from the migration benchmarking
client featured in~\cref{lst:explicit-binding-to-ephem}.

A TAPS system can in support migration to both a new local and a new remote endpoint
since the application is able to add both multiple Local and multiple Remote Endpoints
to a Connection.
Whether a system actually makes use of the multiple endpoints is however left up to the
underlying transport protocol~\cite[Sec. 7.5]{rfc9622}. For QUIC CTaps therefore only 
supports intitating migrating to new Local Endpoints and only supports
migration initiated by a client. Migration initiated by the remote
is supported as long as the \mono{multipath} Selection Property is not
disabled.

\section{Framers}\label{sec:framer-impl}
Framers in CTaps consists of a structure with two function
pointers, which are expected to be implemented by downstream applications.

Since the Connection takes a deep copy of passed Messages, the
Message data itself is owned by the protocol stack. The 
Framer can therefore modify the message in place and does not have to allocate new
memory unless it is extending the length of the message.

There are currently two major limitations with how framers are implemented
in CTaps in that each Connection object currently only supports a single
framer and that framers are designed to only operate on a single message
at a time. Mitigations for both of these limitations are discussed as
pieces of future work in~\cref{sec:future-framer}.

The Framer interface as defined in CTaps can be seen in~\cref{lst:framer-struct-def},
while an example implementation from the CTaps framing tests can be seen in~\cref{lst:framingTestImpl}.

                                   ct_message_t* message,
                                   ct_message_context_t* context,
                                   ct_framer_done_encoding_callback callback) {
    // Prepend the message length as a single ascii character
    // First byte is the length

    memmove(message->content + 1, message->content, message->length);

\section{Protocol Implementations}
\subsection{Supported protocols}
Currently, CTaps natively supports three transportation layer protocols:
UDP, TCP and QUIC. TCP and UDP were added because of their ubiquity and
relative ease of implementation due to native support in libuv. QUIC was
added since having easier access to QUIC was one of the primary motivations
behind the thesis as a whole.

\subsubsection{UDP} \label{sec:udpImpl}
UDP is the simplest of the three currently supported protocols in CTaps
and it has native
support in libuv through the libuv \mono{uv_udp_t} handle~\cite{libuvLibuvUDPHandle}.
The relative protocol complexity is discussed further in
in~\cref{sec:protocolComplexity}.

UDP does not fit perfectly into the model assumed by
TAPS, which is that of a Connection-based system. There are therefore
several pieces of logic in the CTaps UDP implementation which adapt
the underlying UDP to fit into a TAPS model.

When opening a client, CTaps
simply creates a new UDP handle through libuv, starts receiving data over that handle, and
invokes the \mono{ready()}-callback for the associated Connection.
This does not actually transfer any data to the remote endpoint
and CTaps can therefore not actually guarantee that the remote endpoint
is reachable via UDP.

In addition, when CTaps listens over UDP it has to emulate
the concept of a Connection. CTaps uses the Socket Manager
to demultiplex incoming UDP messages, mapping Remote Endpoints
to Connections.

The list of selection properties set for UDP by CTaps is discussed
in~\cref{sec:protocol-impl-sel-props}.

\subsubsection{TCP}
TCP is one of the three protocols currently supported by CTaps, and has native
support in libuv through the
\mono{uv_tcp_t} handle~\cite{libuvLibuvTCPHandle}.

Overall, TCP fits relatively well into the model
provided by the TAPS API.
TCP is however stream-based, while TAPS assumes a message-based model.
For CTaps this means that every time libuv notifies CTaps of having
received new data over a TCP socket then the received data is wrapped into a
single message and passed to the application.

While the application can use a framer to wrap or unwrap received messages, this currently
functions as a proof of concept in CTaps, and does not currently have an internal buffer
to support combining multiple TCP packets into a single Message.

Since TCP is also connection-based, listening and creating new sockets
for incoming connections is handled by the TCP stack itself and there is
therefore far less CTaps-specific logic for listening on TCP than there is
on listening over UDP. For example, libuv
has the built-in function \mono{uv_listen}, which invokes a callback whenever
a new connection is received. Since the whole protocol is connection-oriented,
it also means that the logic of only closing the receiving socket when
the last connection closes is built-in, and does not have to be implemented
by CTaps.

CTaps still uses a Socket Manager to keep track of the open TCP
handle, but this only tracks single listener and can be
closed immediately when the listener is closed. The new TCP handles which
have been spawned from this Listener will have received their own separate
Socket Managers, since they are connected to separate sockets.

The list of selection properties set for TCP by CTaps is discussed
in~\cref{sec:protocol-impl-sel-props}.

\subsubsection{QUIC} \label{sec:quicImpl}
With the standardization of QUIC being fairly recent,
none of the current implementations of TAPS
feature QUIC as one of the underlying protocols.
This makes CTaps particularly unique, as it is the first major piece of
work attempting to map a QUIC to an implementation of the TAPS API.

Within CTaps, QUIC is by far the most complex supported protocol,
which can be seen through the required number of lines of code in 
\cref{sec:protocolComplexity}. During the development of CTaps,
particular care has been taken to find out which features of QUIC
do not map cleanly into the TAPS API. These features are discussed further in
\cref{sec:quicUnderTaps}.

From an application's point of view there are several TAPS configurations which a client has to set in order to use
QUIC through CTaps. This is a symptom of QUIC being the only supported
protocol which uses TLS, and is not a weakness of the TAPS API itself.

When interacting with Picoquic, CTaps is responsible for writing and reading
data to and from the network, while Picoquic formats outgoing QUIC packets and interprets incoming data.
CTaps will regularly query Picoquic for when it needs to be woken up next
and will use a libuv timer to wake up at the appropriate time.
Within these wakeups, CTaps will also query Picoquic for any new packets which it needs to send over the network.
If Picoquic has data to send, it will provide both the source and the destination IP address to use,
in addition to the actual packet data.

Libuv does not currently
support setting the from address on a regular \mono{uv_udp_t} handle,
as \mono{uv_udp_send()} does not feature a from address as a parameter~\cite{libuv_udp_send}.
Packets sent over a \mono{uv_udp_t} handle will therefore use the
IP address the socket is bound to as a from address.
If the socket is bound to a wildcard address, then the kernel will
select the from address itself.

This becomes an issue for CTaps when utilizing the connection migration
workaround described in~\cref{sec:connection-migration-impl}, as 
the socket currently has to be bound to an wildcard address for migration
to work. If the from address does not match the address expected
by Picoquic then it cannot successfully reason about which paths are
available and which are not. This causes Picoquic to be unable to perform connection migration
as expected.

CTaps therefore
creates its own UDP socket through the \mono{socket()} system call instead of using a \mono{uv_udp_t} handle.
The resulting file descriptor is
monitored using a generic
\mono{uv_poll_t} libuv handle. This does not create much extra
complexity, apart from having to manually build the message header
as seen in~\cref{lst:pktinfo-send-packet} in~\cref{app:internal-code-examples}.
A model of the data flow between CTaps, Picoquic and the network can be
seen in~\cref{fig:ctaps-quic-loop}.

When initiating a CTaps Connection over QUIC, CTaps needs to open a socket on
the a Local Endpoint and inform Picoquic about wanting to open a QUIC connection.
If CTaps wants to listen for incoming QUIC connections, then it can simply
create a QUIC context and to start feeding Picoquic any arriving packets.

When the QUIC connection
is ready, Picoquic informs CTaps about this through a callback. CTaps will then
use a context pointer associated with this callback to find the associated
Connection Group and invoke the correct \mono{ready()} callback.

When closing a QUIC connection, CTaps will invoke
\mono{picoquic_close()} which will initiate a graceful teardown.
Picoquic will then inform CTaps about the connection being closed through a
callback, when the teardown has completed.

As mentioned previously, ALPN is mandatory for QUIC and is supplied to Picoquic
by CTaps through a configurable Security Parameter. ALPN is a negotiation
between a QUIC server and client, therefore, both a Listener using QUIC and
a client using QUIC has to take this parameter into account. In order to support
session resumption for QUIC, which only supports a single ALPN value, CTaps
treats the ALPN value as a protocol option and generates separate candidates for
each ALPN value when performing candidate gathering, if the underlying protocol
supports it.

In situations with many possible ALPN values, this
will lead to a larger delay in connection establishment than strictly
necessary. The consequences of this stagger delay as it is currently
implemented in CTaps is discussed further in~\cref{sec:single-endpoint-handshake-discussion}.

The full implementation of the protocol implementation struct for QUIC can be seen in
\cref{lst:quic-impl-struct} in~\cref{app:internal-code-examples}.

when deciding the order of racing and when deciding certain protocol behavior for
an active Connection.

explicitly set a value for a certain property. In addition, each protocol implementation
keeps track of which Selection Properties it can and cannot fulfill.

The default Selection Properties for each application are prescribed by TAPS in

to be to be compatible with as many applications as possible.

\section{Testing} \label{testing}
CTaps uses a test system built around Google Test~\cite{GoogleTest2025} and FFF
\cite{longFakeFunctionFramework2025}. Google Test is a C++ library developed by
Google which provides a rich set of macros to define tests and assertions
to verify behavior. FFF is a C library which provides macros to easily
create function stubs, which can then be used to track invocations
and simulate external behavior.

CTaps features two main types of tests, unit tests and integration tests.
Unit tests are isolated tests which verify the behavior of a single function, while
integration tests perform a larger check on the
whole library and often interact with a set of ping servers which have
been configured to run on localhost. The setup and teardown of these
ping servers are enabled through a standardized test fixture which is used for all
integration tests.

FFF can mock both functions within CTaps itself and functions from external libraries
such as libuv. Since mocking in FFF creates a new symbol with the same name as the mocked function
it is especially useful when combined with wrapping symbols at link time, 
such as with the \mono{-}\mono{-}\mono{wrap} flag in the GNU linker~\cite{gnuLinkerManual}.
This can be used to redirect all invocations of a function
to its mock at link time, avoiding any issues with symbol conflicts.

An example of function mocking using FFF can be seen in~\cref{lst:example-fff-mocking}.

\section{Build System} \label{sec:build-system}
CTaps is built using CMake and attempts to largely follow the same conventions
around build and structure as libuv. This includes
having the API consist of a single header file, \mono{ctaps.h}, which
is included to expose all functionality intended to be used by an application.

CTaps is also compiled with visibility of all symbols set to hidden by default,
with only the symbols marked as \mono{CT_EXTERN} being available in the final
shared library file. This is done to hide internal state from applications,
making it clear which parts of the compiled object file are part of the
external API and not.

By default CTaps is built as a shared library. This means that the library is compiled
into a single shared object file. When compiling an application using CTaps,
this shared object file is not included in the compiled binary, instead the
compiled binary looks for a shared object file at a given location at
run time. This means that any application depending on CTaps will by default
not include CTaps in its own binary, leading to a smaller file size, although this 
is naturally dependent on the build configuration of the application. 

If the application dynamically links against CTaps, then the largest downside
is that it requires its users to have CTaps installed on their
systems, in order for the application find the correct symbols at run time.

In addition to the smaller binary size, the largest upside of a dynamically
linked application is that it is far easier to update its dependencies.
If a new version of CTaps was released then an application could update
its dependency by swapping the local shared object file without having
to recompile the entire application.

If the application was statically linked
then it would have to update its dependency and recompile. This would also mean
that the entire application would have to be redistributed instead of simply
having users update their CTaps installation whenever needed.

This is especially relevant for networking applications where security
is a large concern. If a critical security error was found in CTaps
and it was statically linked
then every user of downstream application would have to update every application
using CTaps. If it was dynamically linked, then each user would only have to update
a single shared object file.

The library is distributed through GitHub~\cite{cTaps}. No binaries
are compiled by default, but the project can be downloaded directly
in CMake using FetchContent, as exemplified in the CTaps example project
\cite{ctaps-example-project}.

        GIT_TAG v0.1.0
)

FetchContent_MakeAvailable(CTaps)

\section{Dependencies}
As with most libraries, CTaps has several external dependencies
used to simplify the logic required to be implemented within CTaps itself.
These are again divided into statically and dynamically linked dependencies.

A description of the difference between statically linked and dynamically linked
dependencies can be found in~\cref{sec:build-system}. CTaps statically links against
Picoquic and log.c, while it uses dynamic linking for
more general dependencies which it expects application developers to have present
on their machine. This includes GLib and libuv.

Care has been taken to keep dependencies away from the external API. This means
that when compiling CTaps, dependencies are required to be present on the compiling
system, while downstream applications only require the runtime libraries to be present.

\subsection{GLib}
As mentioned in~\cref{sec:glib-design-reason}, CTaps uses GLib 
for certain internal data structures to avoid having to re-implement them
internally.
This includes the candidate tree when performing candidate gathering
and hash tables keeping track of Connections associated with a Connection Group.

\subsection{libuv}
Libuv is CTaps' most core dependency and it has therefore shaped the implementation
of CTaps itself. CTaps uses libuv abstractions where possible and relies on libuv
to perform all asynchronous interactions with the network. An example with this
can be seen in the CTaps DNS lookup function~\cref{lst:get-addrinfo-uv}, where CTaps relies on the asynchronous
\mono{uv_getaddrinfo()} to perform an asynchronous DNS lookup, instead of the synchronous
\mono{getaddrinfo()}.

\begin{listing}[htbp]
  \begin{minted}{C}
int ct_perform_dns_lookup(const char* hostname, const char* service,
                       ct_remote_resolve_call_context_t* context) {
    log_trace("Performing DNS lookup for hostname: %s\n", hostname);
    uv_getaddrinfo_t* request = calloc(1, sizeof(uv_getaddrinfo_t));
    if (!request) {
        log_error("Could not allocate memory for uv_getaddrinfo_t request");
        return -ENOMEM;
    }
    request->data = context;

    int rc = uv_getaddrinfo(event_loop, request, on_uv_getaddrinfo_cb, hostname, service, NULL);
    if (rc < 0) {
        log_error("Synchronous error in initiating DNS lookup for hostname %s: %s\n", hostname,
                  uv_strerror(rc));
        free(request);
        return rc;
    }
    return 0;
}
  \end{minted}
  \caption[CTaps DNS lookup function]{Example of using async \texttt{uv\_getaddrinfo} utility function instead
  of synchronous \texttt{getaddrinfo} system call.}
  \label{lst:get-addrinfo-uv}
\end{listing}

\subsection{Picoquic}
As mentioned, Picoquic does not interact with network 
sockets directly in CTaps, but instead integrates directly
with libuv using a libuv timer handle. CTaps queries Picoquic
for the next wakeup time after interacting with a QUIC instance,
and then the timer fires when necessary, to help Picoquic send
the necessary data to the network. The timer callback function
invoked by the libuv timer can be seen in
\cref{lst:timer-callback-function} in~\cref{app:internal-code-examples}.
