\chapter{Background}\label{chap:background}
This chapter will lay out background information which forms the
basis of the motivation as well as the background for both the
current design and limitations of CTaps.
It will describe the low level networking concepts,
transport layer protocols and the current barriers
for innovation on the transport layer. These
are all needed in order to understand the problem
that TAPS is trying to solve, as well as why
the problem exists in the first place.
Extra care is taken in describing QUIC, as exposing
the features of QUIC forms a basis for both the design
of CTaps, as well as the benchmarking in~\cref{chap:eval}.


\section{Network Interfaces and Paths}
A machine is considered to be multihomed if it is connected to more than
one network. In practice, this is often the case for modern devices,
where a laptop may be connected to both a Wi-Fi network and a wired Ethernet.

Each separate network is interacted with through a separate network interface,
although these are often virtualized, for example with the loopback interface.
Since each network interface is connected to a different network, each one
features a different IP address, and therefore may be used as a source
address for outgoing traffic, if packets can be routed through the associated network.

Given that a machine features more than one network interface which can reach
the same destination, this can be used to improve reliability and performance
of a connection. Each
local interface represents a different network path, but can
be used to reach the same destination.

Within QUIC, a network path is a term used for the combination of
source IP, source port, destination IP and destination port~\cite{rfc9000}.
This term has become necessary since QUIC decouples
the path from the connection itself. More details about this can be found in~\cref{sec:quic}.

\section{Network Sockets}
When it comes to network programming,
the network socket in its most basic definition is an abstraction
for a network connection, through which an application can send
and receive data~\cite[p. 8]{donahooTCPIPSockets2009}. 
First mentioned in 1970 in relation to ARPANET, the
simplicity of sockets and their ubiquity in all UNIX distributions
since version 4.2 of the Berkeley Software Distribution have
led to them becoming the de facto standard for network
programming~\cites{rfc33}[p.~502]{tanenbaumComputerNetworks2011}.

Modern sockets are standardized across UNIX-like systems through
the POSIX standard. Implementations of the C POSIX library define
the socket API primarily through the \texttt{sys/socket.h} header file~\cite{posix:2024:socket}.

When creating a socket on an UNIX-like system, the application
invokes the \texttt{socket} system call. This system call
\enquote{creates an endpoint for communication and returns a file 
descriptor that refers to that endpoint}~\cite{Socket2LinuxManual}.
This system call takes three parameters, the domain, the type
and the protocol which will be used for the socket.

The domain specifies the type of
addressing which will be used, for example \texttt{AF\_UNIX}
for local sockets, \texttt{AF\_INET} for IPv4 and \texttt{AF\_INET6}
for IPv6. The type specifies the protocol which will be used over the socket, for
example \texttt{SOCK\_DGRAM} for datagrams and \texttt{SOCK\_STREAM}
for a reliable stream of bytes. The type usually corresponds to a
single transport layer protocol, usually UDP and TCP, the two
most common transport layer protocols.

The protocol parameter is used if there are multiple protocols
which support the requirements given by the domain and type.

\section{Barriers to Protocol Adoption}\label{sec:transport-protocol-barriers}
The fact that an application has to specify both the domain
and the type of the socket means that the
underlying addressing and the transport layer protocol are bound at compile time.
Updating
the underlying transport protocol will therefore require at least recompilation and 
likely a refactoring of the application. 
This naturally makes adopting new transport protocols difficult, even
if a new protocol is better suited to the application's requirements.

Another challenge with the adoption of new transport layer protocols
is network middleboxes. Network Middleboxes have been
optimized to support TCP and UDP and may sometimes
drop packets if the protocol field in the IP header
does not match TCP or UDP.

The resulting difficulty in innovating on the transport layer
is known as network ossification. In an attempt to combat
this ossification, transport layer protocols are often built
using UDP as a substrate, allowing them to use the same IP
protocol number as UDP. Examples of this include QUIC \cite{rfc9000} and
SCTP over UDP \cite{rfc6951}.

While building on top of UDP this way does present some overhead,
it is fairly minimal with the UDP header only being \qty{8}{\Bytes}.

Applications can themselves implement needed features
such as reliability and congestion control.
This way of designing new protocols has become so common that
it has prompted an official RFC~from the IETF, describing
the best practices for implementing congestion control on top of UDP~\cite{rfc5405}.

\section{TCP}
TCP is by far the most common transport layer protocol used on the
internet today. It is a connection-oriented protocol and provides
a reliable, in-order transfer of a stream of bytes to the application layer.
It has also become common to use TCP for message-based applications,
using a framing protocol on top of TCP to divide the TCP byte stream into
discrete messages. A description of TCP was first published as an RFC~in 1974~\cite{rfc675}
and it has since grown to become
ubiquitous across the internet.

In TCP, connections are uniquely identified by a four-tuple consisting
of the source IP address, the source port, the destination IP address
and the destination port.

\section{UDP}
UDP has, along with TCP, grown to become one of the two dominant
transport layer protocols on the internet. It is a connectionless
protocol and provides an unreliable, unordered transfer of datagrams
to the application layer. It is often used for applications which
prioritize low latency over reliability, such as real-time applications.

UDP is designed to be a very lightweight protocol with as little
overhead as possible. It uses a very simple 8-byte header, containing 
only the source port, destination port, length and checksum.
Since UDP does not mandate
advanced features such as in-order delivery or reliability, it
allows protocols using it as a substrate
to develop their own algorithms for congestion control
or reliable delivery as needed.

The primary reason for why protocols choose to use UDP as a substrate
instead of developing directly over IP is network ossification. This
is discussed further in~\cref{sec:transport-protocol-barriers}.

\section{TLS}
TLS is a cryptographic protocol which provides secure communication
over a computer network. It is commonly used on top of TCP, and is
most visible as the basis for HTTPS, the secure version of HTTP.

TLS features its own handshake protocol when establishing a connection,
in order to exchange the cryptographic parameters needed for the connection.
This means that connections using TLS have to spend an additional round trip time (RTT)
during connection establishment, compared to a non-secure TCP connection.

TLS 1.3 also supports 0-RTT handshakes
which allow a client to send data during the initial handshake when resuming
a previous connection to a server~\cite{rfc8446}. This can be a source
of replay attacks if the message being sent is not idempotent.

As of 2026 TLS adoption has been very widespread,
with \qty{95}{\percent} of all connections in Chrome on Windows using an encrypted
connection~\cite{google2026transparency}.

\section{QUIC}\label{sec:quic}
QUIC is a protocol developed by Google and formalized
through RFC~9000 in 2021~\cite{rfc9000}.
Intended to be a replacement for TCP, it has achieved
a fairly high adoption, for example with over \qty{75}{\percent}
of traffic to Meta servers using QUIC in 2020~\cite{joras2020quic}.

QUIC is a connection-oriented protocol and provides reliability,
despite being implemented in user-space on top of
UDP. QUIC enables several features which either are not available
or do not have a wide adoption in TCP, among these are
0-RTT connection establishment and multistreaming.
TLS is a mandatory part of the protocol and QUIC
also encrypts its header, unlike TLS over TCP.

While QUIC does present an overarching connection, the
abstraction for each actual data transfer is called a stream.
A stream is an either
unidirectional or bidirectional ordered stream of data, much
like what an application sees when using TCP. Streams can be created and
closed by either endpoint, and multiple streams can exist over a
single QUIC connection. QUIC mandates support for in-order delivery within a single
stream, but not between streams. This way a new stream can be
used for each object transferred, solving the issue of
head-of-line blocking which is present in TCP.

For QUIC, a single connection can exist over several paths and
congestion control is done on a per-path basis. Multiple streams
can be transmitted over a single connection and data from
multiple stream frames can be transmitted in a single packet.
If a packet is lost, only applications using the streams contained
in that packet will be affected.

The format of a QUIC stream frame as described in RFC~9000 can
be seen in~\cref{lst:stream-frame-format}.

These states differ whether the stream is sending or receiving
data.
A bidirectional stream consists of both sending and receiving parts,
and can therefore exist in states belonging to both sending and
receiving streams as described previously.

Much like TCP, QUIC also implements congestion control, there are
however some key differences between the algorithms. A key concept
to QUIC's congestion control is the 'packet number space'. Packet
number spaces are \enquote{the context in which a packet can be processed
and acknowledged}~\cite{rfc9000}. 
This context reflects the level of encryption and there 
are therefore three separate packet number spaces. Initial space,
for the initial handshake packets, handshake space, for packets
exchanged during the cryptographic handshake, and application
data space, for packets exchanged after the cryptographic handshake
has been completed. Separating these spaces allows QUIC to
avoid spurious retransmission from one space affecting another,
and allows QUIC to handle retransmissions more efficiently, for
example by using a different expected acknowledgment delay for packets sent
in the initial space, as these are not expected to be cached
by the peer. Although the packet number spaces are handled
separately when it comes to acknowledgments, they are still
unified when it comes to congestion control and round trip
time measurements.

Another difference between how QUIC and TCP handle acknowledgments
is that QUIC separates the transmission order from the delivery order.
This is done by using stream offset as delivery order, and packet number as
transmission order. This removes ambiguity about whether the
original or the retransmitted packet is acknowledged in the case
of retransmissions, making RTT measurements in these cases more accurate~\cite{rfc9000}.

Future network ossification related to QUIC is also mitigated 
by mandating TLS 1.3 and by encrypting the header.
Unlike TLS which does not encrypt the TCP header, QUIC obscures
as many details as possible from middleboxes, ensuring that they
cannot grow reliant on the contents of the packet.

\subsubsection{Transport parameters}
Transport parameters in QUIC are settings which are transferred to the
peer during the QUIC handshake. These can be used to for example
limit the initial amount of streams the peer can open or the amount
of data it can send.

\subsection{Flow control}
By default QUIC provides a guarantee of reliable in-order delivery
within streams, but provides so such guarantee between streams.
Applications can use this to transfer each object through a 
separate stream. This avoids head of line blocking where the
transfer of a small object potentially has to wait for the
transfer of a previous large object to complete, due to the
guarantee of in-order delivery. This is a problem with TCP
when several objects are transferred over a single connection,
which is treated as a single stream.

\subsection{Frames}
In QUIC a frame is defined as \mono{a unit of structured protocol information}~\cite{rfc9000}.
The payload in QUIC packets after discounting packet protection consists
of a series of complete frames.
Different types of frames are used to carry different types of information
and data. Examples are \mono{STREAM} frames which are used to carry
data over a stream or \mono{NEW_CONNECTION_ID} which are used
to share new valid connection IDs with the peer.

\subsection{Connection ID}
In order to uniquely identify a connection QUIC uses connection IDs
instead of a four-tuple like TCP. Each endpoint generates a list of
valid IDs which it shares with the peer. The peer can then use one
of these IDs as a destination when transferring data~\cite{rfc9000}.

Since Connection IDs are needed for routing packets they are
carried in the cleartext part of both the short and the long QUIC header.
The long header carries a source connection ID in addition to the mandatory
destination ID.

Since the connection IDs are in the cleartext part of the QUIC header
they could be used to link a QUIC connection accross events such
as path migration. In order to mitigate this each endpoint
can siwtch the used destination ID to a new one from the list
generated by the peer.
In order to prevent observers from being able to link a connection
through its connection IDs, QUIC connections rotate the IDs used.
A connection can switch the destionation connection ID it is using
at any time to one of the approved IDs previously shared by the peer. Endpoints
must also switch connection IDs when transmitting from a new local address, such
as after a path migration~\cite[Sec. 9.5]{rfc9000}.
New valid connection IDs which the peer can use as destination IDs can be
shared through the \mono{NEW_CONNECTION_ID} frame.

Connection IDs make addressing in QUIC independent of addressing in lower layer
protocols~\cite{rfc9000}. This allows QUIC to survive
changes in the underlying four-tuple and therefore enables connection migration.

\subsection{Connection Migration}\label{sec:connection-migration}
% TODO - only clients can currently migrate, disable_active_migration
% multipath vs. migration
Since a QUIC connection uses connection IDs to identify a connection,
this means that QUIC connections can survive changes to the underlying
four-tuple, unlike TCP.

This means that an endpoint can receive packets from a new remote
endpoint which was not previously associated with the peer. This
can take two forms, either a probing packet containing a \mono{PATH_CHALLENGE}
frame or a normal non-probing frame. Probing is used to validate 
that a peer can receive data over a path, while connection migration
can be initiated by simply sending data over a new local endpoint.
Since each endpoint validates the that the peer is willing to receive
data on their remote endpoint the initial
handshake, an endpoint can initiate local path migration without
first validating a new path.

In the current version of QUIC, only clients are able to migrate~\cite[Sec. 1]{rfc9000}.
This means that a client knows that a server will continue to send
on the same local endpoint for the lifetime of the connection, while
a server should support migration from the client. A client can request
that the client migrates to a new local address for the server through
by stating a preferred local IP address through the \mono{preferred_address}
parameter, but this can only be done during the initial TLS handshake~\cite[Sec. 9.6.1]{rfc9000}.


If the server does not
support migration, for example if it is behind a load balancer which multiplexes
by source IP instead of the QUIC connection ID, then the server can mandate
that the client does not change local address by setting the \mono{disable_active_migration}
transport paramter~\cite[Sec. 5.2.3]{rfc9000}.

If an endpoint recieves new non-probing frames from a new remote endpoint which is
not associated with the peer then it must initiate path validation in
order to verify the peers ownership of the endpoint.

Path validation is done by using
\mono{PATH_CHALLENGE} and \mono{PATH_RESPONSE} 
frames. The challenge frame contains \qty{8}{\Bytes} of arbitrary data,
which the peer must echo back in the response frame. 
Connection migration can be initiated without first validating
the new local path, since the remote is
previously validated. The peer initiating the 
path migration can defer local path validation until
after receiving data on the new local endpoint~\cite{rfc9000}.

If the endpoint validation fails, which happens if no matching response
is received within a given time period, the endpoints can continue
communicating using other known addresses. If the connection has no
viable paths the connection fails. This can be signaled to the peer 
using a \mono{NO_VIABLE_PATH} connection error, although this is only
possible if a path exists but does not support the required MTU~\cite{rfc9000}.

An endpoint can start sending from a new local endpoint either voluntarily
or because the previous local path has become unavailable. This means
that in situations where for example a Wi-Fi network goes down then QUIC
is able to continue the same connection over cellular, whereas the TCP
connection would break.

A figure comparing a TCP and QUIC connection before and after an
IP address becomes unavailable can be seen in~\cref{fig:connectionBeforeAfterMigration}.
Here, a local address becomes unavailable and the TCP connection is therefore broken,
as with a new IP address it would be considered a completely new connection. The
QUIC connection on the other hand is able to survive this change, as the connection
ID is recognized by the peer, even if it is changed on connection migration to avoid
linkability as recommended by RFC~9000~\cite{rfc9000}, since
it was picked from the pool of valid IDs generated by the peer.

In addition to supporting multiple paths through connection migration,
support for using multiple paths simultaneously through a single QUIC 
connection is currently actively being
standardized through the QUIC multipath extension draft~\cite{ietf-quic-multipath-21}.
Multipath extension builds on the mechanisms of connection migration,
but introduces a new path ID in order to manage multiple paths.
Using multiple paths for a single connection at once can be used to
improve both the resiliency and the performance of a connection.

The QUIC multipath extension draft does not specify how
alternative paths are discovered, and actual setup and
teardown of paths is handled by the application.

\subsection{Sesson Resumption}
Like TLS, QUIC supports session resumption and 0-rtt
initial data. This requires that the server and the client
have already exchanged cryptographic keys, that the
client remembers critical parameters such as the selected
ALPN and that the client provides the server with a
session ticket for the server to recover the same
information~\cite[Sec. 4.6]{rfc9001}.

\section{Transport Services}
At its core, a protocol is an abstraction which provides a set of
services to an upper-layer protocol or application. In the context of
networking, these services encompass transporting data from one endpoint
to another, with the individual features of a transport service deciding the behavior of
this service. For example, TCP provides a guarantee of reliable
delivery.

At their core, transport layer protocols are chosen based
on the services provided to the application, not because of the
protocols themselves. If an application requires that the
transferred data is received as an in-order stream, then
any protocol offering that as a service to the application layer
could function as a transport layer protocol, and swapping the
underlying transport layer protocol should in theory be simple.

Exactly which Transport Services are provided by which transport
protocols has been mapped by the IETF in RFC~8095, as part of the
TAPS working group~\cite{rfc8095}.

The TAPS WG was an IETF working group which aimed to solve the strong
coupling between application
level code and the underlying transport protocol through
focusing on Transport Services. This is intended to allow faster adaptation
of new protocols and abstract away network-layer details from the
application layer.

From the application's point of view this
means that it will be able to gain access to new protocols, such
as QUIC, without having to implement its own fallback logic
in cases where the new protocol is not supported by the remote.
It will also simplify the required networking logic by
tying it more closely to how modern applications actually interact
with the network.

Despite being implemented in user-space, TAPS conceptually resides
in the transport layer. TAPS provides convenience to the application
layer by expanding the responsibility of the transport layer. Instead
of the application layer being responsible for resolving addresses and
for choosing the specific transport layer API to use,
TAPS provides a unified interface which
both handles simple interactions with the network for resolving hostnames
in addition to unifying the transport protocol APIs. This minimizes
how much the application layer has to adapt to the underlying transport
layer, and minimizes the coupling between the two layers.

The difference in API models for socket and for TAPS can be
seen in~\cref{fig:socket-programming-api-model} and
\cref{fig:taps-api-model}.

\subsection{TAPS Architecture} \label{sec:tapsArch}
RFC~9621 describes the underlying architecture of and a set of requirements
for TAPS, a diagram of this architecture can be seen in~\cref{fig:tapsArch}~\cite{rfc9621}.
A TAPS system consists of two primary parts, the Transport Services
API and the Transport System Implementation. The Transport Services
API is the public facing API used by the application. This contains
concepts defined in RFC~9622 such as a \term{Preconnection}, a \term{Connection}
\term{Messages} or \term{Events} being transferred between TAPS and the application.

The Transport System Implementation is the internal workings
of a TAPS system. This consists of services which are provided to the
public API, such as data transfer or candidate racing. The TAPS RFCs
do not prescribe an actual detailed implementation of the Transport
System Implementation, leaving the details up to each TAPS implementation
itself. RFC~9623 does however offer a set of suggestions to developers
of a TAPS API in order to be in line with the TAPS specification,
however these are not mandatory, and any TAPS system is therefore
free to implement its internal workings as it sees fit.

\subsection{TAPS abstractions}
Abstracting away the underlying transport protocols is done by
presenting a new set of abstractions to the application. The most
central of these abstractions is the Connection. A Connection
represents a \enquote{shared state of two or more endpoints that persists
across Messages that are transmitted between these endpoints}~\cite{rfc8303}.

When working with Connection objects, TAPS provides five high
level groups of actions that an application can perform:
\begin{itemize}
    \item Preestablishment: Pass properties to specify
        requirements and preferences for establishing a Connection.
    \item Establishment: Prepare a Connection object for data
        transfer.
    \item Data transfer: Sending and receiving Messages
        over the Connection.
    \item Event handling: Notifications that an application
        can receive from a Connection, and the handling of them.
    \item Termination: How transmission is stopped and
        a Connection is torn down.
\end{itemize}

In addition, the lifeline of a Connection is divided into three distinct
phases, preestablishment, the established state, and
termination of a Connection~\cite{rfc9621}.
Each of these phases are associated
with one or more of the groups of actions that an application can perform,
as described above. 

After the desired Transport Properties and cryptographic parameters have
been set during the preestablishment phase, a Connection can be established. 
When a Connection is in the established phase the Connection can be used to
transfer data. 

There are three ways of
establishing a Connection: \mono{initiate()}, \mono{listen()} and
\mono{rendezvous()}~\cite{rfc9621}.

Invoking \mono{initiate()} establishes a Connection to a remote endpoint which
is assumed to be listening for incoming Connections. How this Connection
is established internally is dependent on the underlying transport
protocol, which has been selected based on the specified Transport Properties.
The Created Connection object is passed to the application through a callback.

Invoking \mono{listen()} is used to passively open a Connection. This is done in order to
create a Listener object and wait for Connections from Remote Endpoints.
When a Remote Endpoint connects to this Listener, a Connection Received
event occurs. This event contains a Connection object for the 
established Connection.

Invoking \mono{rendezvous()} is used to establish a peer-to-peer Connection between two endpoints
simultaneously.

After a Connection has been established, it can either
be closed or aborted, with closing being associated with normal termination,
and aborting being abnormal termination associated with errors.

In addition to the Connection, TAPS also provides several other objects
abstracting key elements. For example there are Endpoint objects and Message objects.

\subsection{Connection Cloning}
One or more TAPS Connections can be organized into a Connection Group. These
are used to group related Connections together, which have been created
by cloning. Connections in the same Connection Group can be
multiplexed together if this is supported by the underlying protocol.

\subsection{Framer}
In TAPS a Framer is responsible for bridging the TAPS concept
of a Message, with the underlying protocols concept of either a data stream
or a datagram. In their simplest form, a Framer is responsible for encoding
or decoding outgoing and incoming data, as well as sending any data necessary
on initiation or closing of a Connection. Because Framers may need to send
data upon the creation of a Connection, they must be added to a Preconnection,
before any action which may result in the creation of a Connection is taken.
The application can define custom framers and pass them to a Preconnection
by implementing an interface provided by the TAPS system.

Because of this potential need to interact with the network, 
framers are also asynchronous. They are therefore callback-based and need
to notify their associated Connection when the processing of a given Message
has completed.

The concept of a framing protocol also exists outside of TAPS because most
application-level protocols are both message-based and run over TCP. Several
ways of converting TCP's byte-stream into usable messages has been implemented.
These largely fall into two categories, type-length-value (TLV) structures where
the length of the following data is described in a header-structure, and
delimiter-separated formats, where the beginning and end of a piece of
data is separated by delimiters.~\cite{rfc9623}
Examples of the two categories are HTTP/1.1 for a delimiter-separated format
and DHCP for TLV, specifying the length of the given hardware address, allowing
parsers which are unfamiliar with the type to skip it, since the length is known~\cite{rfc2131}.

For a TAPS system, where the application layer should not need to take into
account which application layer protocol it is running on top of, Framers become
useful for datagram or message-based protocols as well. By using Framers, the
application layer protocol can ensure a consistent reception of Messages in
the desired format. Framers can also pack multiple Messages into a single transport
datagram by using the same formatting as it would for a stream-based protocol.

Conceptually, a framer resides on a separate layer between a Connection and the
underlying Transport Protocol Stack as seen in~\cref{fig:tpsStack}.

\subsection{Transport Properties}
Transport Properties are properties which are set in TAPS in order to either
influence which transport protocol should be used, or how the selected transport
protocol should behave during the lifetime of the Connection.

\subsubsection{Selection Properties}
Selection properties are the subset of Transport Properties which
used to select the protocol stack
to use for a given Connection when performing candidate gathering as described
in~\cref{sec:candidate-gathering}.

Typical Selection Properties are specified as
a \term{Preference}, with the possible values being \term{Require},
\term{Prefer}, \term{No Preference}, \term{Avoid} and \term{Prohibit}.
\term{Require} means that the Connection establishment will fail if the property cannot
be satisfied, while \term{Prefer} will allow the application to continue.
This distinction is mirrored in the difference between \term{Prohibit} and \term{Avoid}.
Not all Selection Properties are set as a preference,
instead specified for example as a boolean, an integer or an enum value.

An example of a Selection Property is Reliable Data Transfer.
This property is used to specify whether or not the application
should use a protocol which ensures that all data is received at 
a remote endpoint, in order, without loss or duplication. Setting
this to \term{Require} will force TAPS to use an underlying protocol
which guarantees this, such as TCP or QUIC. Setting this as
\term{Prohibit} will force TAPS to use a protocol which does not
guarantee this, such as UDP.

\subsubsection{Connection Properties}
Connection Properties are the subset of Transport Properties which
is used to control the behavior of the underlying protocol after
a Connection has been established.

An example of a Connection Property is Timeout for Aborting Connection~\cite[Sec. 8.1.3]{rfc9622}.
This property is used to specify the how long to wait before deciding that an
active Connection has failed when trying to reliably deliver data to the Remote Endpoint.

\subsection{Message Properties}
Message Properties are settings set on an
individual Message. These are used to set the behavior
of a single Message when transferring data, but do
not affect any other Messages transferred by the Connection.

An example of a Message Property is \mono{Safely Replayable}.
This property is used to let TAPS know that a Message is idempotent
and does not pose a risk if it is replayed to the endpoint.

\subsection{TAPS configurations explained}
As mentioned above, TAPS allows the application to control
both the behavior and selection of the underlying transport
protocol through the use of \term{Transport Properties}.

In addition to the basic Transport Properties such as reliability
or timeout, TAPS also exposes several more advanced
Transport Properties and these are explained below.

The \texttt{Multistream Connections in Group} Selection Property
allows the application to select whether the
application would prefer multiple Connections within a
Connection Group to be provided by streams of a single
underlying transport Connection where possible~\cite{rfc9621}.

Multistreaming in this context is the act of providing multiple
streams of data over a single underlying Connection. This
allows for more efficient use of the network
since the Connection will better saturate the available
bandwidth, avoiding long convergence times for
subsequent Connections.

The \mono{Full Checksum Coverage on Sending} and equivalent
Selection Property for receiving,
\mono{Full Checksum Coverage on Receiving}, specify
the application's need for protection against corruption for all
data transmitted or received on a given Connection. This is used
to specify how many bytes of the Message which needs to be
protected by a checksum. This checksum is used to protect the
data from corruption caused by lower-layer protocols. The
setting \mono{Full coverage} means that the entire Message is
protected, while \mono{0} specifies no checksum is required.
If any other value than \mono{Full coverage} is specified, the
Connection may still fall back to \mono{Full coverage}.

The \mono{Provisioning Domain Instance or Type} selection
property allows the application
to control path selection by selecting for specific Provisioning
Domain or categories of domains~\cite{rfc9621}. Provisioning
Domains define consistent sets of network properties that might
be more specific than network interfaces~\cite{rfc9621}.

The \texttt{Use Temporary Local Address} Selection Property
allows the application to specify
whether or not to use temporary addresses which are a feature in
IPv6, used for greater privacy~\cite{rfc8981, rfc9621}.

The \texttt{Notification of ICMP soft error message arrival}
Selection Property lets an application specify whether or 
not it should be notified when an ICMP
error, which does not force termination of the Connection,
arrives~\cite{rfc9621}.

In addition to specifying which underlying protocol should be
used through Selection Properties, the application using TAPS
can manage the Connections
themselves through Connection Properties. These can be specific
for a given transport protocol
or generic and be available for all transport protocols.

The \mono{Connection Group Transmission Scheduler} Connection 
property decides which scheduler is used within a given
Connection Group, to divide the available capacity between the
Connections.~\cite{rfc9621}.

The \mono{Isolate Session} Connection Property can be set in order to
have new Connections use as little cached information as
possible from previous Connections not in the same Connection
group~\cite{rfc9621}.

The final type of transport properties are Message Properties.
These are set by the application on a per-Message basis, and
can be used to control the behavior of individual Messages.
These are typically fairly self-explanatory, but do
include certain properties which can override other properties
set for the whole Connection.

For example, the \term{Ordered}
Message Property can be used to ensure in-order delivery
of a certain set of Messages, even if the Connection
does not require it. This naturally requires that the underlying
transport protocol supports switching between ordered and
unordered delivery of Messages.

\subsection{Candidate Gathering}\label{sec:candidate-gathering}
When initiating a Preconnection, a TAPS system will gather a
list of candidates for establishing a Connection. This process
is known as candidate gathering. The candidates are based on
the specified local and remote endpoints of the Preconnection,
as well as the protocols which are available to use. 

RFC~9623 conceptualizes these combinations as a tree,
with the first level of child nodes representing the
possible Local Endpoints, which are resolved to possible local IP
addresses when building the tree~\cite{rfc9623}.
The second level of the candidate tree represent the possible protocol
options which can be used. This includes not only the protocol
itself, but also any combination of options for this protocol which
influence Connection establishment.

The third and final level of child nodes represent any resolved
Remote Endpoints, which have been resolved into specific IP addresses.

While the primary use of Selection Properties is to restrict
which protocols are selected when performing candidate
gathering, they can also be used to affect
other parameters of the raced candidates.

For example the possible types of interfaces
in the first level can be restricted via Selection~Property
and the possible addresses resolved in the third level can
be restricted by requiring the use of temporary local
addresses, as these are only featured in IPv6.

While RFC~9623 uses a tree to conceptualize the different combinations
of candidates, it places no bearing on the actual implementation
in a real TAPS system. It does however recommend to branch in the
same order as the RFC, to place concepts more important to an
eventual Connection higher in the conceptual tree.

After all the candidates have been gathered, a TAPS implementation
has to sort them according to the requirements of the user. The
candidates are sorted from most to least preferred, with the
leaf nodes representing a single possible Connection attempt.

After all the candidates have been gathered and sorted, a single candidate
needs to be selected to be used for this Connection. This is done
through candidate racing.

\subsection{Candidate Racing}\label{candidateRacing}
When a tree or other representation of the candidates has
been constructed, a single candidate needs to be selected for use
by the final application. This is done through
candidate racing. Candidate racing is the process of establishing
a Connection using a single protocol over a single path with
a single endpoint as quickly as possible, optimizing for the
user requirements while minimizing the network load~\cite{rfc9623}.

This is achieved by trying to establish Connections with
the various candidates and finally selecting one of them.
The exact process of racing the candidates and selecting
a winner can vary by implementation. RFC~9623 lists three
ways of performing candidate racing:

\begin{enumerate}
    \item Simultaneous
    \item Staggered
    \item Failover
\end{enumerate}

Simultaneous racing is the process of trying to establish
Connections with all candidates at the same time. RFC~9623
discourages this method, as it can lead to an unnecessarily large
network load.

Staggered racing is the process of trying to establish Connections
with several candidates at once, but with a delay between the start
of each candidate. This delay is configurable, but should be based
on the expected handshake time~\cite{rfc9623}. 
Since the most desirable candidate will be raced first, it
can be chosen if it connects successfully, thereby limiting the
number
of needed attempts to find a Connection. When racing IP addresses,
TAPS implementations should follow the algorithm described
in RFC~8305~\cite{rfc8305}, also known
as Happy Eyeballs. The relevance
of this algorithm for TAPS is described in~\cref{happyEyeballs}.

Failover racing is the process of racing a single candidate
at a time, only attempting the next candidate if the previous
fails. This can be used when an application has a strong preference
of one branch over another.

Connection establishment has been completed when a single candidate,
has successfully established a Connection with a Remote Endpoint, or
when all candidates have failed. 

\section{Happy Eyeballs}\label{happyEyeballs}
Happy Eyeballs, which is described in RFC~8305 is an algorithm
for racing IPv4 and  IPv6 addresses on dual-stack hosts~\cite{rfc8305}. The candidate gathering
and candidate racing algorithm found in TAPS is based on Happy Eyeballs,
so they share a lot of central features.
Happy Eyeballs implements a staggered racing algorithm with a preference for
IPv6, but uses interleaved IPv6 and IPv4 addresses in order to maximize
IPv6 usage but minimize potential latency caused if IPv6 is broken.

The first version of Happy Eyeballs was published in 2012 and describes an algorithm
for racing an IPv4 and IPv6 connection to a server, giving a preferential
head start to the IPv6 connection~\cite{rfc6555}.
Happy Eyeballs version 2 improves
upon problems within Happy Eyeballs version 1 by also racing the
IPv6 and IPv4 DNS queries, avoiding DNS slow family blocking~\cite{rfc8305}.

This concept of racing is a central concept of TAPS, where it is
taken one step further to also race all combinations of local interface,
remote endpoint and underlying transport protocol which fit with the
specifications of the user. RFC~8305 lists four steps for protocol racing,
which conceptually divide into candidate Gathering and candidate Racing 
in the TAPS system.

\begin{enumerate}
  \item Initiation of asynchronous DNS queries
  \item Sorting of resolved destination addresses
  \item Initiation of asynchronous connection attempts
  \item Establishment of one connection, which cancels all other attempts
\end{enumerate}

Step one and two map into the TAPS concept of candidate gathering
while step two and three map into the TAPS concept of candidate racing.

Since waiting for each of the four steps to fully complete before moving
on to the next one would lead to unecessary delay Happy Eyeballs v2 recommends
that connection attempts are started eiteher immediately on
the first response if it is IPv6, or after a \qty{50}{\milli\second} resolution
delay if the first received resolution if IPv4~\cite{rfc8305}.
This helps mitigate potential slow DNS resolutions, especially
if the IPv6 resolution is significantly slower than the IPv4
resolution and the application has a hard requirement to
try IPv6 first.

If no IPv6 addresses are received by the end of the resolution
delay then the application should generate a list of currently
available endpoints and sort it as normal, starting to racing
by initiating a connection attempt to the most desirable
endpoint. If records are received after racing has
started then they should be integrated into the queue
of endpoints waiting to be attempted.

\section{ALPN}
Application layer protocol negotiation is a feature in
TLS where the application layer protocol which will be
used is negotiated between the server and the client on
the transport layer.
ALPN is featured
in in TLS, and a QUIC mandates that endpoints
use some mechanism to agree on application protocol~\cite[Sec. 8.1]{rfc9000}
ALPN allows the client and server to negotiate a common protocol
without adding network round trips. The client puts 
a list of supported protocols in its client hello message
and the server chooses a single protocol and puts it in
its server hello message. If the server does not support any
protocols that the client supports, then it responds with an
error notifying the client of this.

\section{SCTP}
QUIC is not the first transport layer protocol to feature
multistreaming. SCTP was introduced in 2000 and
is both message based and features multi-streaming like QUIC~\cite{leungMultihomedCommunicationSCTP2013}.
However, since SCTP is a transport layer protocol traditionally
built on top of IP, its adoption has struggled likely due
to network ossification.

SCTP over UDP has
struggled with adoption, showing that not only does network ossification
prevent the adoption of new protocols, but also that
the backing of major actors, such as Google backing QUIC,
is required for major innovations on the transport layer.

\section{Asynchronous I/O and event loops} \label{sec:async-background}
The naive way to write a networking application is to treat
any interaction with the network like a normal CPU-bound task,
waiting for I/O bound operations to complete before progressing.
This is a simple, but very inefficient model and the CPU will
be spending much of its time waiting for I/O.

There are two primary ways of mitigating this. The first is multithreading,
where the application creates a thread per connection and threads yield
the processor while waiting for blocking I/O. The second is an event
loop\footnote{Exotic alternatives to event loops for achieving asynchronous
I/O do exist, for example the POSIX AIO interface, but these are rarely
used in practice.}. An event loop is a single-threaded program that executes tasks in order.
Functions that initiate asynchronous work return immediately and the
application is later notified of the result, usually through a callback.
Tasks are added to the queue either by the application itself or in
reaction to some asynchronous result being ready, such as incoming
data.

An illustration of an event loop with a simple queue can be seen in
\cref{fig:event-loop-queue}. Here, epoll is used to monitor non-blocking
sockets and the associated receive calls are added to the queue.
This allows the single-threaded event loop to be handle
several different connections at once.

The threaded model allows code to be written sequentially within each thread,
closer to the mental model of a normal CPU-bound program, but requires synchronisation
of shared data structures between threads. An event-loop model avoids the
synchronisation issue by running in a single thread, but spreads control
flow across callbacks.

The two models also differ in how they scale. Multithreading makes use of
multiple CPU cores easily, but adds a non-negligible overhead if
creating a large number of threads.
Event loops scale well on a single core are more difficult to use with
multiple cores. They are also more difficult to
integrate into external applications.
The two models are not entirely mutually exclusive
as event loop libraries can use threads internally to achieve an asynchronous flow
for synchronous operations, but from the view of an application
interacting with the network they are fairly distinct.

Although real implementations naturally feature more advanced features than
illustrated in \cref{fig:event-loop-queue}, the overall model of the asynchronous
event loop model is very common, and is used for example in Node.js~\cite{nodejsEventLoop}
and Python's asyncio~\cite{pythonAsyncio}.

\section{Libuv}
Libuv~\cite{Libuv2025} is a multi-platform event loop library focused on asynchronous I/O.
It is primarily developed for Node.js, but is used in several other large projects as well.
Libuv is centered the \mono{uv_loop_t} structure which represents a single event loop.
Applications can register work to a loop through the use of handles and requsts~\cite{libuv_design}.
Handles are long-lived objects for certain types of work, while requests are short-lived operations.
When any intial handles and requests have been registered the application must invoke \mono{uv_run()}. This
will take over the invoking thread and by default run until all registered handles
have been closed.

The most
important libuv handles for CTaps are \mono{uv_udp_t}, \mono{uv_tcp_t}, \mono{uv_poll_t} and
\mono{uv_timer_t}. 
Since libuv is cross-platform, the UDP and TCP handles
are abstractions above OS-specific sockets. On linux, the underlying event
queue object is epoll. Epoll is a structure which allows an application
to be notified by the Linux kernel when data is available on a socket.
This can be used to interact with the network, but also to monitor any other
file descriptors which are compatible with epoll.

While the libuv \mono{uv_udp_t}
and \mono{uv_tcp_t} handles mandate the type of the underlying file descriptor,
the generic \mono{uv_poll_t} can be used to monitor a generic event queue object.
This allows an application to use libuv for any file descriptor compatible
with an OS-specific event queue object. Finally, the \mono{uv_timer_t} handle
can be used to perform work at given intervals.

Handles are initiated and registered to a given event loop through
\mono{uv_<handle name>_init}. Requests can then be added as discrete
pieces of work to the given handle. For example \mono{uv_write()} is
used to add a write request to a \mono{uv_tcp_t} handle. When registering
a request with the event loop the application must also register an
associated callback. This callback will be invoked to notify the application
that execution has completed.
When
the application is finished it must close the created handles for
the event loop to stop execution.

\section{Picoquic}\label{sec:picoquic}
Picoquic~\cite{christianhuitemaPicoquic} is a C library which aims to be a
minimalist implementation of the QUIC standard. It is a cross-platform
library compatible with Windows, Linux, MacOs and BSD.

% Talk about how it functions?
For a standalone clients and servers Picoquic features a built-in
event loop through \mono{picoquic_packet_loop()}. This takes
control over the invoking thread and allows the application to
control the behavior of the loop through returning values
from a callback. When using this packet loop Picoquic
manages its own sockets and interacts with the network
independently.

In addition to built in event loop, Picoquic also features
an polling API which integrates with existing event loops.
For this mode the application is required to manage
the interaction with the network and call into
Picoquic at given intervals. These intervals
are specified by Picoquic and must be fetched through
\mono{picoquic_get_next_wake_delay()}. A timer
can then be set to wake at the specified interval
and perform any work necessary by Picoquic.

When this timer fires, the application has to query
Picoquic for any new packets to send to the peer.
This is done through
\mono{picoquic_prepare_next_packet()}
which outputs one QUIC packet at a time along with
the expected from address, expected to address
and the expected interface index. The application
is then responsible for actually transferring
the data over a UDP socket.

Since the application is responsible for interacting
with the network it is also responsible for passing any
received QUIC packets to Picoquic. In order to start
receving or sending QUIC data the application must
first create a \mono{picoquic_quic_t} object through
\mono{picoquic_create()}. This is an object which is
responsible for handling ovarching QUIC state such as
TLS parameters as well as receving incoming QUIC data.
This means that it is also responsible for passing
the incoming data to the correct QUIC connection, or
creating a new Connection and passing it to the application.
Passing incoming QUIC packets to Picoquic is done through
\mono{picoquic_incoming_packet()}.

To handle the mandatory TLS for QUIC
Picoquic integrates with Picotls~\cite{picotls}.
Picoquic can therefore handle all necessary
TLS setup, only requiring that the application
provides the necessary parameters, such as
ALPN values and the various key files.

\section{Glib}
GLib~\cite{GLibGitLab} is a general purpose utility library which
is used at the core of several other larger libraries, such as GTK and GNOME.

Among other features, GLib provides implementations of several key data
structures not provided by the C standard library, such as dynamic arrays,
hash maps, trees and linked lists.
