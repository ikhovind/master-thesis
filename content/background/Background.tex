\chapter{Background}\label{chap:background}
This chapter will lay out background information which forms the
basis for both the current design and the limitations of CTaps.
It will describe the low level networking concepts,
transport layer protocols and the current barriers
for innovation on the transport layer.
Extra care is taken in describing QUIC, as exposing
QUIC features forms the basis for the design
of CTaps and the experiments in~\cref{chap:eval}.


\section{Network Interfaces and Paths}
Modern devices are often connected to multiple networks simultaneously.
For example, a laptop may be connected to both a Wi-Fi network and a wired Ethernet at the same time.
Each separate network is interacted with through a separate network interface
and each network interface is associated with distinct source IP addresses for outgoing data.
If a device can reach the same endpoint through several network interfaces,
then it can utilize multiple interfaces for a single network connection as long as this is supported
by the underlying transport protocol.

The combination of source IP, source port, destination IP and destination port
is referred to as a \term{network path} following the terminology used in RFC~9000~\cite{rfc9000}.
Since each network interface is associated with distinct IP addresses,
each interface that can reach the same destination will represent a distinct network path.
Utilizing multiple network interfaces for a single connection can improve
both stability and the throughput of the connection.
Support for multiple network paths is a notable feature of QUIC
and is described in detail in~\cref{sec:quic}.

\section{Network Sockets}
When it comes to network programming,
the network socket in its most basic definition is an abstraction
for a network connection through which an application can send
and receive data~\cite[p. 8]{donahooTCPIPSockets2009}. 
First mentioned in 1970 in relation to ARPANET, the
simplicity of sockets and their ubiquity in all UNIX distributions
since version 4.2 of the Berkeley Software Distribution have
led to them becoming the de facto standard for network
programming~\cites{rfc33}[p.~502]{tanenbaumComputerNetworks2011}.

Modern sockets are standardized across UNIX-like systems through
the POSIX standard. Implementations of the C POSIX library define
the socket API primarily through the \texttt{sys/socket.h} header file~\cite{posix:2024:socket}.

When creating a socket on an UNIX-like system, the application
invokes the \texttt{socket} system call. This system call
\enquote{creates an endpoint for communication and returns a file 
descriptor that refers to that endpoint}~\cite{Socket2LinuxManual}.
This system call takes three parameters, the domain, the type
and the protocol which will be used for the socket.

The domain specifies the type of
addressing which will be used, for example \texttt{AF\_UNIX}
for local sockets, \texttt{AF\_INET} for IPv4 and \texttt{AF\_INET6}
for IPv6. The type specifies the protocol which will be used over the socket, for
example \texttt{SOCK\_DGRAM} for datagrams and \texttt{SOCK\_STREAM}
for a reliable stream of bytes. The type usually corresponds to a
single transport layer protocol, usually UDP and TCP.

The protocol parameter is used if there are multiple protocols
which support the requirements given by the domain and type.

\section{Barriers to Protocol Adoption}\label{sec:transport-protocol-barriers}
The fact that an application has to specify both the domain
and the type of the socket means that the
underlying addressing and the transport layer protocol are bound at compile time.
Updating
the underlying transport protocol will therefore require at least recompilation and 
likely a refactoring of the application. 
This naturally makes adopting new transport protocols difficult, even
if a new protocol is better suited to the application's requirements.

Another challenge with the adoption of new transport layer protocols
is network middleboxes. Network Middleboxes have been
optimized to support TCP and UDP and may sometimes
drop packets if the protocol field in the IP header
does not match TCP or UDP.

The resulting difficulty in innovating on the transport layer
is known as network ossification. In an attempt to combat
this ossification, transport layer protocols are often built
using UDP as a substrate, allowing them to use the same IP
protocol number as UDP. Examples of this include QUIC \cite{rfc9000} and
SCTP over UDP \cite{rfc6951}.
While this does present some overhead,
it is fairly minimal with the UDP header only being \qty{8}{\Bytes}.
Since UDP presents no guarantees of reliability or ordered
delivery, protocols can choose to develop only the features they need.

\section{TCP}
TCP is by far the most common transport layer protocol used on the
internet today. It is a connection-oriented protocol and provides
a reliable, in-order transfer of a stream of bytes to the application layer.
It has also become common to use TCP for message-based applications,
using a framing protocol on top of TCP to divide the TCP byte stream into
discrete messages. A description of TCP was first published as an RFC~in 1974~\cite{rfc675}
and it has since grown to become
ubiquitous across the internet.

In TCP, connections are uniquely identified by a four-tuple consisting
of the source IP address, the source port, the destination IP address
and the destination port.

\section{UDP}
UDP has, along with TCP, grown to become one of the two dominant
transport layer protocols on the internet. It is a connectionless
protocol and provides an unreliable, unordered transfer of datagrams
to the application layer. It is often used for applications which
prioritize low latency over reliability, such as real-time applications.

UDP is designed to be a very lightweight protocol with as little
overhead as possible. It uses a very simple 8-byte header, containing 
only the source port, destination port, length and checksum.

\section{TLS}
Transport layer security (TLS) is a cryptographic protocol which provides secure communication
over a computer network. It is commonly used on top of TCP and is
most visible as the basis for HTTPS, the secure version of HTTP.

TLS features its own handshake protocol when establishing a connection,
in order to exchange the cryptographic parameters needed for the connection.
This means that connections using TLS have to spend an additional round trip time (RTT)
during connection establishment, compared to a non-secure TCP connection.

TLS 1.3 also supports 0-RTT handshakes
which allow a client to send data during the initial handshake when resuming
a previous connection to a server~\cite{rfc8446}. This can be a source
of replay attacks if the message being sent is not idempotent.

As of 2026 TLS adoption has been very widespread,
with \qty{95}{\percent} of all connections in Chrome on Windows using an encrypted
connection~\cite{google2026transparency}.

\section{QUIC}\label{sec:quic}
QUIC is a protocol which was originally developed by Google and then formalized
through RFC~9000 in 2021~\cite{rfc9000}. It is developed in user-space over UDP
and is
intended to be a replacement for TCP, it has achieved a fairly high adoption. 
By 2020, over \qty{75}{\percent} of traffic to Meta servers was already using
QUIC~\cite{joras2020quic}.

QUIC is a reliable, connection-oriented protocol and provides several
features which either are not available or do not have a wide adoption
in TCP. Among these are 0-RTT connection establishment, multistreaming
and support for multiple network paths. TLS is a mandatory part of QUIC 
and most of the QUIC header is also encrypted, unlike TLS over TCP.
Encrypting the header hides as much as the wire image as possible
from middleboxes, combating future network ossification~\cite{rfc9170}.

While QUIC does present an overarching connection, the
abstraction for each actual data transfer is called a stream.
A stream is a unidirectional or bidirectional ordered
continuous transfer of data, much like what an application sees when using
TCP. Streams can be created and closed by either endpoint,
and multiple streams can exist over a single QUIC connection.
QUIC mandates support for in-order delivery within a single
stream, but not between streams. This avoids the issue
of head of line blocking, where the
transfer of a small object potentially has to wait for the
transfer of a previous large object to complete before being
handed to the application.

For QUIC, a single connection can exist over several paths and
congestion control is done on a per-path basis. Multiple streams
can be transmitted over a single connection and data from
multiple stream frames can be transmitted in a single packet.
If a packet is lost, only the streams with data contained
in that packet will be affected.

\subsubsection{Congestion Control}
Like TCP, QUIC also implements congestion control, but there are
some differences between the algorithms. A key concept
to QUIC's congestion control is the \term{packet number space} which
is defined as \enquote{the context in which a packet can be processed
and acknowledged}~\cite{rfc9000}. 

The packet number space reflects the level of encryption that has been reached and there 
are three separate packet number spaces. \term{Initial space},
for the initial handshake packets, \term{handshake space}, for packets
exchanged during the cryptographic handshake, and \term{application
data space}, for packets exchanged after the cryptographic handshake
has been completed.

Separating these spaces ensures that acknowledgments of packets at one level do not cause
spurious retransmission at another encryption level.
Although the packet number spaces are handled
separately for acknowledgments, they are
unified when it comes to congestion control and round trip
time measurements~\cite[Sec.~4.1]{rfc9002}.

Another difference between how QUIC and TCP handle acknowledgments
is that QUIC separates the transmission order from the delivery order.
This is done by using stream offset as delivery order, and packet number as
transmission order. This removes ambiguity about whether the
original or the retransmitted packet is acknowledged in the case
of retransmissions, making RTT measurements in these cases more accurate~\cite[Sec.~4.2]{rfc9002}.

\subsection{Transport parameters}
Transport parameters in QUIC are settings which are transferred to the
peer during the QUIC handshake. These can be used to for example
limit the initial amount of streams the peer can open or the amount
of data it can send.

\subsection{Frames}
In QUIC, a frame is defined as \enquote{a unit of structured protocol information}~\cite{rfc9000}.
Different types of frames are used to carry different types of information
and data. Examples are \mono{STREAM} frames which are used to carry
data over a stream or \mono{NEW_CONNECTION_ID} which are used
to share new valid connection IDs with the peer.

\subsection{Connection ID}\label{sec:quic-conn-id}
In order to uniquely identify a connection QUIC uses connection IDs
instead of a four-tuple like TCP. Each endpoint generates a list of
valid IDs which it shares with the peer. The peer can then use one
of these IDs as a destination when transferring data~\cite{rfc9000}.

Since Connection IDs are needed for routing packets they are
carried in the clear text part of both the short and the long QUIC header.
The long header carries a source connection ID in addition to the mandatory
destination ID.

In order to prevent observers from being able to link a connection
through its connection IDs, QUIC connections rotate the IDs used.
A connection can switch the destination connection ID it is using
to one of the approved IDs previously shared by the peer at any time.

Connection IDs make addressing in QUIC independent of addressing in lower layer
protocols~\cite{rfc9000} which allows QUIC to survive
changes in the underlying four-tuple and therefore enables
connection migration.

\subsection{Connection Migration}\label{sec:connection-migration}
QUIC connections can survive changes to the underlying four-tuple
because they are identified by connection IDs. This enables connection
migration, where an established connection is moved to a new network path,
for example switching between Wi-Fi and cellular, without requiring a new handshake.
In the current version of QUIC, only clients are able to migrate~\cite[Sec.~1]{rfc9000}.
Servers can state a preferred local IP address and request that the client
migrate by using the \mono{preferred_address} transport parameter,
but this can only be done during the initial TLS handshake~\cite[Sec.~9.6.1]{rfc9000}.

If the server does not support migration, such as when it is behind a simple load balancer which multiplexes
by source IP instead of the QUIC connection ID, then the server can mandate
that the client does not change local address by setting the \mono{disable_active_migration}
transport parameter~\cite[Sec.~5.2.3]{rfc9000}.

In order to initiate path migration, an endpoint will send a non-probing packet 
over a new network path. Non-probing packets are any packets which contain any
frame that is not \mono{PATH_CHALLENGE}, \mono{PATH_RESPONSE}, \mono{NEW_CONNECTION_ID}, or
\mono{PADDING}. When an endpoint receives non-probing packets from a new address, it can assume 
that the peer has migrated to that address. Both peers are required to use a new connection
IDs when performing migration, in order to ensure that the connection cannot
be used to link packets from the same connection over different network paths
\cite[Sec.~9.5]{rfc9000}.

If the receiving endpoint permits connection migration,
then any new packets must be sent to the new address
and path validation must be initiated.
The endpoint initiating connection migration may defer path validation until after
the receiving endpoint has sent non-probing data to the initiating endpoint's new
address~\cite[Sec.~9.2]{rfc9000}.
Path validation is used to verify
that a peer is reachable on a given path and it is done by using \mono{PATH_CHALLENGE} and \mono{PATH_RESPONSE} 
frames. The challenge frame contains \qty{8}{\Bytes} of arbitrary data,
which the peer must echo back in the response frame. This ensures that the source address
is not spoofed and therefore avoids potential amplification attacks.

If path validation fails, which happens if no matching response
is received within the Probe Timeout (PTO)~\cite[Sec. 6.2]{rfc9002},
the endpoints can continue communicating using other known paths.
If no viable paths remain, then the connection fails. This can be signaled to the peer 
using a \mono{NO_VIABLE_PATH} connection error, although the only
situation where this can be communicated is
if a path exists, but it does not support the required
maximum transmission unit (MTU)~\cite[Sec.~8.2.4]{rfc9000}.

A figure comparing QUIC connection migration with TCP can be seen in~\cref{fig:connectionBeforeAfterMigration}.
Here, a local address becomes unavailable and the TCP connection is therefore broken
since the underlying four-tuple has changed. The
QUIC connection is able to survive since the connection
ID is recognized by the peer.
When performing the migration, a new connection ID is picked
from the pool of valid IDs generated by the peer to
avoid linkability.

In addition to supporting multiple paths through connection migration,
support for using multiple paths simultaneously through a single QUIC 
connection is currently actively being
standardized through the QUIC multipath extension draft~\cite{ietf-quic-multipath-21}.
Multipath extension builds on the mechanisms of connection migration,
but introduces a new path ID in order to manage multiple paths.

The QUIC multipath extension draft does not specify how
alternative paths are discovered, and actual setup and
teardown of paths is handled by the application.

\subsection{Session Resumption}
Like TLS, QUIC supports session resumption and 0-rtt
initial data. This requires that the server and the client
have already exchanged cryptographic keys, that the
client remembers critical parameters such as the selected
application-layer protocol negotiation (ALPN)
value and that the client provides the server with a
session ticket for the server to recover the same
information~\cite[Sec.~4.6]{rfc9001}.

\section{Transport Services}
At its core, a protocol is an abstraction which provides a set of
services to an higher-layer protocol or application. In the context of
networking, these services encompass transporting data from one endpoint
to another, with the protocol itself deciding the which services are
provided. For example, TCP provides a guarantee of reliable
delivery while UDP does not.

As an example, if an application requires that the
transferred data is received as an in-order stream, then
any transport layer protocol offering this as a service to the application layer
could function as a transport protocol, and swapping the
protocols should in theory be simple. This is of course
not currently the case due to issues described in \cref{sec:motivation}.

Exactly which services are provided by which transport
protocols has been mapped in RFC~8095 by the Transport Services (TAPS) working group~\cite{rfc8095}.

\section{TAPS}
The TAPS WG was an IETF working group which aimed to solve the strong
coupling between application
level code and the underlying transport protocols through
focusing on the services provided by each transport layer protocol.
This is intended to allow faster adaptation of new protocols and to
abstract away network-layer details from the application layer.

The main RFCs produces by the TAPS WG are RFC~9621~\cite{rfc9621},
RFC~9622~\cite{rfc9622} and RFC~9623~\cite{rfc9623}. Together, these
describe a TAPS System, which is an implementation of of the TAPS
API and the TAPS Implementation. The TAPS API is the API of a
TAPS system which offers a generic way to interact with the network
independent of the underlying protocol. The TAPS Implementation 
is the internal workings of a TAPS System. This includes candidate
gathering, candidate racing, as well as the interaction
to the supported transport protocols.

Despite being implemented in user-space, a TAPS System conceptually resides
in the transport layer. The TAPS API provides convenience to the application
layer by expanding the responsibility of the transport layer.
This minimizes how much the application layer has to adapt to
the underlying transport layer, and reduces the coupling between the two layers.

\subsection{TAPS API}
RFC~9622 describes the TAPS API. This is an abstract API designed to be
able to interact with the network while enabling dynamic selection of
transport protocols and network paths~\cite{rfc9622}.
The TAPS API is object oriented and is primarily interacted
with through the \term{Preconnection}, \term{Connection},
\term{Message}, \term{Listener}, \term{Local Endpoint}, \term{Remote Endpoint},
\term{Transport Properties} and \term{Security Parameters} objects.

A Connection represents a \enquote{shared state of two or more endpoints that persists
across Messages that are transmitted between these endpoints}~\cite{rfc8303}.

When working with Connection objects, TAPS provides five high
level groups of actions that an application can perform:
\begin{itemize}
    \item Preestablishment: Pass properties to specify
        requirements and preferences for establishing a Connection.
    \item Establishment: Prepare a Connection object for data
        transfer.
    \item Data transfer: Sending and receiving Messages
        over the Connection.
    \item Event handling: Notifications that an application
        can receive from a Connection, and the handling of them.
    \item Termination: How transmission is stopped and
        a Connection is torn down.
\end{itemize}

In addition, the lifeline of a Connection is divided into three distinct
phases, preestablishment, the established state, and
termination of a Connection~\cite{rfc9621}.
Each of these phases are associated
with one or more of the groups of actions that an application can perform,
as described above. 

After the desired Transport Properties and Security Parameters have
been set during the preestablishment phase, the Connection can be established. 
After a Connection has been established it is in the established phase and can be used to
transfer data. 

A Connection can be established in one of three ways, \mono{initiate()}, \mono{listen()} or 
\mono{rendezvous()}~\cite{rfc9621}.

Invoking \mono{initiate()} establishes a Connection to a remote endpoint which
is assumed to be listening for incoming Connections. How this Connection
is established internally is dependent on the underlying transport
protocol. The Created Connection object is passed to the application through a callback.
Exactly how the underlying transport protocol is chosen is described in detail
in \cref{sec:candidate-gathering} and \cref{candidateRacing}.

Invoking \mono{listen()} is used to passively open a Connection. Invoking \mono{listen()}
creates a Listener object and wait incoming Connections.
When a client connects to this Listener, a \term{Received}
event occurs. This event contains a Connection object for the 
established Connection.

Invoking \mono{rendezvous()} is used to establish a peer-to-peer
Connection between two endpoints simultaneously.

After a Connection has been established, it can either
be closed or aborted, with closing being associated with graceful teardown
and aborting being abnormal termination associated with errors.

\subsection{Cloning}
One or more TAPS Connections can be organized into a \term{Connection Group}.
The Connection Group is not an object exposed through the TAPS API, but
is maintained internally in a TAPS Implementation.
A Connection Group is used to group related Connections together, which have been created
by cloning. Connections in the same Connection Group can be
multiplexed together if this is supported by the underlying protocol.

\subsection{Framer}
In TAPS a \term{Framer} is an object responsible for bridging the TAPS concept
of a Message, with the underlying protocol concept of either a data stream
or a datagram. In its simplest form, a Framer is responsible for encoding
or decoding outgoing and incoming data, as well as interacting with setup and
teardown of a Connection if necessary~\cite[Sec.~9.1.2]{rfc9622}. Because Framers may need to send
data upon the creation of a Connection, they must be added to a Preconnection.
The application can define custom framers and pass them to a Preconnection
by implementing an interface provided by the TAPS System.

Because they may need to interact with the network, 
framers are also asynchronous and need
to notify their associated Connection when the processing of a given Message
has completed.

The concept of a framing protocol also exists outside of TAPS since most
application-level protocols are both message-based and run over TCP. Several
ways of converting TCP's byte-stream into usable messages has been implemented.
These largely fall into two categories, type-length-value (TLV) structures where
the length of the following data is described in a header-structure, and
delimiter-separated formats, where the beginning and end of a piece of
data is separated by delimiters.~\cite[Sec.~6.1]{rfc9623}
Examples of the two categories are HTTP/1.1 for a delimiter-separated format
and DHCP for TLV~\cite{rfc2131}.

Framers become useful in TAPS because they can be used to ensure Messages
are received in a consistent format no matter what underlying transport
protocol is selected. Framers can be used both when sending and receiving
and can both split and join Messages.

Conceptually, a framer resides on a separate layer between a Connection and the
underlying Transport Protocol Stack as seen in~\cref{fig:tpsStack}.

\subsection{Transport Properties}
Transport Properties are properties to influence the underlying transport protocol.
They can be divided into two categories, \term{Selection Properties}
which influence which underlying transport protocol is selected and
\term{Connection Properties} which influence the behavior of the underlying
protocol.


\subsubsection{Selection Properties}
Selection properties are the subset of Transport Properties which are
primarily used to filter and sort candidate protocols
when performing candidate gathering as described
in~\cref{sec:candidate-gathering}. Certain Selection Properties
can also be used to influence the selected Endpoints.

Typical Selection Properties are specified as
a \term{Preference}, with the possible values being \term{Require},
\term{Prefer}, \term{No Preference}, \term{Avoid} and \term{Prohibit}.
Require means that the Connection establishment will fail if the property cannot
be satisfied, while Prefer means that the TAPS Implementation will prioritize
protocols which can satisfy the given property when performing candidate racing.
This distinction is mirrored in the difference between Prohibit and Avoid.
Not all Selection Properties are a Preference,
and they can also be specified as a boolean, an integer, an enum or other types.

An example of a Selection Property is \mono{reliability}.
This property is used to specify whether or not the application
should use a protocol which ensures that all data is received in order,
without loss or duplication~\cite[Sec.~6.2.1]{rfc9621}. Setting \mono{reliability}
to Require will force TAPS to use an underlying protocol
which handles loss such as TCP or QUIC. Setting this as
Prohibit will force TAPS to use an unreliable protocol
such as UDP.

\subsubsection{Connection Properties}
Connection Properties are the subset of Transport Properties which
are used to control the behavior of the underlying protocol after
a Connection has been established.

An example of a Connection Property is \mono{connTimeout}.
This property is used to specify the how long to wait before deciding that an
active Connection has failed when trying to reliably deliver data to the
Remote Endpoint~\cite[Sec.~8.1.3]{rfc9622}

\subsection{Message Properties}
Message Properties are settings set on an
individual Message. These are used to set the behavior
of a single Message when transferring data, but do
not affect any other Messages transferred by the Connection.

In certain cases these can override the Transport Properties
of the Connection, for example by setting \mono{msgOrdered}
to indicate that this individual message does not have to be
ordered~\cite[Sec.~9.1.3.4]{rfc9622}.

\subsection{TAPS Implementation}
While the TAPS RFCs does not prescribe implementation specific details,
RFC~9623 serves as a guide in how to best develop a TAPS System.
This includes how to present actions which are present in the TAPS API
such as data transfer as well as provided services which are not
directly exposed such as candidate gathering or candidate racing.

\subsection{Candidate Gathering}\label{sec:candidate-gathering}
When the application invokes \mono{initiate()} on a Preconnection,
a TAPS system will gather a list of candidates for establishing a
Connection. This process is known as candidate gathering. The candidates are based on
the specified Local and Remote Endpoints of the Preconnection,
as well as the given Selection Properties which are available to use. 

RFC~9623 conceptualizes these combinations as a tree,
with the first level of child nodes representing the
possible Local Endpoints, which are resolved to local socket addresses
(each a specific IP address and port) when building the tree~\cite{rfc9623}.
The second level of the candidate tree represents the possible protocol
options which can be used. This includes not only the protocol
itself, but also any combination of options for this protocol which
influence Connection establishment.

The selected local interface can be restricted through 
either using specific local endpoints or through the
\mono{interface} Selection Property. The viable
remote socket addresses can be influenced by for
example the \mono{useTemporaryLocalAddress} Selection Property,
as this is only supported in IPv6 \cite[Sec.~6.2.13]{rfc9622}.

RFC~9623 does not mandate that implementations follow the same tree
structure when performing candidate gathering, but it does recommend to branch in the
same order as the RFC, in order to branch by the most important concepts
first~\cite[Sec.~4.1.2]{rfc9623}.

After branching has been completed, incomplete paths and leaf nodes
which are not compatible with the specified Selection Properties can
be pruned and sorted. After pruning, each leaf node represents a single possible
Connection attempt, containing a combination of local socket address, 
protocol options and remote socket address. Each leaf node is referred to
as a candidate and are sorted according to the
preferences of the application before candidate racing is initiated.

\subsection{Candidate Racing}\label{candidateRacing}
Candidate racing is the process of dynamically attempting
to establish Connections over multiple candidates, in order
to find out which one is best suited to the network conditions
and the requirements of the application.

There are three primary ways of performing candidate
racing as listed in RFC~9623:

\begin{enumerate}
    \item Simultaneous
    \item Staggered
    \item Failover
\end{enumerate}

Simultaneous racing is the process of trying to establish
Connections with all candidates at the same time. RFC~9623
discourages this method, as it can lead to an unnecessarily large
network load.

Staggered racing is the process of trying to establish Connections
with several candidates at once, but with a delay between each attempt.
This delay is configurable, but should be based
on the expected handshake time~\cite[Sec.~4.3.2]{rfc9623}. 
The most desirable candidates are raced first, and all subsequent
attempts are canceled after the first success.
RFC~9623 recommends following the algorithm described in
RFC~8305~\cite{rfc8305}, known as Happy Eyeballs, when performing staggered racing of IP addresses.
Happy Eyeballs is a staggered racing algorithm designed to maximize IPv6 usage by
racing IPv4 and IPv6 addresses on dual-stack hosts.

RFC~8305 lists four steps for protocol racing,
which conceptually divide into candidate gathering and candidate racing 
in a TAPS Implementation: 

\begin{enumerate}
  \item Initiation of asynchronous DNS queries
  \item Sorting of resolved destination addresses
  \item Initiation of asynchronous connection attempts
  \item Establishment of one connection, which cancels all other attempts
\end{enumerate}

Step one and two map into the TAPS concept of candidate gathering
while step three and four map into the TAPS concept of candidate racing.

Failover racing is the process of racing a single candidate
at a time, only attempting the next candidate if the previous
fails. This can be used when an application has a strong preference
of one branch over another.

Connection establishment has been completed when a single candidate
has successfully established a Connection, or
when all candidates have failed. 

\section{ALPN}
ALPN is a feature in TLS \cite{rfc7301} where the application layer
protocol is negotiated between on
the transport layer. TLS is a mandatory part of QUIC and QUIC mandates that endpoints
use  agree on application protocol through a 
mechanism such as ALPN~\cite[Sec. 7]{rfc9000}.

ALPN allows the client and server to negotiate a common protocol
without adding network round trips. The client puts 
a list of supported protocols in its \mono{clientHello} message
and the server chooses a single protocol and puts it in
its \mono{serverHello} message. If the server does not support any
protocols that the client supports, then it responds with an
error notifying the client of this.

\section{SCTP}
The first transport layer protocol to support multistreaming was SCTP~\cite{rfc9260} which
was first introduced in 2000~\cite{rfc2960}. SCTP has struggled
with adoption for several reasons.
Firstly, since it uses a separate protocol number from TCP and UDP
it is vulnerable to middlebox interference. This is clearly not the
only reason, as SCTP over UDP has also failed to achieve
widespread use~\cite{rfc6951}.

Another reason for the lack of adoption is that the standard
socket API has been ossified itself \cite{sctpstruggles}.
The ubiquity of TCP and UDP means that the socket API
is tuned to these two transports
and does not present the advantages of a new protocol
such as SCTP to applications. This means that the socket
API had to be extended in order to support SCTP.

\section{Asynchronous I/O and event loops} \label{sec:async-background}
The naive way to write a networking application is to treat
any interaction with the network like a normal CPU-bound task,
waiting for I/O bound operations to complete before progressing.
This is a simple, but very inefficient model and the CPU will
be spending much of its time waiting for I/O.

There are two primary ways of mitigating this. The first is multithreading,
where the application creates a thread per connection and threads yield
the processor while waiting for blocking I/O. The second is an event
loop. An event loop is a single-threaded
program that executes tasks in order from a queue.
Functions that initiate asynchronous work return immediately and the
application is later notified of the result, usually through a callback.
Tasks are added to the queue either by the application itself or in
reaction to some asynchronous result being ready, such as incoming
data.

An illustration of an event loop with a simple queue can be seen in
\cref{fig:event-loop-queue}. Here, epoll is used to monitor non-blocking
sockets and the associated receive calls are added to the queue.
This allows the single-threaded event loop to be handle
several different connections at once.

The threaded model allows code to be written sequentially within each thread,
closer to the mental model of a normal CPU-bound program, but it requires synchronization
of shared data structures between threads. An event-loop model avoids the
synchronization issue by running in a single thread, but spreads control
flow across callbacks.

The two models also differ in how they scale. Multithreading makes use of
multiple CPU cores easily, but adds a non-negligible overhead if
creating a large number of threads.
Event loops avoid the overhead of multiple threads and scale well, but they are more difficult to use with
multiple cores. They are also more difficult to
integrate into external applications.
The two models are not entirely mutually exclusive
as event loop libraries can use threads internally to achieve an asynchronous flow
for synchronous operations, but from the view of an application
interacting with the network, they are fairly distinct.

Although real implementations naturally feature more advanced features than
illustrated in \cref{fig:event-loop-queue}, the overall model of the asynchronous
event loop model is very common, and is used for example in Node.js~\cite{nodejsEventLoop}
and Python's asyncio~\cite{pythonAsyncio}.

\section{Libuv}
Libuv~\cite{Libuv2025} is a multi-platform event loop library focused on asynchronous I/O.
It is primarily developed for Node.js, but is used in several other large projects as well.
Libuv is centered the \mono{uv_loop_t} structure which represents a single event loop.
Applications can register work to a loop through the use of handles and requests~\cite{libuv_design}.
Handles are long-lived objects for certain types of work, while requests are short-lived operations.
When initial handles and requests have been registered, the application must invoke \mono{uv_run()}
in order to execute the event loop. This
will take over the invoking thread and by default run until all registered handles
have been closed.

Since libuv is cross-platform, the \mono{uv_udp_t} and \mono{uv_tcp_t} handles
are abstractions above OS-specific sockets. On Linux, the underlying event
queue object is epoll. Epoll is a structure which allows an application
to be notified by the Linux kernel when data is available on a socket.
This can be used to interact with the network, but also to monitor any other
file descriptors which are compatible with epoll.

While the libuv \mono{uv_udp_t}
and \mono{uv_tcp_t} handles mandate the type of the underlying file descriptor,
the generic \mono{uv_poll_t} can be used to monitor a generic event queue object.
This allows an application to use libuv for any file descriptor compatible
with an OS-specific event queue object. Finally, the \mono{uv_timer_t} handle
can be used to perform work at given intervals.

Handles are initiated and registered to a given event loop through
\mono{uv_*_init()}. Requests can then be added as discrete
pieces of work to the given handle. For example \mono{uv_write()} is
used to add a write request to a \mono{uv_tcp_t} handle. When registering
a request with the event loop, the application must also register an
associated callback. This callback will be invoked to notify the application
that execution has completed.
When the application is finished it must close the created handles for
the event loop to stop execution.

\section{Picoquic}\label{sec:picoquic}
Picoquic~\cite{christianhuitemaPicoquic} is a C library which aims to be a
minimalist implementation of the QUIC standard. It is a cross-platform
library compatible with Windows, Linux, macOS and BSD.
Though it features an optional event loop,
Picoquic is at its core a sans-I/O library \cite{cannon2016sansio}
and does not perform I/O itself.
The application drives all network reads and writes and feeds bytes into the library,
which returns bytes to send in response. This decouples protocol logic from any
particular I/O model, allowing the same library to be integrated with existing
I/O solutions.

The core Picoquic API is accessed through a polling API
which integrates with existing event loops.
For this mode the application is required to
call into Picoquic at given intervals. These intervals
are specified by Picoquic and must be fetched through
\mono{picoquic_get_next_wake_delay()}. A timer
can then be set to wake at the specified interval
and perform any work necessary by Picoquic.

When this timer fires, the application has to query
Picoquic for any new packets to send to the peer.
This is done through
\mono{picoquic_prepare_next_packet()}
which outputs one QUIC packet at a time along with
the expected source address, destination address
and outgoing interface index. The application
is then responsible for actually transferring
the data over a UDP socket.

Since the application is responsible for interacting
with the network it is also responsible for passing any
received QUIC packets to Picoquic. In order to start
receiving or sending QUIC data the application must
first create a \mono{picoquic_quic_t} object through
\mono{picoquic_create()}. This is an object which is
responsible for handling overarching QUIC state such as
TLS parameters as well as receiving incoming QUIC data.
This means that it is also responsible for passing
the incoming data to the correct QUIC connection, or
creating a new connection and passing it to the application.
Passing incoming QUIC packets to Picoquic is done through
\mono{picoquic_incoming_packet()}.

To handle the mandatory TLS for QUIC
Picoquic integrates with Picotls~\cite{picotls}.
Picoquic can therefore handle all necessary
TLS setup, only requiring that the application
provides the necessary parameters, such as
ALPN values and the various key files.

As an alternative to driving the polling API,
simple clients which do not want to run their own I/O
can run Picoquic's built-in event loop through
\mono{picoquic_packet_loop()}. This takes
control over the invoking thread and allows the application to
control the behavior of the loop through returning values
from a callback. When using this packet loop, Picoquic
manages its own sockets and interacts with the network
independently.

\section{Glib}
GLib~\cite{GLibGitLab} is a general purpose utility library which
is used at the core of several other larger libraries, such as GTK and GNOME.

Among other features, GLib provides implementations of several key data
structures not provided by the C standard library, such as dynamic arrays,
