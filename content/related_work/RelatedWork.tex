\chapter{Related work}\label{chap:related-work}
This chapter will describe existing TAPS implementations, compare them with CTaps
and demonstrate why a new open source implementation
is valuable.

\section{NEAT}
NEAT~\cite{khademiNEATPlatformProtocolIndependent2017},
is an open source \cite{neatCode} TAPS implementation whose development was funded by the EU Horizon 2020 program 
from 2015 to 2018~\cite{neatFunding}.
Developed alongside TAPS, NEAT and TAPS naturally
influenced each other. NEAT halted development in 2018, but work on
TAPS continued until the final release of the TAPS RFCs in 2025.

NEAT features an asynchronous, event-driven API. It uses an event loop
powered by libuv in order interact asynchronously with the network.
Underlying connections
can be configured through what is referred to as \term{NEAT Properties}.
NEAT Properties are key-value pairs and serve the same role as Transport Properties
do in TAPS. NEAT uses the concept of a flow 
as an abstraction for an underlying connection.

In NEAT, the transport protocols to be used can either be set directly by the application
via the \mono{transport} property, or be chosen by the \term{Policy Manager}.
The Policy Manager is a separate process configured
to run as a daemon~\cite[p.~81]{neat-d23} and other NEAT components communicate with it through Unix domain
sockets~\cite[p.~72]{neat-d23}. It is responsible for
generating a ranked list of candidates which fulfill the application's requirements. 

NEAT supports TLS, DTLS, TCP, UDP, UDP-Lite, SCTP and SCTP/UDP~\cite[p.~92]{neat-d23}.
QUIC had not yet been standardized by the IETF when development
of NEAT ended. NEAT does feature support for Connection Groups
and multistreaming over SCTP.


\section{NEATPy}
NEATPy~\cite{gundersenNEATPyStandardsConformantProtocolIndependent2020}
is an open source Python library developed as a master's thesis at UiO in 2020.
It was developed as a
Python layer to bridge the growing gap between NEAT and TAPS. Among
the goals was to move away from specifying the protocol at design time
instead using Selection Properties as specified by TAPS.

Because NEATPy is a Python shim on top of NEAT, certain parts of it are
limited by NEAT itself. For example,
using NEATPy it is not possible to specify the \mono{interface}
or the \mono{useTemporaryLocalAddress} Selection Properties 
\cite{NEATPyDocumentationNEATPy}.

Another limitation of NEATPy is that it was developed before the finalization of
the TAPS RFCs. NEATPy is based mostly on
version 4 and 5 of the TAPS interface draft~\cite[p.~109]{gundersenNEATPyStandardsConformantProtocolIndependent2020}.
but several changes have been made to TAPS since then.
An example of a new feature in the TAPS API is the support for passing more than one local and remote
endpoint when creating a Preconnection.
This is used to indicate that several endpoints are eligible for the connection
and is needed to support connection migration.
NEATPy has not had active development since 2020.

\section{Network.framework}
Network.framework~\cite{networkFramework} is Apple's closed source
TAPS implementation. It features several important TAPS abstractions
such as Connections, Listeners
and a common interface to interact with transport protocols.

During initiation, Network.framework requires that
applications specify a \term{NWParameters} object matching
a specific transport layer protocol. This causes the application
to be bound to this protocol at compile time. An example client
can be seen in~\cref{lst:network-framework-example}, where
a new connection to \enquote{example.com} is opened through QUIC
using the custom ALPN \enquote{myproto}.

\begin{listing}[htbp]
\begin{minted}{swift}
// Create a connection using QUIC
let connection = NWConnection(host: "example.com", port: 443, using: .quic(alpn: ["myproto"]))

// Set the state update handler to be notified when the connection is ready
connection.stateUpdateHandler = { newState in
    switch newState {
    case .ready:
        print("Connected using QUIC!")
    default:
        break
    }
}

// Start the connection with callback queue
connection.start(queue: queue)
\end{minted}
\caption[Network.framework QUIC Example]{Network.framework example code of opening a QUIC connection to example.com~\cite{WWDC2021_10094}.}
\label{lst:network-framework-example}
\end{listing}

Network.framework does not support candidate racing between protocols and endpoints
as described by TAPS,
only performing Happy Eyeballs-style IP-family racing.

\section{PyTAPS}
PyTAPS~\cite{frankePythonasynciotaps2025} 
is a standalone open source TAPS implementation
in Python using asyncio. PyTAPS is based on version 4 of the TAPS
interface draft and it has not had active
development since 2020. It features support for TCP and UDP\footnote{
PyTAPS' documentation claims support
for TLS~\cite{pytaps_docs}, but this is not reflected in the code. 
The security context created when passing Security Parameters
to a Preconnection in PyTAPS is overwritten both when creating
a Listener \cite{pytaps_listener_constructor}
and when creating a Connection \cite{pytaps_connection_constructor}.
This means that the \mono{ssl} parameter passed to asyncio's
\mono{create_connection()} and \mono{create_server()} is always
None.
}.

\section{Dynamic TAPS}
dynamic-taps is an open source C implementation of TAPS developed by the
American technology company F5~\cite{dukeDynamictaps2025}. It
is in the very early stages of development and has not had
active development since 2021.

While the most complete and production-ready implementation is Apple's Network.framework, it
also has several limitations. The largest limitation is naturally that it is a closed source
implementation and is only designed to be compatible with the Apple ecosystem.
This also means that the exact features supported
are harder to verify than for open source implementations, since the exact
logic cannot be inspected. In the overview it has therefore been assumed that any feature not
documented clearly, such as connection migration for QUIC, is not supported.

Compared to the existing implementations as shown in \cref{tab:comparison}, CTaps has
had the chance to innovate in two primary areas. First, when compared to Network.framework it is open
source and implements full candidate racing. Secondly, when compared to other open source
implementations it has been developed after the finalization of the TAPS RFCs and is the
first open source TAPS implementation to support QUIC.
