\chapter{Conclusion}\label{chap:conc}
In this thesis we have described CTaps, an implementation of IETF Transport Services
with support for QUIC, TCP and UDP.
We have described the architectural considerations of interest to future TAPS implementations
and how the features of QUIC are best exposed through a TAPS API. 

In this chapter we will provide answers to the stated research questions.
In addition, we will discuss the scope and limitations of the thesis as a whole
and future work if CTaps is to be further developed into a production-ready library.

\section{Research Questions}
At the onset of this thesis two primary research questions were defined
as goals and key motivations behind the development of CTaps. These

\subsection{Research Question One}\label{sec:rq1-answer}
The TAPS API mandates several features such as asynchrony
and flexible protocol handling, but leaves details up to the
implementations themselves. This means that several
implicit architectural requirements are posed by the TAPS standard, but are not
stated explicitly in the RFCs.

One of the largest requirements from the TAPS API is that of flexibility.
All TAPS implementations
should use a generic protocol interface internally, in order to be as maintainable as possible.
Without a generic interface, the internal implementation code
risks becoming very tightly bound to specific protocols, making it
far more difficult to adapt the TAPS implementation to support new protocols.
RFC~9623 also recommends that a Listener should
be able to receive from multiple transport protocols at once over a single
socket \cite[sec. 4.7]{rfc9623}. TAPS implementations should therefore
implement internal logic to listen for incoming data from multiple different protocols
when opening a Listener. If an implementation wants to support this fully and
support simultaneous listening for protocols which use the same protocol number
in the IP header, and therefore the same socket type, then this would have
major architectural implications. This is discussed further in \cref{sec:future-work}.

Since TAPS requires implementations to support connectionless protocols,
multiple Connections must be able to share a single socket. This imposes
a non-trivial lifetime management constraint: the underlying socket must
remain open until the last associated Connection has closed, meaning
that socket lifetime cannot be tied directly to individual Connections.

TAPS explicitly supports connection migration and multipath,
and the underlying architecture must be adapted to make sure that the 
socket and Local Endpoints used for a given Connection are flexible.
This means that a Connection should be able to send
and receive over more than one socket concurrently and can
therefore not be bound one-to-one with sockets.

Supporting multistreaming requires that QUIC connections are treated
as Connection Groups, in order to make streams available to applications.
This leads to a decision having to be made when it comes to
closing QUIC streams. QUIC expects streams to be closed by
each side sending a FIN flag, while TAPS implies that \mono{close()}
initiates a bidirectional close.
Future TAPS implementations therefore have to choose between
presenting either the expected behavior of QUIC streams,
or a consistent semantic of closing Connections.
The CTaps decision of preserving
behavior expected by QUIC is discussed further in
\cref{sec:quic-streams-under-taps}.

In summary, the key implicit architectural constraints of a TAPS System are:
a flexible many-to-many binding between Connections and sockets to support
migration, multistreaming and shared socket lifetime; and a generic internal
protocol interface to enable easy support of new protocols.

\subsection{Research Question Two}
While not all QUIC features are directly controllable through the TAPS API,
TAPS is an abstraction and can infer most desired settings based on the
set Transport Properties.

Several key features of QUIC are directly supported through the TAPS
API, giving the application fine-grained control.
For example, connection migration and multipath QUIC are available
through using multiple Local Endpoints, setting the \mono{multipath}
Selection Property and setting the \mono{multipathPolicy} Connection Property.
While this does not expose functionality to explicitly trigger a migration,
it allows for granular control of the expected behavior, as seen in
\cref{tab:multipath-combinations}.

A similar pattern applies to flow control, where QUIC exposes \mono{initial_max_data}
and other associated transport parameters. These are not available through TAPS directly, but TAPS allows for flow 
control by enqueueing receive callbacks instead. The underlying TAPS System
can then pass a sane default parameter to the peer initially,
increasing the window as the application enqueues \mono{receive()}-callbacks.

Certain QUIC features are supported, but only through coarse options or with a conceptual mismatch in the TAPS API.
For example, the QUIC settings related to ACKs, \mono{ack_delay_exponent} and \mono{max_ack_delay},
can be set by TAPS by inferring a desired value based on the \mono{connCapacityProfile}
Connection Property, but direct numerical values cannot be set.
Another example of a coarse mapping is that TAPS only allows for sending of
\mono{STOP_SENDING} and \mono{RESET_STREAM} together, since these map to
aborting Connections, and TAPS has no concept of a unidirectional abort.
A conceptual mismatch between QUIC and the TAPS API is that path change events are per-Connection,
while QUIC migration is per-Connection Group. The TAPS specification does not prescribe how
implementations should handle this, leaving it as an implementation decision
whether to notify all Connections in a group individually, or only a representative one.

Finally, the features which are genuine gaps are the ones which cannot be accessed directly
and cannot be inferred from the intention-based TAPS API.
Certain gaps could be assumed to be handled by the underlying
TAPS System, such as Connection~ID
handling and stateless reset, while other gaps are QUIC features
which would be reasonable for an application to control, but that are not
available through the TAPS API,
for example unidirectional streams and idle timeout.

For unidirectional streams, TAPS only supports unidirectional Connections through the
\mono{direction} Selection Property, but this must be set before a Preconnection
is initiated and cannot be used to influence
the directionality of Connections which are a result of \mono{clone()}.
In other words, a Connection Group of N Connections will either be unidirectional
or bidirectional, but the API does not allow an application to make an
individual choice for a Connection within the Connection Group.

For QUIC's idle timeout, the TAPS API exposes \mono{connTimeout}, but
the two are not semantically equivalent. In TAPS, \mono{connTimeout}
is set when trying to reliably deliver data to a Remote Endpoint,
while in QUIC, idle timeout is purely an inactivity timer, resetting
on sending or receiving data and closing the connection when it expires.

Of the 28 QUIC features surveyed in \cref{tab:quic-taps-mapping},
19 are available, 5 are partially available, and 4 are
unavailable. Given that 24 of 28 features are fully or partially available,
TAPS exposes the large majority of the queried QUIC features.
Genuine gaps are of two kinds: conceptual mismatches,
where TAPS provides related but insufficiently precise functionality
(directionality when cloning, idle timeout), and intentional abstractions,
where TAPS deliberately omits application-facing control over
implementation-level concerns (stateless reset, connection ID handling).

\section{Scope and Limitations}
CTaps is an exploratory work, intended to show how an implementation of a TAPS
system might look in a real world context, but it is not production-ready.
A goal of CTaps has been to expose as much as possible of
the TAPS API, without necessarily implementing the internal logic associated with
the actual API settings.

A consequence of this push to make CTaps present as much of the API of TAPS
as possible is that most testing and development has been focused on the
happy paths of various features, while less time and testing has been devoted
to testing various error paths. This would naturally make CTaps quite
unstable in contexts where real network errors are encountered.

Out of the TAPS API, the only large feature which was deemed out of scope
and is not presented in CTaps at all is peer to peer connection establishment.
Among the smaller features not present in CTaps, a prominent example is that CTaps
does not support adding endpoints during the lifetime of a connection.
Not being able to add endpoints to an active Connection places a real limitation on
an application's ability to control connection migration, since all Local and Remote
endpoints must be passed to a Preconnection before initiation.
CTaps does not support listening on multiple protocols or Local Endpoints for a single Listener.

\section{Future Work}\label{sec:future-work}
There are several features in CTaps which should be developed
further if CTaps is to be used in a production environment.
The most important of them are discussed here.

\subsection{Error Handling}
The most important improvement which could be made to CTaps is improved error handling.
For example, the \mono{soft_error()} callback is never invoked and
there are several error paths where previously allocated memory is not freed.
This has not been an issue with the experiments since these have been run over
a simulated network on a single computer and therefore have not faced any unexpected network
errors, but they would be encountered in a production setting.

While there are too many possible failure branches to mention a full list,
further development of CTaps should add testing for
all failure paths ensuring that the errors are handled correctly and recovered
from when possible. In addition, memory checks should be added to these tests,
ensuring that CTaps does not leak allocated memory when encountering such
an error path.

\subsection{Faster Connection Establishment}
When it comes to performance, the most important step for CTaps would be to
improve the connection establishment time. This would consist of two major
factors: candidate interleaving and minimizing DNS blocking.

Since protocols are ranked in order of application preference and
TAPS implementations will likely want to give preference to IPv6 links,
care must be taken when deciding exactly how to interleave candidates.
Happy Eyeballs V3 is currently being developed, with suggestions on
how to exactly extend normal IP-family racing to transport-layer
protocols \cite[sec.~5]{ietf-happy-happyeyeballs-v3-03}.

In addition to the ordering of candidates, the connection establishment time
for CTaps is also affected by possible DNS delay. To mitigate this, candidate racing in CTaps
should be modified to start racing as soon as a resolved endpoint arrives with a \qty{50}{\milli\second} resolution delay,
as recommended in Happy Eyeballs~\cite[sec.~3]{rfc8305}.
This would give preferred addresses a chance to return before initiating an
attempt with a less desirable endpoint.

\subsection{Connection Migration}\label{sec:future-conn-mig}
While connection migration is currently supported in CTaps, an important piece
of future work would be to make it available without the workaround
described in~\cref{sec:connection-migration-impl}. This means that the application
would be able to select specific Local Endpoints and migrate between them, instead
of being forced to select a wildcard Local Endpoint to bind to.

The simplest implementation would be to expand the internal protocol interfaces 
to support multiple sockets where applicable. Currently,
this means that the QUIC protocol interface in CTaps would
support a lazily initiated pool of sockets.

When first initiating a Connection, the QUIC protocol interface 
would open a UDP socket on the most desirable Local Endpoint
and have Picoquic initiate a Connection from the given socket.
Since Picoquic informs CTaps about which local address it wants
to send from, CTaps can then check if a socket associated with
the given local address has already been created. Otherwise, it would create a new
UDP socket.

This presents an advantage over the current workaround,
since it would allow the application to have full control over
which interfaces to actually receive data on. Currently,
applications are forced to receive data on all addresses for the given port,
when binding to a wildcard address.

\subsection{Framers}\label{sec:future-framer}
As described in~\cref{sec:framer-impl} the current support for framers
in CTaps is largely a proof-of-concept, and should be improved if CTaps
is to be further developed. The largest limitation is that Connections
only support a single framer.
The Connection structure could be
expanded to support a dynamic array of Framers, iterating over them
whenever a Connection receives or sends data.

In addition, CTaps should add support for Framers to combine
and split Messages.
Framers in CTaps are already asynchronous and so this would require relatively little modification.
Applications are expected to implement
their own Framers and could be expected to maintain their own buffers
for working on multiple Messages.
A user context parameter should be added to the Framing functions,
allowing applications to pass external state into the framing callbacks.
Care would have to be taken that the ownership model of the Messages
created inside Framer functions is clear, so that applications
can implement functions to combine and split Messages
without having to risk memory leaks.

RFC~9622 states that Framers can be used to interact
with the setup and teardown of Connections~\cite[sec.~9.1.2]{rfc9622},
but this is not currently supported in CTaps. The abstract Framer
interface presented through the CTaps API could
be expanded with two function pointers, one for
interacting with a setup and one for interacting with
teardown. CTaps' internal logic would then have to
be updated to check for registered Framers and query
them for data to send, upon setup and teardown.

\subsection{More Flexible Listeners}\label{sec:protocol-ag-listener}
RFC~9621 says that Listeners by default bind to multiple paths,
protocols and Local Endpoints~\cite[sec.~4.1.4]{rfc9621}. This
is not supported in CTaps and would be an important future addition
in order to improve flexibility. CTaps currently binds Listeners
to the most desirable protocol and the first endpoint provided by the
application. Listening on multiple Local Endpoints
at the same time would be fairly simple to implement, only needing
to maintain a list of sockets for listening, similar to the multiple sockets described in
\cref{sec:future-conn-mig}.

Listening on multiple protocols at the
same time is doable, but could be more complicated depending
on the combination of protocols desired. For sockets of different types, that
is \mono{SOCK_STREAM} and \mono{SOCK_DGRAM}, CTaps would be
able to create multiple sockets listening on the same Local Endpoint,
and the kernel would be able to demultiplex incoming data to the
correct socket, since each socket would be associated
with a different transport-layer protocol.

This becomes far more complicated if CTaps should
receive on QUIC and UDP, which use the same underlying
socket type, on the same Local Endpoint at the same time.
This could be done by bypassing
the kernel networking stack through AF\_XDP or by
handling all demultiplexing internally,
both of which would greatly increase internal complexity. The
pragmatic design decision would be to assume that
applications needing to receive both QUIC and UDP data on the
same Local Endpoint are rare, and that the added complexity
does not bring much value for this specific case.

\subsection{TLS}\label{sec:future-work-tls}
The most important protocol which is currently not supported
in CTaps is TLS over TCP. While TLS conceptually resides
above TCP, it would be best suited to be treated as a completely separate
protocol by CTaps, adding TLS over TCP as an implementation of the protocol interface
alongside QUIC, UDP and TCP.

When it comes to which library should be used in order to
support TLS, CTaps already has a transitive dependency on
Picotls through Picoquic. This, alongside its sans-I/O functionality
would make it a good candidate library, as long as supporting
only TLS~1.3 is deemed acceptable.

Adding more encrypted protocols
also means that candidate racing should be reworked to be
based on the Security Parameters given by the application.
This is described in \cref{sec:design-sec} and would ensure
that CTaps does not silently downgrade security.

\section{Broader Significance}
While CTaps could be developed to become a real production-ready library
through the steps described in~\cref{sec:future-work}, the broader
significance is more tightly coupled to it being the first
publicly announced open-source implementation of TAPS to support QUIC.
This has led to insight on not only how to map
QUIC features through a TAPS API, but also on exactly how to build a TAPS
implementation and which internal architectural considerations need to
be made early, in order to support the flexibility mandated by TAPS.

For the broader significance of TAPS, the key finding
is the complexity of the clients used for experimenting as
described in~\cref{sec:client-complexity}. These show that
TAPS clients feature a much richer feature set by default, with
a small increase in complexity.

While the API differs from a typical sequential programming model to a very large degree,
TAPS is a genuinely useful abstraction for both simple and complicated
clients. The primary obstacle for TAPS' wider adoption is therefore
not the standard itself, but that an implementation of a TAPS System must be
fairly feature complete before it becomes useful to downstream
applications. This would require a sustained investment, but
CTaps lowers the bar by providing a library to develop
further and architectural patterns that others can follow.
